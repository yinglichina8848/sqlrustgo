# CI Pipeline

## Definition

CI Pipeline is the GitHub Actions workflow that enforces governance rules. It runs on every PR and is the sole mechanism for proof generation.

---

## Implementation

**File:** `.github/workflows/ci.yml`

**Trigger:** `pull_request` to `main` branch

**Runs on:** `ubuntu-latest`

---

## Pipeline Steps

### Step 1: Detect Test File Changes (R1)

```yaml
- name: Detect test file changes
  run: |
    git fetch origin main
    CHANGED=$(git diff --name-only origin/main...HEAD | grep "^tests/" || true)
    if [ ! -z "$CHANGED" ]; then
      echo "❌ Tests modified:"
      echo "$CHANGED"
      exit 1
    fi
```

**Purpose:** Block PRs that modify `tests/` directory

**R1 Enforcement:** `grep "^tests/"` detects any test file modification

---

### Step 2: Detect NEW #[ignore] Additions (R2)

```yaml
- name: Detect NEW #[ignore] additions
  run: |
    git fetch origin main
    NEW_IGNORES=$(git diff origin/main...HEAD -- "*.rs" | grep "^+.*#\[ignore\]" || true)
    if [ ! -z "$NEW_IGNORES" ]; then
      echo "❌ NEW #[ignore] detected in PR:"
      echo "$NEW_IGNORES"
      exit 1
    fi
```

**Purpose:** Block PRs that introduce new `#[ignore]`

**R2 Enforcement:** `grep "^+.*#\[ignore\]"` matches only added lines

**Note:** Does not detect modified existing ignores (AV6)

---

### Step 3: Setup Rust

```yaml
- name: Setup Rust
  uses: dtolnay/rust-toolchain@stable
```

**Purpose:** Install Rust toolchain for build and test

---

### Step 4: Build

```yaml
- name: Build
  run: cargo build
```

**Purpose:** Verify code compiles before running tests

**Fail condition:** `cargo build` exit code != 0

---

### Step 5: Run Tests

```yaml
- name: Run tests
  run: cargo test
```

**Purpose:** Execute full test suite

**Output:** `stdout + stderr` containing test results

**Note:** Does not use `--all` flag (output format differs)

---

### Step 6: Verify and Generate Proof

```yaml
- name: Generate verification report
  run: |
    python3 scripts/verification_engine.py
```

**Purpose:** Run `verification_engine.py` to generate `verification_report.json`

**R3 Enforcement:** `verification_report.json` is regenerated by CI, not accepted from PR

---

### Step 7: Upload Artifact

```yaml
- name: Upload verification report
  uses: actions/upload-artifact@v4
  with:
    name: verification-report
    path: verification_report.json
```

**Purpose:** Store proof as CI artifact for audit trail

---

## Pipeline Diagram

```
PR submitted to main
    ↓
GitHub Actions triggered
    ↓
Step 1: Check tests/ modified? (R1)
    ├─ YES → exit 1 (REJECT)
    └─ NO → continue
    ↓
Step 2: Check new #[ignore]? (R2)
    ├─ YES → exit 1 (REJECT)
    └─ NO → continue
    ↓
Step 3: Setup Rust
    ↓
Step 4: cargo build
    ├─ FAIL → exit 1 (REJECT)
    └─ PASS → continue
    ↓
Step 5: cargo test
    ↓
Step 6: verification_engine.py → verification_report.json
    ↓
Step 7: Upload artifact
    ↓
Merge Gate
    ├─ baseline_verified = true → OPEN
    └─ baseline_verified = false → CLOSED
```

## Q&A

**Q: Why does CI use `origin/main...HEAD` instead of `HEAD~1..HEAD`?**

A: `origin/main...HEAD` captures all commits in the PR branch compared to main. `HEAD~1..HEAD` only captures the latest commit. A PR may have multiple commits.

**Q: What happens if CI runs but `verification_report.json` is not generated?**

A: The `verification_engine.py` step exits 1 on failure. CI fails. Merge gate closes.

**Q: Can CI be modified by PR?**

A: Yes. `.github/workflows/ci.yml` is in the repository. R7 (out-of-band) detects this. Branch protection should prevent modification.

**Q: Why is `cargo test` used instead of `cargo test --all`?**

A: The output format differs. `--all` produces aggregated output that `verification_engine.py` cannot parse correctly.

**Q: What is the artifact retention policy?**

A: Defined by GitHub Actions defaults. Not governed by this system.
