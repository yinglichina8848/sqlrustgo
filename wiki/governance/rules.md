# Governance Rules

## Definition

Governance Rules are machine-enforceable constraints on PR behavior. Each rule maps a condition to an enforcement action (REJECT/ACCEPT). Rules are implemented in `.github/workflows/ci.yml` and validated by `scripts/ci_self_test.sh`.

## Rule Set

### R1: Test Immutability

**Statement:** Test files define behavioral contracts. PR authors cannot modify them.

**Formal:**
```
IF diff(PR) ∩ path("tests/*") ≠ ∅
THEN CI MUST exit 1
```

**Enforcement:** `.github/workflows/ci.yml` step "Detect test file changes"

**Check:**
```bash
CHANGED=$(git diff --name-only origin/main...HEAD | grep "^tests/")
if [ -n "$CHANGED" ]; then exit 1; fi
```

**Invariant:** `INV1: ∀t ∈ T · expect(t) is CONST`

**Why:** If PR authors can modify tests, they can make failing code appear passing.

---

### R2: Ignore Injection Prevention

**Statement:** New `#[ignore]` annotations cannot be introduced by PR.

**Formal:**
```
IF diff(PR) matches regex "^+.*#\[ignore\]"
THEN CI MUST exit 1
```

**Enforcement:** `.github/workflows/ci.yml` step "Detect NEW #[ignore] additions"

**Check:**
```bash
NEW_IGNORES=$(git diff origin/main...HEAD -- "*.rs" | grep "^+.*#\[ignore\]")
if [ -n "$NEW_IGNORES" ]; then exit 1; fi
```

**Grandfather Clause:** Existing ignores from v2.7.0 are not checked. Only `diff(PR)` is checked.

**Why:** `#[ignore]` removes a test from the proof set. PR authors could hide failures.

---

### R3: Proof Authenticity

**Statement:** Proof must be generated by CI, not submitted by PR.

**Formal:**
```
IF P ∈ PR.diff THEN REJECT
IF P.generated_by ≠ "verification_engine.py" THEN REJECT
IF P.source ≠ "CI execution" THEN REJECT
```

**Enforcement:** CI regenerates `verification_report.json` via `verification_engine.py`. PR cannot commit P.

**Why:** If PR authors could submit P, they could fake proof. Proof must be CI-provenance to be trusted.

---

### R4: Full Execution Requirement

**Statement:** All tests must run and pass. No skipped tests.

**Formal:**
```
IF ∃t ∈ T · runs(t) = FAIL THEN CI MUST exit 1
IF ∃t ∈ T · runs(t) = IGNORED THEN CI MUST exit 1
```

**Enforcement:** `verification_engine.py::parse_results()` counts passed/failed/ignored. If failed > 0 OR ignored > 0 → exit 1

**Baseline:** 226 passed, 0 failed, 0 ignored

**Why:** Tests not run provide no evidence. Tests not passing provide negative evidence.

---

### R5: Baseline Verification

**Statement:** The baseline must be maintained exactly.

**Formal:**
```
P.baseline_verified = true
⇔ (P.passed = 226 ∧ P.failed = 0 ∧ P.blocker_count = 0)
```

**Enforcement:** CI sets `baseline_verified` field during proof generation.

**Why:** `baseline_verified` is the system seal. If true → system was in known-good state when proof generated.

---

### R6: Test Count Monotonicity

**Statement:** Test coverage cannot decrease.

**Formal:**
```
IF P.passed < baseline.passed (226)
THEN CI MUST exit 1
```

**Enforcement:** `verification_engine.py` compares current `passed` count against baseline.

**Why:** Deleting tests is equivalent to ignoring them. Coverage must only increase or stay equal.

---

### R7: CI Workflow Integrity

**Statement:** CI enforcement cannot be removed or modified by PR.

**Formal:**
```
IF diff(PR) modifies .github/workflows/ci.yml
THEN CI self-test MUST detect
```

**Enforcement:** `scripts/ci_self_test.sh` (out-of-band, periodic verification)

**Note:** Not enforced per-PR. Per-PR enforcement would require meta-CI, which creates infinite regress.

**Why:** If PR can modify CI, PR can remove all enforcement.

---

## Complete Validation Function

```
validate(PR) → {
  diff ← git diff origin/main...HEAD

  IF diff ∩ path("tests/*") ≠ ∅         → REJECT("R1: tests modified")
  IF diff matches "^+.*#\[ignore\]"       → REJECT("R2: new #[ignore]")
  IF P ∈ diff                             → REJECT("R3: proof not CI-generated")
  IF P.baseline_verified ≠ true           → REJECT("R5: baseline not verified")
  IF P.passed < 226                       → REJECT("R6: test count decreased")
  IF failed > 0 OR ignored > 0            → REJECT("R4: tests failing/ignored")
  IF diff ⊂ .github/workflows/ci.yml     → FLAG("R7: CI modified (manual review)")

  ELSE → ACCEPT
}
```

## Invariants

```
INV1: Test Suite Completeness
  ∀t ∈ T · runs(t) ≠ UNRUN

INV2: Proof Provenance
  P.generated_by = "verification_engine.py"
  ∧ P.source = "CI execution"

INV3: Baseline Lock
  baseline.passed = 226
  baseline.failed = 0
  baseline.ignored = 0

INV4: Merge Gate Soundness
  merges(PR) ⇔ validate(PR) = ACCEPT
```

## Q&A

**Q: Can a PR author modify a test expectation value (e.g., change `assert_eq!(x, 4)` to `assert_eq!(x, 2)`)?**

A: No. R1 blocks any modification to files in `tests/`. Test expectation values are part of the test contract.

**Q: Can a PR author add `#[ignore]` to a failing test to make CI pass?**

A: No. R2 blocks any new `#[ignore]` introduction. Grandfathered ignores from v2.7.0 are not checked.

**Q: What happens if a PR author commits a fake `verification_report.json` with `baseline_verified: true`?**

A: R3 ensures CI regenerates proof. PR-submitted proof is always overwritten. Fake proof is detected because it is not in CI output.

**Q: What happens if CI passes but `baseline_verified` is set to false?**

A: Merge gate closes. `baseline_verified = false` means proof did not meet requirements. PR cannot merge.

**Q: Can CI be modified by PR to remove enforcement steps?**

A: R7 detects this via `ci_self_test.sh` (out-of-band). However, per-PR enforcement is not implemented. Branch protection is required.

**Q: What is the difference between R2 and R6?**

A: R2 prevents new `#[ignore]` annotations (skipping tests). R6 prevents deleting tests (reducing coverage). Both close different cheating vectors.
