# Trust Boundary

## Definition

Trust Boundary defines which components are trusted (assumed correct) and which are untrusted (potentially malicious) within the governance system.

---

## Trust Classification

### Trusted (Assumed, Not Verified Within System)

```
T1: GitHub Actions runner integrity
    Assumption: ubuntu-latest runners are not compromised
    Implication: CI execution results are authentic

T2: Rust toolchain not tampered
    Assumption: cargo, rustc from official distribution are authentic
    Implication: Test execution is reproducible

T3: cargo test output format stable
    Assumption: "test result:" format does not change across versions
    Implication: V can reliably parse output

T4: verification_engine.py logic correctness
    Assumption: V correctly implements parse_results()
    Implication: Proof reflects actual test results

T5: Branch protection prevents force-push
    Assumption: Repository settings block direct main push
    Implication: All changes go through PR/CI

T6: Repository maintainers non-malicious
    Assumption: Admin users do not intentionally bypass governance
    Implication: T5 cannot be disabled maliciously
```

---

### Untrusted (PR Can Modify)

```
U1: src/ files (business logic)
    PR can modify: Any .rs file in src/
    Risk: Semantic regression (AV5)

U2: tests/ directory
    PR can attempt to modify (blocked by R1)
    Risk: Test immutability violation

U3: verification_engine.py
    PR can modify: V is in repository
    Risk: Verification engine tampering (AV4)

U4: .github/workflows/ci.yml
    PR can modify: CI workflow in repository
    Risk: CI disabling (AV3)

U5: verification_report.json
    PR can attempt to commit (overwritten by CI)
    Risk: Proof replay (AV8 — mitigated)
```

---

### Conditionally Trusted

```
C1: verification_report.json
    Condition: P.generated_by = "verification_engine.py"
               AND P.source = "CI execution"
    If true: TRUSTED
    If false: UNTRUSTED

    Reasoning: Proof is only trusted if it came from CI execution.
    PR-submitted proof is not trusted.
```

## Trust Chain

```
GitHub Actions (T1)
    ↓
cargo test (T2, T3)
    ↓
stdout/stderr
    ↓
verification_engine.py (T4)
    ↓
verification_report.json (C1)
    ↓
Merge Gate
    ↓
PR Merged OR Rejected
```

### Chain Dependencies

```
Proof Valid IF AND ONLY IF:
  T1 ∧ T2 ∧ T3 ∧ T4 ∧ T5 ∧ T6 ∧ C1

If ANY assumption fails → Proof invalid
```

## What Trust Is NOT Assumed

The governance system explicitly does NOT assume:

```
TRUST IS NOT ASSUMED FOR:

1. Test Correctness
   Tests may have bugs. Passing tests ≠ correct code.

2. Code Semantic Correctness
   passes(tests) does NOT imply semantically correct.
   See: AV5 (Semantic Regression)

3. Non-Determinism Detection
   System cannot detect tests that fail intermittently.
   See: AV9 (Non-Deterministic Test Masking)

4. CI Runner Compromise
   If GitHub Actions is compromised, all bets are off.
   Mitigation: Organizational security controls.

5. verification_engine.py Correctness
   V could have bugs producing false proof.
   Mitigation: V should be in separate protected repo (v2.9.0)
```

## Q&A

**Q: Why is GitHub Actions considered trusted?**

A: It is a foundational assumption. If GitHub Actions is compromised, the attacker can make CI report anything. Defending against this requires organizational security controls outside the scope of this system.

**Q: Can a PR author modify `verification_engine.py` to fake proof?**

A: Yes (AV4). V is in the repository and not integrity-checked. A malicious PR could modify V to output fake proof. Mitigation: V should move to separate protected repository (v2.9.0).

**Q: Why is `tests/` untrusted if R1 blocks modifications?**

A: R1 is enforced by CI. But CI itself is in the repository (U4). If a PR simultaneously modifies CI to disable R1 and modifies tests, the attack succeeds. Mitigation: Branch protection + R7.

**Q: What happens if all trust assumptions T1-T6 are violated simultaneously?**

A: The system provides no defense. This is the catastrophic failure mode. Governance systems cannot defend against a fully compromised environment.

**Q: Is `verification_report.json` trusted or untrusted?**

A: Conditionally trusted (C1). If generated by CI execution, it is trusted. If PR-submitted, it is untrusted (overwritten by CI anyway).

**Q: What is the difference between T5 (branch protection) and R7 (CI integrity)?**

A: T5 is a GitHub-level control. R7 is a CI-level check. T5 prevents direct pushes. R7 detects if CI enforcement is modified. Both are needed.
