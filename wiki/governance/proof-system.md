# Proof System

## Definition

Proof System is the mechanism by which PR correctness is verified. Proof is generated by CI, not declared by PR. The proof artifact `verification_report.json` serves as evidence of test suite execution.

## Core Principle

```
Proof is GENERATED by CI, not DECLARED by PR.
```

**Why:** If PR authors could declare proof, they could fake evidence. Proof must have CI provenance.

---

## Proof Artifact

### Schema: `verification_report.json`

```json
{
  "proof_version": "1.0",
  "generated_by": "verification_engine.py",
  "source": "CI execution (not PR declaration)",
  "passed": 226,
  "failed": 0,
  "baseline_verified": true,
  "blocker_count": 0,
  "status": "VERIFIED"
}
```

### Field Semantics

| Field | Type | Meaning |
|-------|------|---------|
| `proof_version` | string | Schema version (currently "1.0") |
| `generated_by` | string | Must be "verification_engine.py" |
| `source` | string | Must be "CI execution (not PR declaration)" |
| `passed` | integer | Count of passing tests |
| `failed` | integer | Count of failing tests |
| `baseline_verified` | boolean | True iff `passed=226 AND failed=0 AND blocker_count=0` |
| `blocker_count` | integer | Number of failing tests (blocking merge) |
| `status` | string | "VERIFIED" or "FAIL" |

## Proof Generation Pipeline

```
PR submitted
    ↓
CI triggered
    ↓
STEP 1: Diff Analysis (R1, R2)
    - Check tests/ modified?
    - Check new #[ignore]?
    ↓
STEP 2: Build
    - cargo build
    - Exit on failure
    ↓
STEP 3: Test Execution
    - cargo test
    - Capture stdout/stderr
    ↓
STEP 4: Parse Results
    - Extract "test result: ok. N passed; M failed"
    - Sum across all test suites
    ↓
STEP 5: Generate Proof
    - Write verification_report.json
    - Set baseline_verified based on results
    ↓
STEP 6: Merge Gate
    - If baseline_verified = true → OPEN
    - Else → CLOSED
```

## Proof Properties

### PP1: Tamper-Evidence

```
P is regenerated by CI on every run.
PR cannot submit P — CI overwrites any PR-submitted P.
```

**Enforcement:** R3 — CI regenerates, not accepts.

---

### PP2: Provenance

```
P.generated_by = "verification_engine.py"
P.source = "CI execution"
```

**If violated:** Proof is invalid, merge gate closes.

---

### PP3: Completeness

```
P.passed = 226 ∧ P.failed = 0 ∧ P.ignored = 0
⇔ All 226 tests ran and passed.
```

---

### PP4: Determinism

```
Given identical T and CI environment, V produces identical P.
```

**Note:** Non-deterministic tests (AV9) can break this property.

## Verification Engine

### Identity

```
V = scripts/verification_engine.py
Language: Python 3
Input: cargo test stdout/stderr
Output: verification_report.json
```

### Functions

```
run_tests() → (exit_code: int, output: string)
  Executes: cargo test
  Returns: (return code, combined stdout+stderr)

parse_results(output: string) → (passed: int, failed: int, ignored: int)
  Regex: r'test result:.*?(\d+) passed.*?(\d+) failed.*?(\d+) ignored'
  Sum across all "test result:" lines

generate_report()
  Runs run_tests()
  Runs parse_results()
  Sets baseline_verified = (failed == 0 and passed >= 226)
  Writes verification_report.json
  Exits 0 if verified, exits 1 if failed
```

### Properties

```
PROPERTY V1: Deterministic
  ∀output · parse_results(output) is deterministic

PROPERTY V2: Faithful
  parse_results(output) accurately reflects cargo test output
  (no invented data)

PROPERTY V3: Complete
  Sums all test result lines from all test suites
```

## Trust Chain

```
[T1, T2, T3] → cargo test produces stdout
    ↓
[T4]          → verification_engine.py parses stdout
    ↓
[T5]          → CI writes verification_report.json
    ↓
[C1]          → Proof validated by merge gate

BREAK AT ANY [Tx] → Entire chain invalid
```

## Q&A

**Q: What is the difference between "source: CI execution" and "source: PR declaration"?**

A: "CI execution" means proof was generated by running `cargo test` in CI. "PR declaration" means PR author manually wrote proof. Only CI execution is trusted.

**Q: Can a PR author bypass the proof system by not running tests locally?**

A: No. CI always runs tests. PR cannot skip CI execution.

**Q: What happens if `verification_engine.py` has a bug that miscounts tests?**

A: System has no defense against this (AV4). V is in repository and not integrity-checked. Mitigation: V must be in separate protected repository (v2.9.0 roadmap).

**Q: Can proof be replayed from a previous successful run?**

A: No (AV8). CI always regenerates proof. PR-submitted proof is overwritten.

**Q: What is the baseline for v2.8.0?**

A: `passed: 226, failed: 0, ignored: 0, baseline_verified: true`

**Q: Can proof be generated without running tests (e.g., by manually calling `generate_report()`)?**

A: Yes, locally. But CI regenerates proof on every run. Local proof is not accepted — only CI-generated proof is trusted.
