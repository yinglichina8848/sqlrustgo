## 二、OO 纵向执行链路性能瓶颈分析

### 2.1 执行流水线架构

```text
SQL Input
    │
    ▼
┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
│ Parser │───▶│Planner │───▶│Optimizer│───▶│Executor│───▶│Storage│
└────────┘    └────────┘    └────────┘    └────────┘    └────────┘
    │            │             │             │             │
    ▼            ▼             ▼             ▼             ▼
   AST      Logical      Physical       Iterator       Page I/O
             Plan         Plan          Model
```

### 2.2 各阶段性能指标

| 阶段 | 当前延迟 | 目标延迟 | 瓶颈等级 |
|------|----------|----------|----------|
| Parser | < 1ms | < 1ms | ✅ 达标 |
| Planner | < 10ms | < 10ms | ✅ 达标 |
| Optimizer | < 50ms | < 50ms | ⚠️ 接近上限 |
| Executor | Query 依赖 | - | ❌ 主要瓶颈 |
| Storage | - | - | ❌ I/O 瓶颈 |

### 2.3 关键性能瓶颈

#### 瓶颈 1: Hash Join 内存独占

```text
问题:
┌─────────────────────────────────────────────────────────────┐
│  HashJoinExecutor 执行时:                                     │
│                                                              │
│  1. Build Phase: 将整个右表加载到内存                        │
│  2. Probe Phase: 遍历左表进行匹配                           │
│                                                              │
│  问题: 大表 JOIN 时内存峰值高，单线程执行                     │
└─────────────────────────────────────────────────────────────┘

优化方案:
1. 增量 Hash Join (分批处理)
2. 并行执行 (Parallel Hash Join)
3. Grace Hash Join (溢出到磁盘)
```

#### 瓶颈 2: 无计划缓存 (Plan Cache)

```text
问题:
┌─────────────────────────────────────────────────────────────┐
│  相同 SQL 模板每次都重新:                                    │
│                                                              │
│  1. Parse → AST                                             │
│  2. Bind → Logical Plan                                     │
│  3. Optimize → Physical Plan                                  │
│                                                              │
│  浪费: ~10-50ms 重复工作                                     │
└─────────────────────────────────────────────────────────────┘

优化方案:
1. 实现 Plan Cache (LRU, 大小可配置)
2. 基于 SQL 模板哈希缓存
3. 模板参数化
```

#### 瓶颈 3: Storage I/O 放大

```text
问题:
┌─────────────────────────────────────────────────────────────┐
│  Volcano Iterator Model 每次 next() 调用:                    │
│                                                              │
│  1. 算子层次深时函数调用开销大                              │
│  2. 页面预读不智能 (Sequential Scan 才有效)                  │
│  3. BufferPool LRU 策略简单                                 │
└─────────────────────────────────────────────────────────────┘

优化方案:
1. 批量预读 (Batch Prefetch)
2. 索引跨度预读 (Index Span Prefetch)
3. Adaptive Buffer Pool (根据访问模式调整)
```

#### 瓶颈 4: 无向量化执行 (Vectorization)

```text
问题:
┌─────────────────────────────────────────────────────────────┐
│  当前: 行式处理 (Row-at-a-time)                             │
│                                                              │
│  for row in rows {                                          │
│      process(row);  // 每次处理一行                          │
│  }                                                          │
│                                                              │
│  浪费: CPU 缓存失效，无法利用 SIMD                           │
└─────────────────────────────────────────────────────────────┘

优化方案:
1. 实现 Columnar Format 中间结果
2. Batch 级别处理 (Vectorized Operations)
3. 利用 SIMD 加速
```

### 2.4 性能优化优先级矩阵

| 优化项 | Impact | Effort | Risk | 优先级 |
|--------|--------|--------|------|--------|
| Plan Cache | 高 | 中 | 低 | P1 |
| Batch Prefetch | 高 | 低 | 低 | P1 |
| 增量 Hash Join | 高 | 高 | 中 | P2 |
| 向量化执行 | 高 | 高 | 高 | P3 |
| 并行 Hash Join | 高 | 高 | 高 | P3 |

## 七、总结与建议

### 7.1 核心发现

1. **Beta 门禁 3 大 Blocker 阻塞发布**：
   - B1 Build 编译错误 (duplicate key)
   - B5 Coverage 42.97% 远低于 75% 目标
   - B7 SQL Compat 0% 严重不足

2. **性能瓶颈集中在 Executor 层**：
   - Hash Join 内存独占
   - 无计划缓存导致重复优化
   - I/O 预读不智能

3. **测试覆盖存在结构性缺口**：
   - 43 个覆盖缺口，MERGE 语句 0% 覆盖
   - 窗口函数 6 个实现缺口
   - DCL 语句缺少权限链测试

### 7.2 Top 5 行动建议

| # | 行动 | Impact | Effort | 预期效果 |
|---|------|--------|--------|----------|
| 1 | 修复 B1 Build 编译错误 | 高 | 低 | 解除阻塞 |
| 2 | 补全 execution_chain_regression_test | 高 | 中 | 覆盖率 +15% |
| 3 | 实现 Plan Cache | 高 | 中 | 重复查询延迟 -50% |
| 4 | 集成 SQLancer | 中 | 低 | 自动化正确性检测 |
| 5 | 增强 Batch Prefetch | 中 | 低 | I/O 效率 +20% |

### 7.3 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Build 错误修复复杂 | 高 | 分解依赖，逐步解决 |
| 覆盖率提升工作量大 | 中 | 并行任务分解 |
| 性能优化引入回归 | 中 | 完善性能回归测试 |
| SQLancer 集成周期 | 低 | 使用现有 POC 改造 |
