## 性能分析

### 平均/最小/最大耗时

| 数据库 | 平均耗时 | 最短查询 | 最长查询 | 标准差 |
|--------|----------|----------|----------|--------|
| SQLRustGo | 4879.5 ms | 4258.0 ms (Q13) | 6277.0 ms (Q21) | 576.7 ms |
| SQLite | 1147.0 ms | 0.3 ms (Q21) | 7893.3 ms (Q5) | 1917.6 ms |
| PostgreSQL | **759.6 ms** | 107.5 ms (Q11) | 2948.0 ms (Q3) | 715.9 ms |

### 性能特征分析

#### SQLRustGo
- **无查询胜出**：所有查询都慢于 SQLite 和 PostgreSQL
- **时间稳定**：所有查询耗时 4258-6277ms，波动较小（标准差 576.7ms）
- **查询复杂度无关**：简单查询（如 Q2）和复杂查询（如 Q7）耗时相近
- **推测原因**：SF=10 数据量下，内存存储和执行引擎开销固定约 4.5-5 秒

#### SQLite
- **12/22 最优**：在简单查询和部分复杂查询上表现最好
- **两极分化**：简单查询极快（<10ms），复杂查询极慢（>5s）
  - 最快：Q21 (0.3ms), Q20 (0.4ms), Q9 (9.3ms), Q17 (10.7ms)
  - 最慢：Q5 (7893.3ms), Q11 (5208.8ms), Q7 (2724.7ms)
- **原因**：嵌入式数据库，无复杂优化器，简单查询直接索引，复杂查询全表扫描

#### PostgreSQL
- **10/22 最优**：在复杂 JOIN 查询上表现最好
- **稳定可靠**：所有查询 107-2948ms，无极端值
- **复杂查询优势**：
  - Q7: 122.9ms（比 SQLite 快 22x）
  - Q11: 107.5ms（比 SQLite 快 48x）
  - Q1: 1175.4ms（比 SQLRustGo 快 3.7x）

## 问题与优化建议

### SQLRustGo 问题

1. **查询时间与复杂度无关**
   - 所有查询约 4.3-6.3 秒，无视复杂度
   - **原因**：内存存储 + 执行引擎初始化开销固定

2. **无查询优化**
   - 无法利用索引加速
   - 无查询计划优化

3. **优化建议**：
   - 添加查询计划器和成本估算
   - 实现索引支持（当前仅有全表扫描）
   - 添加结果缓存减少重复计算

### SQLite 问题

1. **复杂查询退化**
   - Q5: 7893ms, Q11: 5208ms
   - **原因**：无复杂 JOIN 优化器

2. **优化建议**：
   - 添加 EXPLAIN QUERY PLAN 分析
   - 对复杂查询添加 hints

## 结论

1. **PostgreSQL** 在 SF=10 规模下综合性能最优（10/22 最快）
2. **SQLite** 在简单查询上表现极佳（12/22 最快）
3. **SQLRustGo** 需要重大优化才能在 SF=10 下与竞品竞争
4. **MySQL** 因认证问题未能测试
