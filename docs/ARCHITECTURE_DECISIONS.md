## ADR-001: 使用 MemoryStorage 作为默认存储引擎

| 字段 | 值 |
|------|-----|
| 日期 | 2026-04-28 |
| 状态 | ✅ 已实施 |
| 作者 | SQLRustGo Team |

### 背景
数据库需要高性能查询引擎，存储层需支持快速读写。

### 决策
选择 `MemoryStorage` 作为默认实现，所有数据在内存中操作。

### 理由
- 开发阶段最快迭代
- QPS 基准可达 10K+
- FileStorage 作为持久化备选

### 后果
- ✅ 开发速度
- ❌ 数据重启后丢失（需 WAL 恢复）
- ❌ 内存限制

## ADR-002: 使用 Git Worktree 隔离开发工作流

| 字段 | 值 |
|------|-----|
| 日期 | 2026-04-30 |
| 状态 | ✅ 已实施 |
| 作者 | Hermes Team |

### 背景
多个 AI Agent 同时开发不同功能，需要隔离工作目录避免冲突。

### 决策
使用 `git worktree` 为每个 Agent 创建独立工作目录。

### 理由
- 每个 Agent 有独立 `target/`
- 无编译缓存冲突
- 分支隔离

### 后果
- ✅ 完全隔离
- ❌ 磁盘空间增加（每个 worktree 一套 target/）

## ADR-003: 门禁体系分四层（R4/R7/R9/GA）

| 字段 | 值 |
|------|-----|
| 日期 | 2026-05-01 |
| 状态 | ✅ 已实施 |
| 作者 | Hermes Team |

### 背景
代码质量需要自动化保证，尤其是性能回归不能被忽略。

### 决策
建立 R4/R7/R9/GA 四层门禁：
- R4: 测试通过
- R7: Clippy + Formatting
- R9: QPS 性能回归检测
- GA: 最终发布门禁

### 理由
- 每层有明确定义的通过条件
- 性能指标受 CI 保护
- 门禁脚本可本地执行

### 后果
- ✅ 质量门禁自动化
- ✅ 性能回归自动告警
- ❌ CI 运行时间增加

## ADR-004: Parser 使用递归下降解析器

| 字段 | 值 |
|------|-----|
| 日期 | 2026-04-28 |
| 状态 | ✅ 已实施 |
| 作者 | SQLRustGo Team |

### 背景
需要解析 SQL 99 标准语法，支持 MySQL 5.7 兼容。

### 决策
选择传统递归下降解析器（手工编写），不使用 Parser Generator。

### 理由
- 完全控制解析过程
- 错误信息更友好
- 无外部依赖
- 对 CTE、窗口函数等语法友好

### 后果
- ✅ 精确错误位置
- ❌ 代码量较大（~5500 行）
- ❌ 新的 SQL 语法需手动添加解析路径

## ADR-005: 使用 SSI 实现 Serializable 隔离级别

| 字段 | 值 |
|------|-----|
| 日期 | 2026-05-01 |
| 状态 | ✅ 已实施（FROZEN） |
| 作者 | Formal Verification Team |

### 背景
需要支持 Serialized 隔离级别以提供强一致性保证。

### 决策
基于 SSI（Serializable Snapshot Isolation）算法实现，使用 TLA+ 模型验证。

### 理由
- SSI 比 2PL 性能更好
- TLA+ 保证无 deadlock
- Phase B S0-S5 已验证

### 后果
- ✅ 无死锁保证
- ⚠️ Write Skew 已知问题（PROOF-026，延至 Phase C）
- ❌ 高并发下性能下降
