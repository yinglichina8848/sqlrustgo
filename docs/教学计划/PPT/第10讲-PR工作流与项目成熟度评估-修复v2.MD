# Part 1: AI 辅助开发的"信任危机"

---

## 1.1 AI 写代码很快，但"敢用"吗？

### 一个真实的困境

```
开发者（打开 AI 工具）：
  "帮我实现一个 JOIN 查询功能"

AI（几秒后）：
  生成了 200 行代码，看起来不错

开发者（犹豫）：
  "代码编译过了，测试也过了..."
  "但是——这段代码在生产环境真的没问题吗？"
  "它会不会在并发场景下出 Bug？"
  "改了 JOIN 实现，会不会影响子查询？"
```

### 为什么会有这种不信任

```
┌─────────────────────────────────────────────────────────┐
│              AI 的上下文窗口（Context Window）            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  AI 能看到：当前的代码文件 + 对话历史                     │
│                                                         │
│  AI 看不到：                                             │
│  • 这个函数被系统中哪些其他模块调用                       │
│  • JOIN 改动后，25 个其他查询会不会受影响                  │
│  • 100 个并发连接下的锁竞争情况                           │
│  • 数据量从 100 行涨到 1000 万行时的性能                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

> **核心矛盾：AI 擅长"写出代码"，但不擅长"保证代码在所有场景下都正确"。**

---

## 1.2 传统解决方案的局限

### 人工 Review 够用吗？

```
传统软件工程：
  人写代码 → 人 Review → 人测试 → 上线
  
  人的优势：理解整体架构、能预判风险
  人的劣势：会疲劳、会遗漏、审查速度跟不上 AI 生成速度

AI 时代的实际情况：
  AI 一天能生成 5000 行代码
  人一天能认真 Review 500 行
  → 缺口 4500 行——这些代码"裸奔"着上线
```

### 需要一个新的答案

```
旧思路：让人更努力地 Review（不太现实）
新思路：让"流程"替人做一部分检查
       ↓
  Harness 工程 = 用软件工程的"外部流程"，补偿 AI 的"内部不确定性"
```

# Part 2: 提示词-上下文-Harness 三层模型

---

## 2.1 What：三层模型是什么

### 一张图理解三层

```
┌─────────────────────────────────────────────────────────┐
│              AI-Native 开发的三层协作模型                  │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  第一层：提示词（Prompt）                                │
│  ────────────────────────                               │
│  开发者说："帮我实现一个 JOIN 查询功能"                   │
│                                                         │
│  关键问题：需求描述清楚了吗？                             │
│  风险：需求模糊 → AI 输出偏差大                          │
│                                                         │
│  ═══════════════════════════════════                    │
│                                                         │
│  第二层：上下文（Context）                               │
│  ────────────────────────                               │
│  AI 看到：当前文件 + 对话历史                            │
│  AI 看不到：调用链、并发场景、性能边界                    │
│                                                         │
│  关键问题：信息够不够？                                  │
│  风险：信息不足 → AI 做"局部最优"但"全局最差"的决定       │
│                                                         │
│  ═══════════════════════════════════                    │
│                                                         │
│  第三层：Harness（约束/检查）        ← 本课程的核心       │
│  ──────────────────────────────────                    │
│  不管谁写的代码（人/AI），合并前必须通过：                 │
│  • 编译检查（能跑）                                      │
│  • 测试通过（功能对）                                    │
│  • 覆盖率达标（测试够）                                  │
│  • 影响分析（没炸到别处）                                │
│  • 回归检测（没变慢）                                    │
│                                                         │
│  关键问题：检查项够不够严格？                             │
│  核心思想：用流程的"确定性"，弥补 AI 的"不确定性"          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 记住这个公式

> **提示词越模糊 → AI 输出越不确定 → Harness 层越重要**

---

## 2.2 Why：为什么需要这个模型

### 一个生活中的类比

```
你让朋友帮你买菜：
  提示词："帮我买点菜" → 朋友买了土豆（你不爱吃）
  上下文：朋友知道你的口味吗？知道家里有什么吗？
  Harness：列出清单"西红柿 2 个、鸡蛋 6 个" → 朋友买不错

软件开发的映射：
  提示词："帮我实现 JOIN" → AI 可能用错了 JOIN 算法
  上下文：AI 不知道你的表有多大、有多少索引
  Harness：Gate 检查——TPC-H 22 个查询全部通过才放行
```

### 这个模型的价值

```
三层模型让你清楚地知道：
  
  问题出在哪一层：
    • AI 生成的代码不符合需求 → 提示词层的问题（需求没说清楚）
    • AI 没考虑并发场景 → 上下文层的问题（信息不够）
    • 代码通过了但上线就崩 → Harness 层的问题（检查不够严）
  
  每层有各自的改进方法：
    • 提示词层 → 学 Prompt Engineering
    • 上下文层 → 用 GitNexus 把全局信息喂给 AI
    • Harness 层 → 加 Gate 检查项 → 本课程的重点
```

---

## 2.3 How：三层协作的实际例子

### 一个 JOIN 功能的开发过程

```
Step 1：提示词层
  开发者写清楚需求（不是"实现 JOIN"，而是...）
  需求：
    • 支持 INNER JOIN 和 LEFT JOIN
    • ON 条件支持等值比较和范围比较
    • 示例 SQL：[给 3 个例子]
  
Step 2：上下文层
  开发者给 AI 提供更多信息
  "这是现有的 Parser 结构：[贴代码]
   这是现有的 Executor 接口：[贴代码]
   请遵循这个模式实现"
  
Step 3：Harness 层
  代码生成后，Gate 自动检查：
    ✓ 编译通过
    ✓ 单元测试通过（覆盖 90% 的代码路径）
    ✓ TPC-H 基准测试通过（22 个标准查询）
    ✓ 回归检测通过（没有让其他查询变慢）
    ✓ GitNexus 影响分析通过（改动只影响 JOIN 相关模块）
```

# Part 3: PR工作流在 Harness 中的角色

---

## 3.1 传统 PR vs Harness PR

```
传统 PR 流程：
  创建 PR → 人工 Review → 讨论 → 合并
  质量保障主要靠：Reviewer 的经验和细心

Harness PR 流程：
  创建 PR → [自动 Gate] → Issue 关联检查 → 人工 Review → 合并
  
  自动 Gate 做了什么（无人值守）：
    ┌──────────────────────────────────────┐
    │  BP1 静态检查（~2分钟）              │
    │  • 编译                                │
    │  • 格式检查                             │
    │  • 代码隐患扫描（Clippy）               │
    ├──────────────────────────────────────┤
    │  BP2 集成检查（~10分钟）             │
    │  • 全部测试通过率 ≥ 90%               │
    │  • 覆盖率 ≥ 75%                       │
    │  • TPC-H 基准测试 22/22               │
    │  • 性能回归检测                        │
    ├──────────────────────────────────────┤
    │  BP3 风险检查（合并前）              │
    │  • GitNexus 影响分析                  │
    │  • 高风险变更自动拦截                  │
    │  • 关键 Invariant 保持验证             │
    └──────────────────────────────────────┘
```

---

## 3.2 PR 描述模板：Gate 驱动的自检

### 提交 PR 前，先问自己这些问题

```markdown
## 我改了 什么？
修复了 JOIN 在大表场景下的内存溢出问题

## 我为什么 这样改？
原来是一次性加载全部数据，改为流式处理

## 我验证了什么？
- [x] cargo build ✅
- [x] cargo test ✅ (91% pass)
- [x] cargo clippy ✅ (零警告)
- [x] 覆盖率检查 ✅ (78% > 75%)

## 我确认了什么？
- [x] GitNexus 影响分析：LOW 风险
- [x] 回归检测：无性能退化

## 关联
Closes #512
```

### 这个模板的价值

```
对 Reviewer：不用从头理解，直接聚焦高风险点
对 AI 工具：结构化的信息更容易被检索和分析
对未来：任何时候回头查，"这个改动的来龙去脉"都清楚
```

---

## 3.3 Issue → PR → Gate → 关闭的闭环

```
提出问题（Issue）
    │  "修复 JOIN 的内存溢出 Bug"
    │
    ▼
实现修复（PR）
    │  PR 描述中写 "Closes #512"
    │
    ▼
Gate 验证
    │  BP1 ✅ → BP2 ✅ → BP3 ✅
    │
    ▼
合并 + Issue 自动关闭
    │
    ▼
知识沉淀
    │  PR 内容 → 文档更新 → 团队学习

闭环的价值：
  任何时候问"这个 Bug 怎么修的？谁修的？验证了什么？"
  → 答案全在 Issue → PR → Gate 的链路里
  → 不用问人，不用翻聊天记录
```

# Part 4: 项目成熟度评估

---

## 4.1 成熟度不是"感觉"，是"数字"

### 为什么量化很重要

```
模糊的成熟度（没用）：
  "我们的项目质量不错"
  "最近没什么大问题"
  "应该快到 Beta 了吧"

量化的成熟度（有用）：
  覆盖率：78%（目标 75%）      ← 达标了吗？达标了
  QPS：1200（目标 1000）       ← 达标了吗？达标了
  TPC-H：22/22（目标 22/22）   ← 达标了吗？达标了
  死锁率：0 次/月（目标 0）     ← 达标了吗？达标了
```

### 为什么 AI 时代更需要量化

```
人类开发：人可以凭经验"感觉"质量
  → "这个改动应该没问题"（基于经验，可能靠谱）

AI 开发：人无法凭直觉判断 AI 的输出
  → "这段 AI 生成的代码可靠吗？"（无法靠直觉）

量化 = 把人从"凭感觉判断"中解放出来
  → 数字说行就行，说不行就不行
  → 不用纠结"我感觉..."
```

## 4.3 Beta Phase 3（BP3）预览

### 从 BP2 到 BP3 的范式转移

```
BP2（功能验证阶段）：
  问题："这个功能实现了吗？"
  方法：写更多测试，提高覆盖率
  思维：量的思维

BP3（风险验证阶段）：
  问题："在生产环境最坏的情况下，系统扛得住吗？"
  方法：针对"最危险场景"设计验证
  思维：质的思维

BP3 的核心验证（后续课程详细展开）：
  • BP3-1: 写入过程中断电 → 数据能恢复吗？（WAL Crash Injection）
  • BP3-2: 两个事务同时改同一行 → 结果正确吗？（MVCC Isolation Matrix）
  • BP3-3: 100 个并发随机锁调度 → 会死锁吗？（Deadlock Fuzzer）
  • BP3-7: 审计记录被篡改 → 能发现吗？（Audit Integrity Verification）
```

> **这一讲建立了 Harness 的"骨架"（PR工作流 + Gate思想），后续每讲会填充"肌肉"（CI/CD、测试、安全、发布）。**

# 核心知识点总结

---

## 1. 三层模型（核心框架）

- **提示词层**：需求描述 → 越清晰越好
- **上下文层**：AI 能看到的信息 → 永远不够
- **Harness 层**：外部检查流程 → 弥补信息不足

## 2. Harness PR 工作流

- **传统 PR**：人工 Review 为主
- **Harness PR**：Gate 自动检查 + Issue 闭环 + 量化指标
- **闭环**：Issue → PR → Gate → 合并 → Issue 关闭

## 3. 成熟度评估

- **量化**：数字说话，不凭感觉
- **演进**：原型 → Alpha → Beta → BP2 → BP3
- **BP3**：从"功能验证"到"风险验证"的范式转移（后续课程展开）
