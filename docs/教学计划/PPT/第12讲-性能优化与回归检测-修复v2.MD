# Part 1: AI 性能优化的"局部最优"陷阱

---

## 1.1 AI 能看到什么 vs 看不到什么

```
开发者："帮我优化这个查询的性能"
AI："好的，加个索引就好了"

AI 看到的：
  当前这个查询 → 加索引可以加速 ✓

AI 看不到的（上下文盲区）：
  • 这个表主要是写入，索引会拖慢 INSERT
  • 其他 20 个查询也依赖这个表，索引改变了执行路径
  • 索引额外占用 2GB 磁盘空间
  • 并发场景下索引维护的锁竞争

结果：AI 做的是"局部最优"，但可能是"全局最差"
```

---

## 1.2 性能优化的三层验证

### Harness 如何补偿 AI 的盲区

```
Layer 1：单查询验证（AI 能做）
  这个查询变快了吗？→ 用 cargo bench 测

Layer 2：全基准验证（需要 Harness）
  所有其他查询受影响了吗？→ 用 TPC-H 22个查询全部跑一遍

Layer 3：真实负载验证（最难）
  并发 100 个连接下表现如何？→ 用 sysbench 压测

AI 做 Layer 1，Harness 做 Layer 2 + Layer 3
```

---

## 1.3 提示词-上下文-Harness 在性能中的映射

```
提示词层："优化这个查询" → 需求模糊，AI 可能乱优化
上下文层：GitNexus 展示"哪些查询依赖这个表"
Harness 层：
  • TPC-H 全量验证 → 任何一个查询变慢就被发现
  • check_regression.sh → 自动对比优化前后性能
  • 量化阈值：性能退化 ≤ 5% 可接受
```

# Part 2: TPC-H — 数据库的"标准体检"

---

## 2.1 TPC-H 是什么（通俗版）

### 一个类比

```
人的体检：
  身高体重、血压血糖、心电B超...一系列标准检查
  → 医生看报告："整体健康"

数据库的"体检" = TPC-H：
  22 个标准 SQL 查询，模拟真实商业分析场景
  → 全部通过 = 数据库"功能完整 + 性能达标"
```

### TPC-H 的 22 个查询

```
TPC-H 模拟一个"零件供应商"的商业数据仓库：
  8 张表：客户、订单、零件、供应商...
  22 个查询：从"这个季度收入多少"到"全球哪些客户贡献最大"
  
SQLRustGo TPC-H SF=1 结果：
  Parser：22/22 解析成功 ✅
  Executor：22/22 可执行 ✅
  （SF=1 = 数据量约 1GB，包含 600 万行 Lineitem 表）
```

---

## 2.2 TPC-H 在 Gate 中的角色

```
BP2 Gate 的 TPC-H 检查：
  生成 SF=1 测试数据 → 执行 22 个查询 → 验证结果

通过标准：
  ✅ 所有查询成功执行
  ✅ 无超时（单个查询 ≤ 30 秒）
  ✅ 结果正确

任一失败 → Gate Fail → PR Blocked
```

# Part 3: 回归检测 — 防止"优化变劣化"

---

## 3.1 什么是回归

```
功能回归：
  昨天能跑的测试，今天跑不过了 ← 你把功能改坏了！

性能回归：
  昨天查询 2 秒，今天查询 8 秒 ← 性能退化 400%！

覆盖回归：
  昨天覆盖率 78%，今天 65% ← 新代码没写测试
```

---

## 3.2 check_regression.sh 的工作原理

### 自动化性能对比

```
1. 读取基准数据（上次测试的性能数据）
2. 运行当前版本的测试
3. 对比：有没有变慢？

结果：
  concurrent_select_8t: 基准 200K QPS → 当前 88K QPS → Δ = -56% ❌
  simple_select: 基准 400K QPS → 当前 398K QPS → Δ = -0.5% ✅

检查逻辑：
  Δ ≤ 5% → Pass
  Δ > 5% → Fail → PR Blocked
```

### 缓存陷阱

```
--skip-run 标志：
  不实际运行测试，直接用缓存数据 → 结果永远是 Δ = 0%

风险：
  缓存是旧的 → 实际性能已经退化 → 但报告说"一切正常"

教训：
  --skip-run 只能用于快速调试
  Gate 决策必须真实运行测试
```

# Part 4: 自优化闭环 — CMM-5 的实操

---

## 4.1 一个真实的回归案例

### concurrent_select_8t 性能退化 56%

```
发现：
  check_regression.sh 真实运行 → concurrent_select_8t Δ = -56%

根因分析（用 GitNexus）：
  改动影响了 B+Tree 的锁粒度 → 并发锁竞争加剧

修复流程（完整的自优化闭环）：
  1. Gate 检测到回归 → PR Blocked
  2. 自动创建 Issue 记录问题
  3. 开发者用 GitNexus 定位根因（"这个改动影响了哪些模块"）
  4. 修复代码 + 测试
  5. 新 PR → 重新 Gate → 通过 → 合并
  6. Issue 自动关闭
  7. 知识沉淀到 GBrain SSOT（Episode："concurrent_select_8t 56% 回归事件"）
```

---

## 4.2 这就是 CMM-5

### CMM-5 不是"达到了"，而是"一直在做"

```
CMM-4（已管理）：
  量化指标被定义和监控 ✓

CMM-5（优化中）：
  系统自己在改进自己 ✓

自优化闭环的四个步骤（PDCA）：
  
  Plan（计划）    →  下个版本目标：死锁率降到 0
  Do（执行）      →  Deadlock Fuzzer 发现死锁 → 修复
  Check（检查）   →  "本月死锁 0 次" VS "上月 5 次"
  Act（改进）     →  把 Deadlock Fuzzer 加入永久 BP3 检查项
       │
       └──────────→ 回到 Plan，开始下一轮
```

### 自审计：Gate 检视自己

```
不只是"用 Gate 检查代码"，
还要"用数据检查 Gate 本身"：

  • BP2 经常失败 → BP1 不够严 → 加强 BP1
  • 某项检查 3 个月零失败 → 是不是阈值太低了？→ 考虑收紧
  • BP3 发现 BP2 漏掉的问题 → BP2 有盲区 → 新增 BP2 检查项

这就是"用流程的自我审视，实现流程的自我进化"
```

# 核心知识点总结

---

- **AI 的局部最优陷阱**：只看到当前查询，看不到系统全局
- **TPC-H 基准**：22 个标准查询 = 数据库的"全身体检"
- **回归检测**：check_regression.sh 自动对比，Δ > 5% 就拦截
- **自优化闭环**：发现 → 根因分析 → 修复 → 验证 → 知识沉淀 → 流程优化
- **CMM-5 精髓**：不只是"量化管理"，更是"系统自己进化"
