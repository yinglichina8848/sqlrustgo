# Part 1: 问题 —— AI 写 JOIN，你敢用吗？

---

## 1.1 一个场景

```
星期一早上，开发者打开 Claude Code：

  "帮我实现 JOIN 查询功能，支持 INNER JOIN 和 LEFT JOIN"

AI（10秒后）：
  生成了 350 行 Rust 代码
  • HashJoinExec —— JOIN 的物理执行器
  • JoinPlanner —— JOIN 顺序规划器  
  • 对应的 Parser 修改

开发者（看了一遍代码）：
  "逻辑好像是对的...编译也过了...测试也写了..."
  
  "但是——"
  
  "这代码在 5 张表 JOIN 的时候会不会炸？"
  "它选的 JOIN 顺序是最优的吗？"
  "改了 Parser，有没有影响子查询？"
  "在生产环境跑，100 个并发同时 JOIN，内存够吗？"
```

---

## 1.2 为什么会有这种不信任

```
┌─────────────────────────────────────────────────────────┐
│              AI 的上下文窗口（Context Window）            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  AI 能看到：                                             │
│    当前文件 + 对话历史 + 你喂给它的代码片段               │
│                                                         │
│  AI 看不到：                                             │
│    ✗ 这个 HashJoinExec 被系统中哪些查询调用              │
│    ✗ TPC-H Q5 是 5 表 JOIN，数据量 150 万行             │
│    ✗ 贪心算法在 3 表以下 OK，5 表以上容易选错顺序        │
│    ✗ 并发 100 个 JOIN 查询时内存会不够                   │
│    ✗ 上次有人"优化"了哈希函数，导致小表 JOIN 慢了 30%    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

> **AI 写代码 = 在有限信息下做决策。信息越不够，决策越不可靠。**

---

## 1.3 传统方式怎么解决？—— 靠人 Review

```
传统软件工程：
  人写代码 → 人 Review → 人测试 → 上线

在 AI 时代的问题：
  AI 一天能生成 5000 行代码
  人一天能认真 Review 500 行
  → 4500 行"裸奔"着上线
  → 人的注意力根本跟不上 AI 的速度
```

---

## 1.4 我们需要的答案

```
旧思路：让人更努力地 Review（不现实）
新思路：让"流程"替人做一部分检查
       ↓
  Harness 工程 = 用软件工程的"外部约束"，
               补偿 AI 的"内部不确定性"

类比：AI 是司机，Harness 是交规 + 驾考 + 电子眼
      不是让司机更小心（人都会累），
      而是让违规行为一定被检测到
```

# Part 2: 框架 —— 提示词-上下文-Harness 三层治理

---

## 2.1 三层模型全景

```
┌─────────────────────────────────────────────────────────────┐
│              AI-Native 开发的三层治理模型                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第一层：提示词（Prompt）                                    │
│  ────────────────────────                                   │
│  开发者说："帮我实现 JOIN 查询功能"                          │
│                                                             │
│  治理问题：需求说清楚了吗？                                  │
│  风险：需求模糊 → AI 自由发挥 → 方向跑偏                    │
│                                                             │
│  ═══════════════════════════════════                        │
│                                                             │
│  第二层：上下文（Context）                                   │
│  ────────────────────────                                   │
│  AI 看到：当前文件 + 对话历史 + 你贴的代码                   │
│  AI 看不到：调用链、并发场景、历史教训、全局性能影响         │
│                                                             │
│  治理问题：信息缺口有多大？                                  │
│  风险：信息不足 → AI 的"局部最优"可能是"全局最差"            │
│                                                             │
│  ═══════════════════════════════════                        │
│                                                             │
│  ★ 第三层：Harness（约束层）—— 本讲的核心                    │
│  ─────────────────────────────────                          │
│  不管谁写的代码（人/AI），合并前必须通过：                    │
│                                                             │
│  BP1 静态约束（~2分钟）                                      │
│    编译 / 格式 / 代码隐患扫描                                │
│    "代码至少是规范的"                                        │
│                                                             │
│  BP2 行为约束（~10分钟）                                     │
│    全量测试 / 覆盖率 / TPC-H基准 / 性能回归                  │
│    "代码在标准场景下行为正确"                                │
│                                                             │
│  BP3 风险约束（~30分钟）                                     │
│    GitNexus影响分析 / 崩溃恢复 / 并发死锁 / 审计完整性       │
│    "代码在最坏情况下不会出大问题"                            │
│                                                             │
│  核心理念：用流程的"确定性"，弥补 AI 的"不确定性"            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 记住这个公式

> **提示词越模糊 → AI 输出越不确定 → Harness 层必须越严格**

---

## 2.2 三层如何协作：JOIN 开发的完整链条

```
Step 1: 提示词层
  开发者写清楚：
    "实现 JOIN，支持 INNER/LEFT/RIGHT，
     参考现有的 Executor 接口，
     需要处理 5 表以上的 JOIN 场景，
     TPC-H Q5 必须在 30 秒内完成"

Step 2: 上下文层
  开发者给 AI 提供信息：
    "这是现有的 Executor 接口：[贴代码]
     这是 Parser 中 AST 的定义：[贴代码]
     GBrain SSOT 里上次 JOIN 优化的教训：[Episode引用]"

Step 3: Harness 层（自动触发）
  AI 生成的代码 → 提交 PR →
    BP1: "编译过了吗？格式对吗？" ✅
    BP2: "测试过了吗？TPC-H 5表JOIN能在30秒内吗？" ❌ → PR被拦截
    GitNexus: "这个改动影响了哪些模块？" → 报告:"影响3个执行流"
    GBrain: "上次类似的优化是怎么做的？" → 返回知识

Harness 不是"帮 AI 写代码"，
而是"告诉 AI：你现在写的还不够好，问题在这里"。
```

---

## 2.3 一个类比帮你记住三层

```
你让朋友帮你买菜：

  提示词："帮我买点菜" 
    → 朋友买了土豆（你不爱吃！）
    → 问题：需求没说清楚

  改进提示词："买西红柿 2 个、鸡蛋 6 个"
    → 朋友买对了种类
    → 但不知道你冰箱里已经有 12 个鸡蛋
    → 问题：上下文不够

  加入 Harness（清单+检查）：
    出门前核对清单："西红柿2个 ✅ | 鸡蛋6个 ✅ | 不要土豆 ✅"
    买完回来检查："鸡蛋有裂缝吗？西红柿新鲜吗？"
    → 需求清晰 ✅ + 上下文完整 ✅ + 结果验证 ✅
```

# Part 3: 实战 —— Harness 如何约束 AI 走向正确

---

## 3.1 场景初始化

```
时间：星期一
事件：开发者提交了 PR #500 "实现 JOIN 功能"
代码：AI 生成的 HashJoinExec + JoinPlanner，共 420 行
PR 描述："实现了 INNER JOIN 和 LEFT JOIN，写了 8 个单元测试"

接下来，Harness 接管——
```

---

## 3.2 Gate 1：BP1 静态约束（"像样的代码"）

```
PR #500 自动触发 BP1 检查：

  ┌──────────────────────────────────────┐
  │  cargo build --all-features          │
  │  → 编译成功 ✅                        │
  │                                      │
  │  cargo fmt --check                   │
  │  → 格式通过 ✅                        │
  │                                      │
  │  cargo clippy --all-features         │
  │  → 零警告 ✅                          │
  │                                      │
  │  文档链接检查                         │
  │  → 无失效链接 ✅                      │
  ├──────────────────────────────────────┤
  │  BP1 结果: PASS (~1.5分钟)           │
  └──────────────────────────────────────┘

这一步的意义：
  保证了 AI 写出来的代码"至少是规范的"。
  格式统一 → 团队不会因为格式问题浪费 Review 时间。
  零 Clippy 警告 → AI 没有引入已知的代码隐患模式。
```

---

## 3.3 Gate 2：BP2 行为约束（"功能正确 + 性能达标"）

```
PR #500 继续进入 BP2：

  cargo test --all-features
  → 通过率 94% ✅
  → 8 个单元测试通过，2 个集成测试通过

  覆盖率检查
  → 78%（目标 ≥75%）✅

  TPC-H SF=1 基准测试
  → Q1（单表聚合）   0.68秒 ✅
  → Q3（3表JOIN）    2.1秒  ✅
  → Q5（5表JOIN）   31.2秒  ❌ ← 超时！（阈值 30 秒）
  → Q9（5表JOIN）   28.5秒  ⚠️ ← 接近超时

  ┌──────────────────────────────────────┐
  │  BP2 结果: FAIL                      │
  │  原因: TPC-H Q5 超时 (31.2s > 30s)   │
  │  PR #500 被 Blocked                   │
  └──────────────────────────────────────┘
```

### 关键：BP2 不是"说你不行"，而是"告诉你哪里不行"

```
为什么单元测试都过了，TPC-H 却没过？

因为单元测试是"2张表 × 100行"的简单场景
TPC-H Q5 是"5张表 × 数十万行"的真实场景

AI 在写代码时，上下文里只有"2张表的小测试"
它不知道实际生产环境是"5张表的大数据"

Harness 的价值：
  用外部标准（TPC-H）给 AI 提供了一个"全场景验证"
  不管 AI 上下文里有什么、没什么，
  TPC-H 22 个查询全部通过 = 你的实现在所有标准场景下都 OK
```

---

## 3.4 Blocker 分析：为什么 Q5 超时？

### GitNexus 上场：影响范围分析

```
开发者问 GitNexus：
  "Q5 查询涉及哪些代码路径？"

GitNexus 返回：
  Q5 执行流（execution flow）:
    Parser → Planner → JoinPlanner::build_join_plan()
    → HashJoinExec (orders) → HashJoinExec (customer)
    → HashJoinExec (supplier) → HashJoinExec (nation)
    → HashJoinExec (region)

  直接调用者：3 个执行流依赖 JoinPlanner
  影响范围：如果修改 JoinPlanner 的排序逻辑，
            会影响所有多表 JOIN 查询

GitNexus 的价值：
  不是"帮 AI 找 Bug"
  而是"帮人理解 AI 写的代码影响了什么"
  
  AI 不知道"改了这行代码会影响 3 个执行流"
  GitNexus 知道 → 告诉开发者 → 开发者验证
```

---

## 3.5 根因定位：不是算法错了，是算法"不够好"

### AI 写的贪心 JOIN 排序有什么问题？

```
AI 写的 JoinPlanner（贪心算法）的逻辑：

  "从 FROM 的第一张表开始，每次找一个'能连上'的表加入"

这个逻辑在小表场景（3表以下）完全 OK。
但在 Q5 的 5 表场景：

  orders (150万行) ⋈ customer (15万行) → 产生 150万行中间结果
  再 ⋈ supplier (1万行) → 继续在 150万行上操作
  再 ⋈ nation (25行) → 已经在巨大的中间结果上了

问题：
  贪心算法没有考虑"表的大小"
  把最大的表放在最前面 JOIN，
  导致中间结果从一开始就很大，
  后续每一步都在处理海量数据。
```

### 这恰恰是 AI 容易犯的错误

```
AI 看到的是："我实现了一个贪心算法，逻辑正确"
AI 看不到的是："这个算法在 5 表场景下，因为没考虑表大小，
              选了一个糟糕的 JOIN 顺序"
              

不是 AI "写错了"，而是 AI "没写够好"。
不是功能错误，而是工程不足。

这就是为什么 Gate 不能只检查"测试过没过"，
还要检查"在真实数据量下表现如何"。
```

---

## 3.6 修复与重新验证

### 开发者的修改（加入表大小约束）

```
开发者（不是 AI）做的改进：

  在贪心选择时，加入一条简单规则：
  "在能连上的表中，优先选行数最少的表"

这不需要重写算法，只需要在选择时多一个排序条件。

效果：
  新 JOIN 顺序：region(5行) ⋈ nation(25行) ⋈ supplier(1万)
               ⋈ customer(15万) ⋈ orders(150万)
  → 从小表开始，中间结果始终很小
  → Q5: 31.2秒 → 3.1秒 （优化 10 倍！）
```

### 重新 Gate

```
PR #501（修复后）提交：

  BP1: ✅ 编译/格式/Clippy
  BP2: ✅ cargo test (93% pass)
  BP2: ✅ 覆盖率 78%
  BP2: ✅ TPC-H SF=1
       Q1: 0.68s  Q3: 2.1s  Q5: 3.1s ✅  Q9: 5.2s ✅
  BP2: ✅ 回归检测：无性能退化

PR #501 合并 → Issue #X 自动关闭
```

---

## 3.7 修复之后：知识沉淀

### 不是"修完就完了"

```
修复完成后，Harness 还要求做三件事：

1. 更新基准数据（current.json）
   → 把新的 Q5=3.1s 写入基准
   → 以后如果 Q5 退回到 6 秒以上，Gate 会拦截
   → "让 Gate 记住什么是'好'"

2. 记录知识到 GBrain SSOT
   → 创建 Episode: "Q5 JOIN 顺序优化 —— 从31.2s到3.1s"
   → 创建 Pattern: "优先 JOIN 小表的贪心策略"
   → 创建 Fact: "TPC-H Q5 基线 = 3.1s"
   
   意义：下次另一个开发者（或 AI Agent）做 JOIN 优化时，
        查 GBrain："上次 JOIN 优化有什么教训？"
        → GBrain 返回这条 Episode → 不再踩同样的坑

3. 评估：Gate 本身需要改进吗？
   → "TPC-H 所有查询都通过了，但 Q5 在 30s 阈值边缘"
   → "如果下次 Q5 从 28s 退到 29.5s，还是'通过'的"
   → "要不要加一条：性能比基准退化 > 20% 就触发警告？"
```

# Part 4: 闭环 —— Harness 的自我进化

---

## 4.1 完整的治理闭环

```
┌─────────────────────────────────────────────────────────────┐
│              Harness 治理闭环（以 JOIN 为例）                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① Issue 创建                                               │
│     "需要实现 JOIN 功能"                                     │
│          │                                                  │
│          ▼                                                  │
│  ② AI 生成代码 + PR 提交                                    │
│     HashJoinExec + JoinPlanner，420 行                       │
│          │                                                  │
│          ▼                                                  │
│  ③ Harness Gate 执行                                        │
│     BP1 ✅  BP2 ❌（Q5超时）                                 │
│     GitNexus 影响分析 → 发现瓶颈                             │
│          │                                                  │
│          ▼                                                  │
│  ④ 人工修复（人做决策，AI 辅助）                             │
│     加入表大小约束 → Q5 从 31.2s 降到 3.1s                  │
│          │                                                  │
│          ▼                                                  │
│  ⑤ 重新 Gate → 全部通过 → PR 合并                           │
│          │                                                  │
│          ▼                                                  │
│  ⑥ 知识沉淀                                                 │
│     基准更新 + GBrain Episode + Gate 规则优化               │
│          │                                                  │
│          └──→ 回到 ①（下一个 Issue 有了更多的知识支持）      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 4.2 Harness 各组件在闭环中的角色

| 组件 | 在 JOIN 案例中的角色 | 治理价值 |
|------|---------------------|---------|
| **BP1 Gate** | 检查编译/格式/Clippy | 过滤 AI 的低级错误（拼写、格式） |
| **BP2 Gate** | TPC-H Q5 发现超时 | 用真实数据量验证 AI 的代码 |
| **GitNexus** | 定位影响范围（3个执行流） | 弥补 AI 对全局调用链的盲区 |
| **回归检测** | 确认修复没让其他查询变慢 | 防止"修一个、坏一片" |
| **GBrain SSOT** | 沉淀 Episode + Pattern | 让下次的 AI（和人）不重复踩坑 |
| **Gate Spec** | 自身优化：增加退化警告 | 让 Gate 系统自己进化 |

---

## 4.3 这个闭环的本质

```
不是"AI 写代码 → Gate 检查 → 通过/失败"的一次性流程

而是"AI 写代码 → Gate 发现 AI 的盲区 → 
      人修复 → 知识沉淀 → 
      Gate 自己变强 → 
      下次 AI 遇到类似问题有更多上下文"

每一次 Gate 失败都不是"浪费"，
而是在训练一个越来越聪明的治理系统。
```

---

## 4.4 三层模型在这个案例中的最终映射

```
┌─────────────────────────────────────────────────────────────┐
│          三层模型在 JOIN 案例中的体现                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  提示词层：                                                  │
│    "实现 JOIN 功能" → 需求没说"5表JOIN要在30秒内"          │
│    治理改进：PR 模板要求写清楚性能目标                       │
│                                                             │
│  上下文层：                                                  │
│    AI 看到：2表测试 → 贪心算法 OK                            │
│    AI 没看到：5表场景 → 贪心算法选错顺序                     │

│                                                             │
│  Harness 层：                                                │
│    TPC-H Gate → 发现 Q5 超时（AI 不知道的事被发现了）        │

│        AI 每犯一次错，Gate 就多学一点，下次就更聪明           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

# Part 5: 度量与展望

---

## 5.1 项目成熟度：从"能跑"到"可信"

### 成熟度演进

```
v1.0 原型  →  代码能跑就行，测试靠手动
v2.0 Alpha →  引入 CI，编译+测试自动化
v2.9 Beta  →  Gate 系统，量化指标管控
v3.0 BP2   →  功能验证完成（TPC-H 22/22，覆盖率 78%）
v3.0 BP3   →  风险验证进行中（Crash注入、MVCC隔离、死锁Fuzz）
```

### 为什么不满足于"TPC-H通过了"

```
BP2 证明了 "功能在标准场景下正确"
BP3 要证明 "系统在最坏情况下也不出大问题"

BP2 的问题：          BP3 的问题：
  Q5 能跑通吗？        写入过程中断电了，数据会损坏吗？
  覆盖率达标吗？        两个事务同时改同一行，结果正确吗？
  SQL 兼容吗？         100 个并发随机锁调度，会死锁吗？
                      审计记录被篡改，能发现吗？

从 BP2 到 BP3 = 从"功能验证"到"可信度证明"的范式转移。
```

## 5.2 本节核心知识点总结

---

## 1. 三层治理模型

- **提示词层**：需求 → 越清晰，AI 越不容易跑偏
- **上下文层**：AI 的信息缺口 → 永远存在，用 Harness 补偿
- **Harness 层**：Gate 系统 → 用流程的确定性约束 AI 的不确定性

## 2. Harness 在 JOIN 案例中的作用

- **BP1**：过滤 AI 的低级错误（格式、拼写、隐患模式）
- **BP2（TPC-H）**：发现 AI 不知道的"大数据量性能问题"
- **GitNexus**：弥补 AI 对全局调用链的盲区
- **回归检测**：防止"修一个、坏一片"
- **GBrain**：让每次教训变成下次的上下文

## 3. Harness 的自我进化

- 不只是"Gate 检查代码"
- 更是"从 Gate 的反馈中学习，让 Gate 自己变强"
- 每次 Gate 失败 → 分析根因 → 优化规则 → 知识沉淀
- 这就是 CMM-5（优化级）的核心
