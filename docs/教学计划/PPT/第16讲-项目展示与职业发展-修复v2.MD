# Part 3: AI-Native 工程师能力模型

---

## 3.1 能力金字塔

```
                    ▲
                   /│\
                  / │ \
                 /  │  \
                /   │   \
               /    │    \
              /     │     \
             ┌──────┴──────┐
             │  架构设计    │  ← 能设计系统、做技术选型
             ├─────────────┤
             │  流程工程    │  ← 能设计 Gate/CI/CD、量化管理
             ├─────────────┤
             │  代码实现    │  ← 能写代码、能测试、能优化
             ├─────────────┤
             │  AI 协作     │  ← 会用 AI 辅助、会审查 AI 输出
             └─────────────┘
```

### 各层详解

```
第一层：AI 协作
  • 会用 AI 写代码（Prompt Engineering）
  • 会审查 AI 的代码（不是"看起来对"，是"验证它对"）
  • 知道 AI 的边界（什么时候 AI 靠谱，什么时候必须人判断）

第二层：代码实现（基本功）
  • 语言能力（Rust/C++/Java 精通至少一门）
  • 测试能力（不只是写测试，而是设计测试策略）
  • 优化能力（定位瓶颈 + 定量验证优化效果）

第三层：流程工程（AI 时代的核心能力）
  • 设计 Gate 系统（"什么检查、什么标准、失败了怎么办"）
  • 量化管理（"数字说话，不凭感觉"）
  • 风险驱动验证（"测最危险的地方，不是测最多的地方"）

第四层：架构设计
  • 系统架构（模块划分、接口设计、技术选型）
  • 权衡决策（性能 vs 可维护性、简单 vs 灵活）
```

---

## 3.2 Harness 工程专家：一个新兴角色

### 这个角色做什么

```
传统角色分工：
  开发 → 写代码
  测试 → 找 Bug
  运维 → 管服务器

Harness 工程专家（跨三个领域）：
  设计自动化流程 → 让代码质量不依赖"人细心"
  设计量化指标 → 让"好不好"变成"数字"
  设计验证矩阵 → 让"最危险场景"被优先覆盖
```

### 为什么这个角色 AI 难以替代

```
AI 能做：执行（写代码、运行测试）
AI 难做：设计 & 判断

Harness 工程专家的核心工作 = 设计 + 判断：
  "什么检查是必须的？"
  "阈值为 75% 还是 85%？为什么？"
  "这个风险可以豁免吗？"
  "流程怎么改进？"
```

## 4.2 Harness 思维的迁移价值

### 不是只有 SQLRustGo 需要

```
你在任何项目中都能应用 Harness 思维：

  Web 应用项目：
    Gate = 部署前检查（lint + test + 安全扫描 + 性能基准）
    量化 = "API 响应时间 ≤ 200ms，可用性 ≥ 99.9%"

  AI 产品项目：
    Gate = Prompt 变更前检查（回归测试 + 输出质量评估）
    量化 = "回复准确率 ≥ 95%，有害内容过滤率 100%"

  数据分析项目：
    Gate = 数据管道检查（Schema 一致性 + 数据量偏差 ≤ 5%）
    量化 = "数据延迟 ≤ 1小时，完整率 ≥ 99.5%"

核心不变的是：
  定义 Gate → 量化指标 → 自动化执行 → 闭环改进
```

## 4.4 从这门课带走什么（三条建议）

```
1. 用 GitNexus 思维看代码
   "改这个函数 → 先查谁在调用它 → 评估影响范围"
   这个习惯会让你少写 80% 的 Bug

2. 用 Gate 思维定义"完成"
   "任务完成 = 代码 + 测试 + Gate 通过 + Issue 关闭"
   不只是"写完了"，而是"验证过了，可以发布"

3. 用 GBrain 思维管理知识
   不要让知识散落在聊天记录里
   养成"遇到坑 → 记录 → 索引 → 以后能查到"的习惯
```

# Part 5: 课程总结

---

## 最重要的三句话

```
1. AI 擅长"做"，不擅长"保证做了不出问题"
   → Harness 工程 = 用流程补偿 AI 的不足

2. 没有量化，就没有管理
   → "覆盖率 78%" 比 "质量不错" 有用 100 倍

3. 系统必须自我进化
   → 不只是"用 Gate 检查代码"
   → 还要"用数据优化 Gate 本身"
```
