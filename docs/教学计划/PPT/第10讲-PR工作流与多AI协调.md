# Part 1: PR工作流：AI协作的"交通规则"

---

## 1.1 为什么PR是多AI协作的核心？

### 没有PR的后果

```
┌─────────────────────────────────────────────────────────────────────┐
│                    没有PR = 灾难                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  AI #1 直接 push 到 main：                                        │
│  ├── 无人审查 → 错误代码直接上线                                   │
│  ├── 风格混乱 → 代码不可读                                         │
│  ├── 测试缺失 → 上线就崩溃                                         │
│  └── 责任不清 → 不知道谁的问题                                     │
│                                                                      │
│  结果：系统崩溃，用户流失，团队背锅                                  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 有PR的好处

```
┌─────────────────────────────────────────────────────────────────────┐
│                    有PR = 安全保障                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  AI #1 提交 PR → 自动化检查 → 人工审查 → 合并                     │
│       │              │              │                             │
│    发起请求       CI验证        把关                               │
│       │              │              │                             │
│    记录变更      质量门槛      知识传递                           │
│                                                                      │
│  结果：代码质量有保证，问题可追溯，团队共同进步                     │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

---

## 1.2 PR工作流详解

### 完整流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    PR工作流                                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  1. 创建功能分支                                                    │
│     git checkout -b feature/ai-parser-add-join                      │
│                                                                      │
│  2. 开发功能（AI工作）                                             │
│     AI #1 编写代码、编写测试                                       │
│                                                                      │
│  3. 提交代码                                                       │
│     git add . && git commit -m "feat: add JOIN support"             │
│                                                                      │
│  4. 推送分支                                                       │
│     git push origin feature/ai-parser-add-join                     │
│                                                                      │
│  5. 创建PR ← 关键步骤！                                            │
│     GitHub: Compare & pull request                                 │
│                                                                      │
│  6. CI自动检查                                                     │
│     cargo build + test + clippy + fmt                              │
│                                                                      │
│  7. 代码审查（人类或AI）                                           │
│     检查逻辑、风格、测试                                            │
│                                                                      │
│  8. 修改（如需要）                                                  │
│     AI根据反馈修改代码                                              │
│                                                                      │
│  9. 合并                                                           │
│     Squash merge / Merge commit / Rebase                            │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 1.3 PR的要素

### PR标题规范

```
┌─────────────────────────────────────────────────────────────────────┐
│                    PR标题格式                                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  type(scope): description                                          │
│                                                                      │
│  type:                                                             │
│  ├── feat     - 新功能                                             │
│  ├── fix      - Bug修复                                            │
│  ├── refactor - 重构                                               │
│  ├── docs     - 文档更新                                           │
│  ├── test     - 测试更新                                            │
│  ├── chore    - 杂项                                               │
│                                                                      │
│  示例：                                                             │
│  ├── feat(parser): add JOIN support                                 │
│  ├── fix(storage): resolve memory leak                             │
│  └── refactor(executor): optimize hash join                        │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### PR描述模板

```markdown
## 变更类型
- [ ] 新功能
- [ ] Bug修复
- [ ] 重构
- [ ] 文档更新

## 变更说明
[详细描述本次变更]

## 影响范围
- 修改了哪些模块？
- 对其他AI的工作有影响吗？
- 需要其他AI配合吗？

## 测试说明
- 单元测试：✅ 添加了xxx测试
- 集成测试：✅ 通过了xxx场景

## AI协作说明
- 负责AI: AI #1 (Parser)
- 依赖关系: 无
- 冲突风险: 低

## 相关Issue
Closes #123
```

# Part 2: 多AI Agent协调机制

---

## 2.1 AI Agent的角色分配

### 典型的多AI分工

```
┌─────────────────────────────────────────────────────────────────────┐
│                    多AI协作架构                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│                       ┌─────────────┐                               │
│                       │  任务分配器  │ ← 人类或AI Agent协调器       │
│                       └──────┬──────┘                               │
│                              │                                      │
│          ┌──────────────────┼──────────────────┐                  │
│          │                  │                  │                  │
│          ▼                  ▼                  ▼                  │
│   ┌────────────┐    ┌────────────┐    ┌────────────┐             │
│   │  AI #1     │    │  AI #2     │    │  AI #3     │             │
│   │  Parser    │    │  Executor  │    │  Storage   │             │
│   │  开发      │    │  开发      │    │  开发      │             │
│   └─────┬──────┘    └─────┬──────┘    └─────┬──────┘             │
│         │                  │                  │                    │
│         └──────────────────┼──────────────────┘                    │
│                            ▼                                         │
│                    ┌─────────────┐                                  │
│                    │  PR合并     │ ← 所有AI代码汇合               │
│                    │  CI检查     │                                  │
│                    │  人工审查   │                                  │
│                    └─────────────┘                                  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 协调原则

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| 分工明确 | 每个AI有明确职责 | 分支隔离 |
| 接口稳定 | 模块间通过固定接口 | API文档 |
| 依赖管理 | 明确谁依赖谁 | Issue关联 |
| 冲突处理 | 合并时的协调机制 | PR审查 |

---

## 2.2 AI Agent之间的通信

### 通过PR进行通信

```
┌─────────────────────────────────────────────────────────────────────┐
│                    AI间通信示例                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  AI #1 (Parser) 创建PR:                                           │
│  ─────────────────────                                              │
│  ## 我修改了什么                                                    │
│  - 添加了JOIN语法支持                                               │
│  - 新增JoinOperator结构体                                           │
│                                                                      │
│  ## 我需要什么                                                      │
│  - 需要AI #2 (Executor) 实现JoinOperator的执行                       │
│  - 需要AI #3 (Storage) 支持Join的数据读取                          │
│                                                                      │
│  ## 关键接口                                                        │
│  ```rust                                                           │
│  pub trait JoinExecutor {                                          │
│      fn execute(&self, left: RecordSet, right: RecordSet)          │
│          -> Result<RecordSet>;                                      │
│  }                                                                 │
│  ```                                                                │
│                                                                      │
│  AI #2 看到PR后，开始实现...                                       │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 协调机制

| 场景 | 协调方式 |
|------|---------|
| A需要B的接口 | 在PR中说明，@相关AI |
| A改了B的依赖 | 通知B需要同步修改 |
| 合并冲突 | 通过PR讨论谁的决定正确 |
| 依赖缺失 | 在PR中创建Issue关联 |

---

## 2.3 AI Agent的冲突预防

### 预防策略

```
┌─────────────────────────────────────────────────────────────────────┐
│                    冲突预防措施                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  1. 分支隔离                                                        │
│     ├── 每个AI独立分支                                              │
│     └── 避免同时改同一文件                                          │
│                                                                      │
│  2. 接口约定                                                        │
│     ├── 先定义接口再开发                                            │
│     ├── 接口变更需要通知                                            │
│     └── 遵守语义版本                                                │
│                                                                      │
│  3. 频繁同步                                                        │
│     ├── 定期从主分支拉取最新代码                                     │
│     ├── 尽早发现冲突                                                │
│     └── 减少合并难度                                                │
│                                                                      │
│  4. 小步提交                                                        │
│     ├── 每次只改一件事                                              │
│     └── 便于定位问题                                                │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

# Part 4: 实践练习

## 4.1 创建多AI协作PR

```bash
# 1. 模拟AI #1 创建功能分支
git checkout -b feature/ai-test-pr

# 2. 创建空提交（模拟AI工作）
git commit --allow-empty -m "feat: simulate AI #1 work"

# 3. 推送并创建PR
git push origin feature/ai-test-pr
# 然后在GitHub创建PR
```

# 核心知识点总结

---

## 1. PR工作流

- **What**：代码合并请求机制
- **Why**：强制审查、保证质量、记录变更
- **How**：创建PR → CI检查 → 审查 → 合并

## 2. 多AI协调

- **分工**：明确每个AI的职责
- **通信**：通过PR和Issue沟通
- **冲突预防**：分支隔离 + 接口稳定

## 3. 成熟度评估

- **多维度**：代码、质量、流程、协作
- **持续改进**：从混乱到规范到优化
