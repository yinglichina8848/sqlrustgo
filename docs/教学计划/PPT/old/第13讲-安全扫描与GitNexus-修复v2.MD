# Part 1: AI 时代的安全困境

---

## 1.1 为什么 AI 写的代码更容易有安全隐患

### AI 的安全盲区

```
开发者（提示词）：帮我写一个用户登录的功能

AI 生成的代码（表面上看）：
  ✅ 用户名密码验证 → 逻辑正确
  ✅ 返回登录结果 → 功能完整
  
AI 可能隐藏的问题（上下文外）：
  ❌ 密码用 MD5 加密（MD5 已被破解，不安全！）
  ❌ SQL 查询用字符串拼接（有 SQL 注入风险！）
  ❌ 错误信息暴露"用户名不存在"（信息泄露！）
  ❌ API 没有限流（可以被暴力破解！）
```

### 为什么 AI 会犯这些错误

```
AI 的训练数据来自互联网 → 互联网上有大量不安全的代码
AI 不理解"安全原则" → 它只是模仿模式，不是理解原理
AI 的上下文有限 → 看不到"这个函数被谁调用、传什么参数"

结论：AI 能看到代码的表面结构，
      但看不到"这段代码在生产环境被攻击会怎样"。
```

---

## 1.2 安全扫描的三层架构

### 用一张类比理解三层防护

```
机场安检：
  托运行李扫描（SAST）  → 查行李里有没有违禁品（分析代码本身）
  身份核实（SCA）       → 查这人有没有犯罪记录（查依赖库有没有漏洞）
  安检门（DAST）        → 走进去时会响（运行时检测异常行为）

软件开发的安全扫描：
  SAST（静态分析）  → clippy + GitNexus：不运行代码，直接分析
  SCA（依赖扫描）   → cargo audit：检查第三方库有没有已知漏洞
  DAST（动态分析）  → 集成测试 + TPC-H：运行起来看行为对不对
```

### 三层各自解决什么问题

```
SAST（静态应用安全测试）
  原理：看源代码，不需要运行
  工具：cargo clippy、GitNexus impact analysis
  发现：SQL 注入风险、硬编码密码、不安全加密算法

SCA（软件组成分析）
  原理：扫描依赖库，查 CVE 数据库
  工具：cargo audit
  发现："你用的 serde_json 1.0.108 版本有个已知漏洞 CVE-2024-XXXX"

DAST（动态应用安全测试）
  原理：运行程序，模拟攻击
  工具：集成测试、TPC-H 基准
  发现："执行这个 SQL 时，系统异常退出了"
```

## 1.3 cargo audit 的豁免机制

### 为什么不能一刀切

```
理想情况：
  cargo audit 发现漏洞 → 立即升级 → 问题解决

现实情况：
  • 漏洞影响极小（比如只在特定平台触发，而你不支持那个平台）
  • 修复依赖上游发布（你用的库的维护者还没修）
  • 升级可能引入兼容性问题（新版本改了 API）

一刀切"必须修复"会导致：
  → 要么项目卡住不动
  → 要么冒险升级引入新问题
```

### 合规的豁免方式

```
正确的做法（E-09 Floor 机制）：
  1. cargo audit 发现 CVE-XXXX
  2. 分析：影响范围？是否可被利用？
  3. 决策：风险可控 → 记录豁免 + 设置复查日期
  4. 追踪：Issue #XXX 记录豁免原因和处理计划
  5. 复查：到期后重新评估

错误的做法：
  cargo audit || true  ← 直接忽略所有漏洞（违规！）
```

# Part 2: GitNexus — 代码的"搜索引擎"

---

## 2.1 传统搜索 vs 代码智能

### 一个类比帮你理解

```
Google 搜索：
  你问："北京到上海高铁多久"
  Google 返回：网页列表（包含答案的页面）
  背后技术：网页索引 + 排名算法

GitNexus：
  你问："这个函数被谁调用了？"
  GitNexus 返回：调用者列表 + 调用链 + 影响范围
  背后技术：代码符号索引 + 调用关系图谱
```

### GitNexus 的数据规模

```
SQLRustGo 项目的 GitNexus 索引：
  • 52,324 个符号（函数、类、结构体、接口...）
  • 80,141 个关系（谁调用谁、谁继承谁...）
  • 300 个执行流（从入口到出口的完整调用链）

对于人类：不可能记住 52,324 个符号的关系
对于 GitNexus：精确查询，秒级返回
```

---

## 2.2 核心功能一：影响范围分析（impact analysis）

### "我改了这行代码，会发生什么？"

```
场景：你计划修改 execute_join() 函数

修改前，先问 GitNexus：
  gitnexus_impact(target="execute_join", direction="upstream")

GitNexus 返回：
  {
    直接调用者：[query_executor, subquery_handler, view_executor],
    影响执行流：[QueryExecution, SubqueryProcessing, ViewMaterialization],
    风险等级：HIGH
  }

决策：
  风险 LOW → 正常修改，常规 Review 即可
  风险 HIGH → 需要更仔细的 Review + 更多测试
  风险 CRITICAL → 必须架构师批准 + 完整回归测试
```

### 为什么这对 AI 辅助开发特别重要

```
传统开发（人为主）：
  开发者修改 execute_join() → 经验告诉他可能影响哪些模块
  → 主动检查相关模块

AI 辅助开发：
  AI 修改 execute_join() → AI 不知道有哪些调用者（上下文有限）
  → 不会主动检查相关模块

GitNexus 的价值：
  弥补 AI 的"上下文盲区"
  AI 只管写代码，GitNexus 告诉你"这个改动会炸到哪里"
```

---

## 2.3 核心功能二：变更范围检测（detect changes）

### "我只想改 3 个文件，实际上改了多少？"

```
场景：PR 创建前，确认改了什么

运行：gitnexus_detect_changes()
结果：
  检测到变更：
    ✓ executor/query.rs     （预期的）
    ✗ executor/join.rs      ← 意外的！怎么这个也被改了？
    ✗ README.md             ← 意外的！我改文档了？

预期变更：executor/query.rs
风险：2 个文件不在预期范围内
  → join.rs 可能需要同步测试
  → README.md 变更需要文档复查
```

### 这个检查为什么重要

```
AI 经常会"顺手"修改关联文件，而开发者没注意到：
  • 改了一个函数的签名 → AI 顺手改了所有调用处
  • 这些调用处的改动可能引入新 Bug
  • 如果开发者没注意到这些"顺手改动"，PR 就这样合入了

gitnexus_detect_changes() 的价值：
  不管 AI 改了多少文件，GitNexus 都帮你列出来。
  让你在提交前有机会问自己："这些改动我都有意识吗？"
```

## 2.4 GitNexus 在 Gate 中的角色

### PR 合并前的强制检查

```
PR 创建
    │
    ├──→ gitnexus_impact() 分析变更影响
    │       │
    │       ├──→ LOW → 正常流程
    │       ├──→ HIGH → 需要额外 Review + 更多测试
    │       └──→ CRITICAL → 必须架构师批准
    │
    └──→ gitnexus_detect_changes() 验证变更范围
            │
            └──→ 范围不符 → 要求开发者说明或补充
```

### 传统 Review vs GitNexus 增强 Review

```
传统 Review：
  Reviewer 逐行看代码 → 人眼有限，可能遗漏跨文件影响

GitNexus 增强 Review：
  1. GitNexus 预先分析 → "这里改了，影响 A/B/C 三个模块"
  2. Reviewer 聚焦高风险区域 → "这三个模块的测试够不够？"
  3. 人工判断 + 机器分析 = 完整覆盖
```

# Part 3: 从"测试"到"证明"

---

## 3.1 测试 ≠ 证明：一个数学类比

### 用初中数学理解核心差异

```
测试（Testing）：
  已知 f(1)=2, f(2)=4, f(3)=6
  猜测 f(x) = 2x
  验证 f(4)=8, f(5)=10 ✓
  结论："f(x)=2x 应该是正确的"

  问题：你只测试了 5 个点，f(100) 一定等于 200 吗？
        f(x) 可能是 "当 x≤5 时等于 2x，否则等于 0"

形式化证明（Formal Verification）：
  已知 f(1)=2
  已知 f(n+1) = f(n) + 2
  通过数学归纳法证明：
    基础：n=1 时 f(1)=2 ✓
    归纳：假设 f(k)=2k，则 f(k+1)=2k+2=2(k+1) ✓
  结论：∀n ∈ ℕ, f(n)=2n

  这是"数学保证"——对所有 n 都成立，不依赖"猜"
```

### 映射到数据库开发

```
测试（Testing）：
  写了 100 个测试 → 100 个都通过了
  → 结论："代码应该是正确的"
  → 风险：第 101 个场景可能失败

形式化证明（Formal Verification）：
  INVARIANT（不变式）：
    "如果一个事务提交了，它的修改对所有后续事务可见"
  
  验证方式：
    • 不是测试"100 个事务"
    • 而是证明"对所有可能的事务组合，这个规则都成立"
  
  → 结论："在所有可能情况下，规则都不被违反"
  → 这是"数学保证"
```

---

## 3.2 为什么 AI 时代更需要形式化证明

### AI 让"测试"的局限性更致命

```
人类写代码：
  人类对代码有"整体理解"
  → 写的测试倾向于覆盖典型场景 + 已知边界条件
  → 测试质量相对可控

AI 写代码：
  AI 对系统没有"整体理解"（只有上下文窗口内的代码）
  → AI 写的代码边界行为不可预测
  → 测试再多也不能保证"AI 没引入新问题"

形式化证明的价值：
  不管是谁写的代码（人/AI），只要证明了 INVARIANT 保持不变
  → 就可以信任系统的核心行为
```

---

## 3.3 形式化证明的四个层次

### 从简单到复杂

```
Layer 1：单元测试（最基础）
  ───────────────────────
  证明："f(1)=2, f(2)=4"
  范围：单个函数的单个输入
  工具：cargo test

Layer 2：属性测试（Property Testing）
  ───────────────────────
  证明："对任意输入，排序后结果都是升序的"
  范围：任意输入下的属性保持
  工具：proptest、quickcheck

Layer 3：GitNexus 图谱证明（当前能力）
  ───────────────────────
  证明："这个代码修改不影响已证明的 INVARIANT"
  范围：代码变更不破坏调用链上的正确性
  工具：GitNexus impact analysis + 测试关联

Layer 4：形式化规格驱动（未来目标）
  ───────────────────────
  证明："整个系统满足规格说明"
  范围：系统级别的数学证明
  工具：Dafny、TLA+、Formulog
```

### SQLRustGo 当前处于什么层级

```
当前：Layer 1-3
  ✅ 单元测试覆盖核心逻辑
  ✅ GitNexus 影响分析保护调用链
  ✅ Gate 系统量化验证关键指标

进行中：Layer 4
  ⏳ Dafny 验证 B+Tree 正确性
  ⏳ TLA+ 验证 MVCC 并发安全性
  ⏳ Formulog 验证查询等价性
```

---

## 3.4 IF/THEN 规则：形式化思维的起点

### 给 AI 下指令，需要明确的逻辑

```
模糊的描述（AI 容易理解错）：
  "这个 PR 修复了一个并发 Bug"
  → AI 不知道具体什么场景、什么条件下触发

IF/THEN 格式（逻辑清晰，AI 和人都能验证）：
  ## IF 条件
  两个事务同时 UPDATE 同一行，隔离级别为 Snapshot Isolation
  
  ## THEN 结果
  只有一个事务成功提交，另一个返回 WriteConflict 错误
  
  ## INVARIANT（不变式）
  任何时候，每行数据最多被一个未提交事务修改
```

### 为什么要训练这种思维方式

```
IF/THEN 不是格式问题，是思维问题。

当你习惯用 IF/THEN 描述问题时：
  → 你会想清楚"到底什么情况下出问题"
  → 你会明确"修完之后应该是什么行为"
  → 你会定义"什么不能破坏"

这种思维方式 = Harness 工程的核心能力
```

---

## 3.5 形式化证明的局限性（诚实对待）

### GitNexus 能做的和不能做的

```
GitNexus 能做到的：
  ✅ 分析代码调用关系（谁调用了谁）
  ✅ 识别影响范围（改了这里会影响哪里）
  ✅ 量化风险等级（HIGH/CRITICAL）
  ✅ 关联测试用例（这个函数有哪些测试覆盖）

GitNexus 做不到的：
  ✗ 数学形式化证明（"对所有输入..."不是它的事）
  ✗ 穷举所有输入组合（这在理论上就不可能）
  ✗ 绝对保证正确性（任何工具都做不到）

结论：
  GitNexus 是 Harness 的一部分，提高"信心"而非提供"证明"
  真正的形式化证明需要 Dafny/TLA+ 等专用工具
```

# 核心知识点总结

---

## 1. AI 与安全

- **AI 的安全盲区**：上下文有限，不理解安全原则
- **三层防护**：SAST（静态分析）+ SCA（依赖扫描）+ DAST（动态测试）
- **豁免机制**：可追溯的风险容忍（E-09 Floor），不是简单忽略

## 2. GitNexus 代码智能

- **代码的"搜索引擎"**：52,324 个符号的关系图谱
- **impact analysis**：修改前告诉你"会影响什么"
- **detect changes**：提交前确认"改了什么"
- **Gate 角色**：HIGH/CRITICAL 风险自动拦截

## 3. 从"测试"到"证明"

- **测试**：通过有限的样例验证（"应该是正确的"）
- **证明**：通过数学方法保证所有情况（"一定是正确的"）
- **IF/THEN 规则**：形式化思维的起点——用明确的逻辑描述需求和不变式
- **四个层次**：单元测试 → 属性测试 → GitNexus 图谱 → 形式化工具
