# Part 1: Gate系统的设计思想

---

## 1.1 为什么需要Gate

### Gate的定义

> **Gate（门禁）是代码合并或版本发布前必须通过的检查点。每个Gate定义了明确的检查项和通过标准。**

### 没有Gate的问题

```
没有Gate的项目：
  • PR可以随时合并
  • 没有强制测试要求
  • 性能退化无人知晓
  • 依赖漏洞静默引入
  • 发布版本质量参差不齐

结果：技术债务累积 → 维护成本指数增长
```

### 有Gate的效果

```
有Gate的项目：
  • PR必须通过所有Gate才能合并
  • 问题在开发早期发现
  • 性能基线持续保持
  • 安全漏洞及时修复
  • 每个发布版本都可追溯

结果：质量稳定 → 技术债务可控
```

## 1.2 ISO 9001与Gate设计

### ISO 9001的过程控制原则

```
ISO 9001 要求：
  • 过程必须被定义
  • 过程必须被执行
  • 过程结果必须被记录
  • 过程必须被持续改进

Gate系统实现：
  ✓ gate_spec_v300.md → 定义Gate过程
  ✓ check_beta_v300.sh → 执行Gate检查
  ✓ PR评论/SSOT → 记录Gate结果
  ✓ 定期复审 → 持续改进
```

### Gate与QA的关系

```
传统QA：
  人为质量把关 → 依赖人员经验 → 质量不稳定

Harness QA：
  Gate自动化检查 → 量化质量标准 → 质量稳定可控
```

## 1.3 Gate的分层设计

### 为什么需要分层

```
单一Gate的问题：
  • 检查项太多 → 执行时间过长
  • 部分检查依赖其他模块 → 无法并行
  • 失败定位困难 → 不知道哪个检查项失败

分层Gate的优势：
  • BP1（静态）→ 快速反馈（~2分钟）
  • BP2（集成）→ 完整验证（~10分钟）
  • BP3（Beta）→ 业务验收（~30分钟）

失败定位：哪层Fail → 查哪层 → 快速修复
```

### 三层Gate架构

```
┌─────────────────────────────────────────────────────────┐
│                    三层Gate架构                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  BP1 静态门禁（开发时触发）                               │
│  ├─ cargo build --all-features                          │
│  ├─ cargo fmt --check                                  │
│  ├─ cargo clippy --all-features -- -D warnings         │
│  └─ 文档链接检查                                         │
│                                                         │
│  BP2 集成门禁（PR创建时触发）                            │
│  ├─ cargo test --all-features (≥90% pass)             │
│  ├─ cargo llvm-cov --all-features --json (≥75%)        │
│  ├─ check_tpch.sh (TPC-H SF=1)                         │
│  └─ check_regression.sh (性能基线)                      │
│                                                         │
│  BP3 Beta门禁（合并前触发）                              │
│  ├─ 31 Issues闭环追踪                                   │
│  ├─ 5类KPI指标验证                                     │
│  ├─ BetaPhase2功能完整性                                │
│  └─ E-09 Floor豁免评估                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 1.4 E-09 Floor：风险容忍机制

### 为什么需要E-09

```
Gate失败 = 必须修复 = PR blocked

实际情况：
  • 某些检查项失败，但风险可控
  • 修复需要时间，可能阻塞其他工作
  • 已知的性能退化，有明确修复计划

E-09 Floor：
  允许在风险可接受时，有条件地继续推进
  而不是一刀切的"必须修复"
```

### E-09 vs 绕过Gate

| 对比维度 | E-09 Floor | 绕过Gate |
|---------|-----------|----------|
| 是否记录 | ✅ 日志完整 | ❌ 无记录 |
| 是否分析 | ✅ 根因分析 | ❌ 无分析 |
| 是否追溯 | ✅ SSOT可查 | ❌ 不可追溯 |
| 是否合规 | ✅ ISO 9001合规 | ❌ 违规 |
| 是否安全 | ✅ 风险评估 | ❌ 高风险 |

### E-09的使用场景

```
适用E-09：
  • 已知漏洞，上游尚未修复
  • 性能轻微退化，已有修复计划
  • 测试覆盖率暂时不足，有明确补充计划

不适用E-09：
  • 安全漏洞（必须修复）
  • 数据损坏风险（必须修复）
  • 功能完全不可用（必须修复）
```

# Part 2: BP1/BP2/BP3三层门禁

---

## 2.1 BP1：静态门禁

### 检查项

```
BP1 静态门禁检查项：

1. cargo build --all-features
   → 编译成功

2. cargo fmt --check
   → 格式符合规范

3. cargo clippy --all-features -- -D warnings
   → 零警告

4. 文档链接检查 (check_docs_links.sh)
   → 无失效链接
```

### 执行时机

```
BP1触发条件：
  • 开发者push代码到任何分支
  • PR创建时
  • 定时巡检

BP1执行时间：~2分钟
BP1失败 → PR blocked（快速反馈）
```

### BP1的快速反馈机制

```
开发者push代码
        │
        ▼
   Git webhook触发
        │
        ▼
    BP1 Gate
        │
        ├──→ Pass → 继续
        │
        └──→ Fail → 立即通知
                   │
                   ├──→ PR评论通知
                   ├──→ Slack/Email通知
                   └──→ 开发者修复
```

## 2.2 BP2：集成门禁

### 检查项

```
BP2 集成门禁检查项：

1. cargo test --all-features
   → 通过率 ≥ 90%

2. cargo llvm-cov --all-features --json
   → 覆盖率 ≥ 75%
   （注：tarpaulin已在Gate中禁用）

3. check_tpch.sh
   → TPC-H SF=1，22个Query全部通过

4. check_regression.sh
   → 性能基线保持，无显著退化
```

### BP2与BP1的区别

| 维度 | BP1 | BP2 |
|------|-----|-----|
| 检查类型 | 静态 | 动态 |
| 执行时间 | ~2分钟 | ~10分钟 |
| 需要编译 | 否 | 是 |
| 需要数据库 | 否 | 是 |
| 触发时机 | push/PR | PR创建 |
| 失败影响 | 警告 | Block |

### BP2的覆盖率和性能要求

```
覆盖率要求：
  cargo llvm-cov --all-features --json
  → 阈值：≥75%

为什么不用tarpaulin：
  • tarpaulin精度问题
  • 已被Gate系统禁用
  • llvm-cov是Rust官方推荐

性能要求：
  check_tpch.sh
  → SF=1所有Query通过
  → 无超时

check_regression.sh
  → Δ ≤ 阈值（如5%）
  → --skip-run 不能用于Gate决策
```

## 3.2 Issue闭环追踪

### 闭环的定义

```
Issue → PR → Gate检查 → 合并 → Issue关闭

闭环要求：
  • 每个Issue必须有PR关闭
  • PR必须通过所有Gate
  • Issue不能手动关闭（无PR时）
```

### Issue与Gate的联动

```
创建Issue (#465)
    │
    ▼
创建PR修复Issue
    │
    ▼
PR描述: Closes #465
    │
    ▼
Gate检查
    │
    ├──→ Pass → PR合并 → Issue自动关闭
    │
    └──→ Fail → Issue保持Open
                修复后重新Gate
```

### Issue关闭验证

```
错误做法：
  Issue #123 → 手动点击关闭 → 无PR → 追踪丢失

正确做法：
  Issue #123 → 查看关联PR
             │
             ├──→ 有PR且merged → 可以关闭
             └──→ 无PR或未merged → 禁止关闭
```

## 3.4 Gate与持续改进

### Gate数据的分析

```
Gate历史数据 → 分析趋势 → 优化流程

示例分析：
  • Q1: BP2平均失败次数2.3次/PR
  • Q2: BP2平均失败次数1.1次/PR

优化措施：

```

# 核心知识点总结

---

## 1. Gate设计思想

- **Why**：防止质量退化，技术债务可控
- **What**：代码合并前必须通过的检查点
- **How**：ISO 9001过程控制 + CMM量化管理

## 2. 三层Gate

- **BP1**：静态检查（~2分钟），快速反馈
- **BP2**：集成检查（~10分钟），量化指标
- **BP3**：Beta检查（~30分钟），业务验收

## 3. 闭环追踪

- **Issue → PR → Gate → 合并 → Issue关闭**
- **E-09 Floor**：合规的风险容忍机制
- **持续改进**：Gate数据 → 分析趋势 → 优化流程
