## 1.1 一次完整的版本发布长什么样

### 从需求到发布的旅程

```
需求提出（Issue #507 "EXPLAIN ANALYZE 功能缺失"）
        │
        ▼
开发实现（PR #516）
        │  代码 + 测试 + 文档
        │
        ▼
Gate 验证
        │  BP1 静态检查 ✅ (~2分钟)
        │  BP2 集成检查 ✅ (~10分钟)
        │  BP3 风险检查 ✅ (~30分钟)
        │
        ▼
代码合并 → Issue 自动关闭
        │
        ▼
版本准备
        │  所有相关 Issue 关闭
        │  所有 Gate 通过
        │  Release Notes 编写
        │
        ▼
版本发布
        │  Git Tag: v3.0.0-beta.3
        │  Release 页面发布
        │  通知相关团队
        │
        ▼
知识沉淀
        │  经验写入 GBrain SSOT
        │  文档同步到 Wiki
        │  团队复盘
```

---

## 1.2 Beta 版本 vs 正式版本

### 不同阶段的发布标准

```
Beta 版本（v3.0.0-beta.x）：
  标准：BP1 + BP2 必须通过，BP3 可使用 E-09 Floor
  含义："功能完整了，质量在验证中，可能有已知问题"
  受众：内部测试者、早期使用者

RC 版本（v3.0.0-rc.x）：
  标准：BP1 + BP2 + BP3 全部通过
  含义："质量验证完成，准备发布"
  受众：全员测试

GA 版本（v3.0.0）：
  标准：所有 Gate + 形式化证明 + 文档完整
  含义："生产就绪"
  受众：所有用户
```

## 1.3 语义化版本号（SemVer）

### 版本号告诉你什么信息

```
格式：MAJOR.MINOR.PATCH

  v3 . 0 . 1
   │   │   │
   │   │   └── PATCH：Bug 修复（兼容）
   │   └────── MINOR：新功能（兼容）
   └────────── MAJOR：大版本（可能不兼容）

举例：
  v2.9.0 → v2.9.1  修复了一个 Bug
  v2.9.1 → v2.10.0 加了一个新功能
  v2.10.0 → v3.0.0 Beta Phase 2 完成（重大里程碑）
```

# Part 2: GBrain SSOT — 知识管理的三层架构

---

## 2.1 问题：项目的知识都去哪了？

### 知识分散的困境

```
一个典型项目中的"知识"散落在：
  • README.md（仓库里）
  • docs/ 目录里的 50 个 Markdown 文件
  • Gitea/GitHub Issues 里的讨论
  • PR 里的 Review 评论
  • Slack/微信聊天记录
  • 某人的笔记本

问题：
  → 新成员加入："这个功能为什么这样设计的？" → 问老人
  → AI Agent 需要查知识："在哪里能找到 JOIN 的设计文档？" → 找不到
  → 老人离职："这个模块的坑都在我这"
```

### SSOT 是什么

> **SSOT = Single Source of Truth（单一事实来源）**
> 
> 一句话理解：项目所有知识的"官方答案库"。
> 任何问题，查 SSOT 就够了，不需要到处问人。

## 2.3 为什么需要 GBrain 这个中间层

### 问题：QMD 文件很多，但 AI Agent 查不到

```
没有 GBrain 的时候：
  开发者找到了 50 个 Markdown 文件
  想查"JOIN 功能的设计原理"
  → 用 grep 搜 "JOIN" → 返回 200 行匹配
  → 逐行读 → 花了 20 分钟

有 GBrain 的时候：
  AI Agent 查询 GBrain："JOIN 功能的设计原理"
  → GBrain 返回："JOIN 设计在 article_004 中，核心策略是 Hash Join"
  → 直接定位 → 花了 10 秒
```

### GBrain 存什么（28 条知识的结构）

```
GBrain SSOT 的知识分类：
  ┌────────────────────────────────────────┐
  │  10 条 Pattern（设计模式）              │
  │  例如："JOIN 使用 Hash Join 策略"       │
  │  例如："WAL 使用 Group Commit 批量刷盘" │
  ├────────────────────────────────────────┤
  │  5 条 Strategy（策略决策）              │
  │  例如："为什么选 Nomad 而非 K8s"        │
  │  例如："为什么覆盖率定在 75% 而非 85%"   │
  ├────────────────────────────────────────┤
  │  3 条 Episode（关键事件）              │
  │  例如："concurrent_select_8t 56% 回归" │
  │  例如："TPC-H SF=1 首次 22/22 通过"    │
  ├────────────────────────────────────────┤
  │  10 条 Fact（核心事实）                │
  │  例如："项目有 52,324 个代码符号"       │
  │  例如："目前覆盖 90% SQL-92 标准"       │
  └────────────────────────────────────────┘
```

## 2.4 知识管理与 Harness 的关系

### Agent 如何利用 GBrain

```
场景：一个新 Issue "添加 EXPLAIN ANALYZE 功能"

1. 开发者（或 AI Agent）先查询 GBrain：
   "EXPLAIN 相关功能的状态是什么？"
   
2. GBrain 返回：
   Pattern: "EXPLAIN 在 executor/explain.rs 中已有基础实现"
   Episode: "BP2-5 EXPLAIN ANALYZE 测试已于 2026-05 完成"
   
3. 开发者知道：
   ✓ 不需要从头开始
   ✓ 现有实现在哪里
   ✓ 之前遇到过什么问题

结果：避免了重复造轮子和掉进已知的坑
```

### 三层模型中的位置

```
提示词层：开发者提出需求
上下文层：GBrain SSOT 提供项目全局知识（弥补 AI 上下文不足）
Harness 层：Gate 系统检查交付质量

GBrain 的角色 = 扩大的"上下文层"
  让 AI（和开发者）看到的不只是当前文件，
  而是整个项目的知识图谱。
```

## 3.2 传统开发 vs AI 工厂

### 核心差异

```
传统开发模式：
  人想 → 人写 → 人测 → 人发布
  瓶颈：人的精力和经验
  特点：质量依赖个体

AI 工厂模式：
  人定方向（需求+设计）→ AI 写代码 → Gate 自动检查 → GBrain 沉淀知识
  瓶颈：Gate 的严格程度
  特点：质量依赖流程，而非个人
```

### 各个角色的变化

```
开发者角色的变化：
  过去：80% 时间写代码，20% 时间测试/Review
  现在：20% 时间写关键代码，40% 时间设计验证流程
        40% 时间审查 AI 输出 + 利用 GitNexus 分析影响

AI 的角色：
  执行者：快速生成代码、运行测试、生成文档
  不是：决策者（由人做架构决策）、验证者（由 Gate 做质量判断）

Gate 的角色：
  裁判：不管代码是谁写的，达标就过，不达标就打回去
```

# 核心知识点总结

---

## 1. 版本发布闭环

- Issue → PR → Gate → 合并 → 发布 → 知识沉淀

## 3. AI-Native 软件工厂

- **Agent 编排** → **代码生成** → **Harness 验证** → **CI/CD 执行** → **知识沉淀**
- **核心转变**：质量依赖"流程"而非"个人"
- **知识飞轮**：越用越快，越积累越强
