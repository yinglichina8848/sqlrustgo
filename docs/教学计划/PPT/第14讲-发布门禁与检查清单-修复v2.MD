# Part 1: Gate系统的设计思想

---

## 1.1 从一个类比开始：驾考的启示

### 为什么驾考要分科目二和科目三？

```
科目二（场地）：倒车入库、侧方停车 → 相当于 BP1（静态检查）
科目三（路考）：真实道路驾驶       → 相当于 BP2（集成检查）
安全文明考试：理论 + 应急处理      → 相当于 BP3（风险验证）

三科全部通过 → 拿驾照（发布）
任一科不过 → 重考（修复后重新 Gate）
```

> **Gate 系统的核心思想：用分阶段的外部检查流程，补偿内部能力的不确定性。**

### 在软件开发中的映射

```
AI 写的代码（能力不确定）
        │
        ▼
    Gate 系统（外部检查流程，确定性）
        │
        ▼
    质量可靠的软件（结果确定）
```

---

## 1.2 What：Gate 到底是什么

### 定义

> **Gate（门禁）是代码合并或版本发布前必须通过的检查点。**
> **每个 Gate 有明确的"检查什么"和"达到什么标准才算通过"。**

### 没有 Gate 的项目长什么样

```
没有 Gate：
  • PR 可以随时合并，不管测试过没过
  • 性能退化没人知道（直到用户投诉）
  • 依赖漏洞悄悄引入
  • 代码格式五花八门
  • 每个版本质量像"抽盲盒"

结果：技术债务越滚越大，维护成本指数增长
```

### 有 Gate 的项目长什么样

```
有 Gate：
  • PR 必须通过所有 Gate 才能合并
  • 问题在开发早期就被发现
  • 性能基线始终可追踪
  • 安全漏洞及时暴露
  • 每个版本质量有据可查

结果：质量稳定可控，技术债务受控
```

---

## 1.3 Why：为什么 AI 时代更需要 Gate

### 核心矛盾

```
传统的软件开发：
  人写代码 → 人 Review → 人测试 → 人判断质量
  问题：人也会出错，但出错模式可预测

AI 辅助的开发：
  AI 写代码 → 人 Review → Gate 验证 → 量化判断质量
  问题：AI 出错模式不可预测，必须用"流程"补偿
```

### AI 带来的新风险

```
AI 擅长做的事：
  ✅ 快速生成大量代码
  ✅ 处理常见编程模式

AI 不擅长做的事：
  ❌ 保证一段修改不影响其他模块（需要 GitNexus）
  ❌ 判断性能是否退化（需要基准测试）
  ❌ 发现跨文件的逻辑漏洞（需要集成测试）
  ❌ 知道"这个写法可能在生产环境出问题"（需要风险验证）

结论：AI 写代码越多，Gate 就越重要
```

## 1.4 提示词-上下文-Harness 三层模型

### 三层各自关心什么

```
┌─────────────────────────────────────────────────────────┐
│                    三层协作模型                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  提示词层（Prompt）                                      │
│  ─────────────────                                      │
│  "帮我实现一个 JOIN 功能"                                │
│  开发者关心：需求是否清晰                                │
│                                                         │
│  上下文层（Context）                                     │
│  ─────────────────                                      │
│  AI 看到：当前函数的代码                                 │
│  AI 看不到：调用链、并发场景、全局性能影响                │
│  关键：上下文永远不够 → 需要外部验证                     │

│  BP1 编译 → BP2 测试 → BP3 风险验证                     │
│  核心：用流程的"确定性"补偿 AI 的"不确定性"              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

> **记住这个公式：提示词越模糊 → AI 输出越不可控 → Harness 层越重要**

## 1.5 Gate 的分层设计

### 为什么不能只有一个 Gate

```
单一 Gate 的问题：
  • 检查项太多 → 每次都要等半小时
  • 失败定位困难 → "哪一项没过？"
  • 无法并行 → 浪费计算资源

分层 Gate 的优势：
  • BP1（静态）→ ~2分钟，快速反馈低级错误
  • BP2（集成）→ ~10分钟，验证功能正确性
  • BP3（风险）→ ~30分钟，验证生产可信度

类比：
  单一 Gate = 把所有考试合在一张卷子上批改
  分层 Gate = 分科考试，哪科挂了补哪科
```

### 三层 Gate 全景

```
┌─────────────────────────────────────────────────────────┐
│                    三层 Gate 架构                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  BP1 静态门禁（每次 push 触发，~2分钟）                   │
│  ├─ cargo build --all-features    ← 能编译吗？           │
│  ├─ cargo fmt --check             ← 格式对吗？            │
│  ├─ cargo clippy -- -D warnings   ← 有代码隐患吗？        │
│  └─ 文档链接检查                   ← 文档能看吗？          │
│                                                         │
│  BP2 集成门禁（PR 创建时触发，~10分钟）                   │
│  ├─ cargo test（通过率 ≥ 90%）     ← 功能对吗？            │
│  ├─ 覆盖率 ≥ 75%                  ← 测试够吗？            │
│  ├─ TPC-H SF=1（22个查询全通过）   ← SQL 兼容吗？         │
│  └─ 回归检测（性能基线保持）        ← 没变慢吧？           │
│                                                         │
│  BP3 风险门禁（合并前触发，~30分钟）                      │
│  ├─ WAL Crash Injection（9个故障点） ← 崩溃能恢复吗？    │
│  ├─ MVCC Isolation（4种隔离级别）   ← 并发正确吗？       │
│  ├─ Deadlock Fuzzer（随机锁调度）   ← 会死锁吗？          │
│  └─ Audit Integrity（审计链验证）   ← 记录可信吗？       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 1.6 E-09 Floor：为什么需要"豁免通道"

### 现实场景

```
理想情况：
  Gate 失败 → 必须修复 → 修复后重新 Gate → 通过 → 合并

现实情况：
  • 某个 CVE 漏洞影响微乎其微
  • 修复依赖上游发布，需要等 2 周
  • 性能退化 3%，已知原因，下个 PR 会修

一刀切"必须修复"会阻塞所有工作。
```

### E-09 Floor = 合规的风险容忍

```
E-09 Floor 不是"绕过 Gate"，而是"记录在案的有条件推进"：

  E-09 Floor                         绕过 Gate
  ────────────                       ─────────
  ✅ 有日志："为什么豁免"             ❌ 无记录
  ✅ 有分析："风险有多高"             ❌ 无分析
  ✅ 可追溯："什么时候、谁做的决策"    ❌ 不可追溯
  ✅ 合规（ISO 9001 可接受）          ❌ 违规
```

> **一句话：E-09 让你在"风险可控"时继续前进，但必须留下完整的决策记录。**

# Part 2: BP1 和 BP2 详解

---

## 2.1 BP1：静态门禁（"能编译吗？格式对吗？"）

### 为什么要最先检查

```
BP1 检查最基础的问题——这些问题是"低级错误"，但 AI 经常犯：
  • AI 生成的代码拼写错误
  • AI 引入不存在的 API
  • AI 格式不符合项目规范

如果 BP1 都没过，后面的测试根本没必要跑。
```

### BP1 检查什么

```
1. cargo build --all-features
   → 代码能编译成功吗？

2. cargo fmt --check
   → 代码格式统一吗？（避免"格式打架"）

3. cargo clippy --all-features -- -D warnings
   → 有潜在代码问题吗？（如未使用的变量、可能溢出的计算）

4. 文档链接检查
   → 文档里的链接有效吗？
```

### BP1 的反馈速度

```
开发者 push → Webhook 触发 → BP1 检查 → 2 分钟内出结果

如果失败：立即通知开发者 → 修复 → 重新 push
如果通过：自动进入 BP2
```

---

## 2.2 BP2：集成门禁（"功能对吗？测试够吗？"）

### BP2 vs BP1

| 维度 | BP1（静态） | BP2（集成） |
|------|------------|------------|
| **检查什么** | 代码长什么样 | 代码动起来对不对 |
| **要不要运行** | 不需要（只看代码） | 需要（运行测试） |
| **执行时间** | ~2分钟 | ~10分钟 |
| **失败意味什么** | 代码有低级错误 | 功能有问题或测试不足 |

### BP2 的量化标准

```
所有 BP2 检查都是"数字说话"，避免了"我觉得代码还可以"的模糊判断：

1. cargo test --all-features        → 通过率 ≥ 90%
2. cargo llvm-cov --all-features     → 覆盖率 ≥ 75%
3. TPC-H SF=1                       → 22/22 全部通过
4. check_regression.sh              → 性能退化 ≤ 5%

这些数字就是你的"成绩单"——过了就是过了，没过就是没过。
```

### 覆盖率的含义（给初学者的解释）

```
"测试覆盖率 75%" 是什么意思？

假设你的代码有 100 行，
  → 75 行被测试执行过
  → 25 行从来没被测试跑到

为什么不是 100%？
  • 有些代码是防御性的（如 never 分支），很难触发
  • 追求 100% 的代价远超收益
  • 75% 是业界公认的"性价比黄金点"

类比：不是所有房间都需要每天打扫（比如储藏室），
      但客厅和厨房必须干净。
```

# Part 3: BP3 — 风险驱动验证新范式

---

## 3.1 从"功能测试"到"风险验证"的范式转变

### BP2 和 BP3 的本质区别

```
BP2 的问题："这个功能实现了吗？"
  → 用测试覆盖率衡量
  → 关心的是"写了多少测试"
  → 思维模式：量的思维

BP3 的问题："这个功能在生产环境可靠吗？"
  → 用风险矩阵衡量
  → 关心的是"最坏情况会怎样"
  → 思维模式：质的思维
```

### 为什么需要这个转变

```
传统的功能测试（BP2 已经覆盖）：
  ✅ SELECT 能查到数据
  ✅ INSERT 能写入数据
  ✅ JOIN 能关联多表

但这些"正常情况"测试再多，也不能回答：
  ❓ 写入过程中突然断电，数据会损坏吗？
  ❓ 两个事务同时改同一行，结果正确吗？
  ❓ 100 个并发连接同时操作，会死锁吗？

BP3 就是为这些问题设计的。
```

---

## 3.2 BP3 的核心任务（P0 必须完成）

### 从"场景"出发，而非从"代码"出发

```
BP3 的设计原则：
  不是"这段代码有没有测试"（BP2 的思维）
  而是"如果这里出问题，影响有多大"（BP3 的思维）
```

### BP3 风险驱动验证矩阵

| ID | 名称 | 验证的风险 | 为什么重要 |
|----|------|-----------|-----------|
| **BP3-1** | WAL Crash Injection | 写入过程中断电/崩溃 | 数据库最怕丢数据，这是"生死测试" |
| **BP3-2** | MVCC Isolation Matrix | 并发事务互相干扰 | 多个用户同时操作时，不能出现"读到一半的修改" |
| **BP3-3** | Deadlock Fuzzer | 随机锁调度导致死锁 | 并发越高，死锁风险越大 |
| **BP3-4** | Optimizer 最小测试集 | 执行计划选择错误 | 优化器选错计划 → 查询慢 100 倍 |
| **BP3-5** | Planner 最小测试集 | 逻辑计划生成错误 | Planner 错 → 整个查询结果错 |
| **BP3-6** | Differential Testing | 与 SQLite 结果不一致 | "别人算出来是 100，我们算出来是 99" |
| **BP3-7** | Audit Integrity Verification | 审计记录被篡改 | GMP 制药合规要求：审计链必须不可篡改 |

## 3.5 BP3 与提示词-上下文-Harness 的映射

### 为什么 AI 无法胜任 BP3

```
开发者（提示词）：
  "帮我写一个数据库的崩溃恢复测试"

AI 能做的（上下文内）：
  ✅ 写一个测试函数
  ✅ 模拟崩溃场景

AI 无法做的（上下文外，需要 Harness）：
  ❌ 确定"所有可能的崩溃点"（9个还是15个？）
  ❌ 验证"恢复后数据与崩溃前一致"
  ❌ 保证"测试覆盖了所有隔离级别的组合"

BP3 的价值：
  把 AI 不知道的东西，变成 Gate 检查项。
  AI 负责执行（写代码），Gate 负责验证（定标准）。
```

# Part 4: 自审计与持续改进

---

## 4.1 "量化"为什么是 CMM-4 的核心

### 模糊 vs 量化（一张表说清楚）

```
模糊的说法（CMM-3 以下）            量化的说法（CMM-4 以上）
─────────────────────────────────  ──────────────────────────
"代码质量不错"                      "覆盖率 78%，Clippy 零警告"
"性能还可以"                        "QPS 1200，查询延迟 < 50ms"
"最近没出什么大问题"                "本月死锁次数 0，Crash 次数 0"
"SQL 兼容性挺好的"                  "TPC-H 22/22 全部通过"
```

### AI 时代为什么更需要量化

```
传统开发（人为主）：
  人写代码 → 人有经验判断质量
  "这个改法应该没问题" ← 可能靠谱

AI 辅助开发：
  AI 写代码 → 人无法凭经验判断 AI 的输出
  "这段 AI 生成的代码有没有隐患？" ← 无法靠直觉

解决方案：
  不管你信不信 AI，数字不会骗你。
  覆盖率 75%、QPS 稳定、TPC-H 通过 → 可以发布。
  任一数字不达标 → 不能发布。
```

---

## 4.2 自审计循环：Gate 检视 Gate

### 什么是"自审计"

```
简单说：Gate 系统不仅要检查代码，还要检查自己。

问题：
  • 如果 BP2 的回归检测脚本有 Bug，谁来发现？
  • 如果 Gate 阈值设得太低（形同虚设），怎么优化？
  • 如果某个检查项从来没人失败过，它还有意义吗？

自审计就是回答这些问题。
```

### 自审计的三个层次

```
第一层：Gate 数据的趋势分析
  ──────────────────────────
  BP2 失败率趋势：
  第1月 ██████████ 10次失败
  第2月 ██████░░░░ 6次失败    ← 在改善
  第3月 ████░░░░░░ 4次失败    ← 持续改善
  
  → 说明团队质量意识在提升，或者 AI Prompt 优化有效

第二层：Gate 自身的有效性
  ──────────────────────────
  BP3 经常发现 BP2 漏掉的问题
  → 说明 BP2 的检查项不够全面，应该增强

  某个 BP1 检查项 3 个月零失败
  → 可能阈值太低，考虑收紧

第三层：流程的自我优化
  ──────────────────────────
  如果 BP1 + BP2 总是通过，但上线后还有问题
  → BP3 需要增加新的风险检查项
  → 这个循环永不停歇
```

---

## 4.3 CMM-5 的实操：持续优化

### CMM-5 不是"达到了"，而是"一直在做"

```
CMM-3（已定义）：Gate 过程被文档化了 ✓
CMM-4（已管理）：Gate 指标被量化了 ✓
CMM-5（优化中）：Gate 自己在不断进化 → 永无止境

SQLRustGo 的实际做法：
  • 每次 Gate 失败 → 分析根因 → 更新 Gate Spec
  • 每次版本发布 → 回顾"哪些风险没被 Gate 捕获"
  • 每月复审 → "阈值需要调整吗？需要新增检查项吗？"
```

### 一张图理解自优化闭环

```
┌─────────────────────────────────────────────────────────┐
│                   CMM-5 自优化闭环                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  计划（Plan）                                           │
│  "下个版本的目标：死锁率从 5 次/月 降到 0"               │
```

# 核心知识点总结

---

## 1. Gate 设计思想

- **What**：代码合并前必须通过的检查点，每个检查有明确标准
- **Why**：AI 写代码不可预测 → 需要外部流程保证质量
- **How**：分层 Gate（BP1 静态 → BP2 集成 → BP3 风险） + 量化指标 + 闭环追踪

## 2. 三层 Gate

- **BP1**：编译 + 格式 + Clippy（~2分钟），过滤低级错误
- **BP2**：测试 + 覆盖率 + TPC-H + 回归（~10分钟），验证功能正确
- **BP3**：Crash Injection + MVCC + Deadlock Fuzz + Audit（~30分钟），验证生产可信度

## 3. 风险驱动验证（BP3 的核心创新）

- **旧思维**：写更多 Happy Path 测试 → 覆盖率虚高但不安全
- **新思维**：从"最危险场景"出发 → Crash 点、隔离级别、死锁可能、审计链
- **测试分布目标**：Happy Path 30% + Edge 30% + 并发 20% + 故障 10% + 随机 10%

## 4. 自审计与持续改进（CMM-5）

- **量化**：数字说话，不说"还可以"
- **自审计**：Gate 检查自己——哪些检查项失效了？哪些阈值需要调？
- **持续改进**：Plan → Do → Check → Act → 回到 Plan
