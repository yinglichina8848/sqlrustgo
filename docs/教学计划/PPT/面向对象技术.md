# **Part 1: 面向对象分析（OOA）**

***

## **1.1 什么是面向对象分析**

### **定义**

面向对象分析（Object-Oriented Analysis, OOA）是在问题域中识别和理解**对象**及其**关系**的过程。

### **核心目标**

- 理解业务需求
- 识别问题域中的概念
- 建立问题域的模型

### **关键问题**

- 系统要做什么？（What）
- 问题域中有哪些实体？（Who）
- 实体之间有什么关系？（How）

***

## **1.2 OOA的三个层次**

### **第1层：对象发现**

- 识别问题域中的实体
- 区分对象和属性
- 寻找名词和名词短语

### **第2层：结构识别**

- 分类结构（泛化-特化）
- 组装结构（整体-部分）
- 关联关系

### **第3层：行为识别**

- 对象的行为
- 对象之间的交互
- 系统的动态特性

***

## **2.1 什么是面向对象设计**

### **定义**

面向对象设计（Object-Oriented Design, OOD）是将分析模型转化为**设计模型**的过程，定义**类**、**属性**、**方法**和**关系**。

### **核心目标**

- 定义类的结构和职责
- 设计类之间的协作
- 满足非功能需求（性能、可扩展性、可维护性）

### **关键问题**

- 系统如何做？（How）
- 需要哪些类？（Class）
- 类之间如何交互？（Interaction）

## **2.5 设计原则在OOD中的应用**

### **单一职责原则（SRP）**

> 一个类应该只有一个引起它变化的原因

text

```
// ❌ 不好的设计
class QueryHandler {
    fn parse_sql(&self) { }
    fn optimize_query(&self) { }
    fn execute(&self) { }
    fn format_result(&self) { }
}

// ✅ 好的设计
class Parser {
    fn parse(&self) -> AST { }
}
class Optimizer {
    fn optimize(&self, ast: AST) -> Plan { }
}
class Executor {
    fn execute(&self, plan: Plan) -> Result { }
}
class Formatter {
    fn format(&self, result: Result) -> Output { }
}
```

***

### **开闭原则（OCP）**

> 对扩展开放，对修改关闭

text

```
// ✅ 通过抽象实现开闭原则
trait StorageEngine {
    fn read(&self, key: &str) -> Option<Value>;
    fn write(&mut self, key: &str, value: Value);
}

class MemoryStorage implements StorageEngine { }
class DiskStorage implements StorageEngine { }
class CloudStorage implements StorageEngine { }

// 查询执行器依赖抽象，可以支持任何存储引擎
class QueryExecutor<T: StorageEngine> {
    engine: T,
    fn execute(&self, query: &Query) -> Result {
        // 使用 engine 执行查询
    }
}
```

***

### **依赖倒置原则（DIP）**

> 依赖抽象，不依赖具体实现

text

```
// ❌ 违反DIP：高层依赖低层具体实现
class QueryService {
    storage = DiskStorage::new()
}

// ✅ 符合DIP：都依赖抽象
class QueryService<T: StorageEngine> {
    storage: T
}
```

## **3.2 Rust中的面向对象特性**

### **封装（Encapsulation）**

rust

```
pub struct Page {
    id: PageId,           // 私有字段
    data: Vec<u8>,        // 私有字段
    dirty: bool,          // 私有字段
}

impl Page {
    pub fn new(id: PageId) -> Self {  // 公共构造函数
        Page {
            id,
            data: Vec::with_capacity(PAGE_SIZE),
            dirty: false,
        }
    }
    
    pub fn read(&self, offset: usize) -> &[u8] {  // 公共方法
        &self.data[offset..]
    }
}
```

### **继承的替代：Trait**

rust

```
pub trait StorageEngine {
    fn read(&self, key: &str) -> Result<Option<Value>, Error>;
    fn write(&mut self, key: &str, value: Value) -> Result<(), Error>;
}

pub struct MemoryStorage {
    data: HashMap<String, Value>,
}

impl StorageEngine for MemoryStorage {
    fn read(&self, key: &str) -> Result<Option<Value>, Error> {
        Ok(self.data.get(key).cloned())
    }
    
    fn write(&mut self, key: &str, value: Value) -> Result<(), Error> {
        self.data.insert(key.to_string(), value);
        Ok(())
    }
}
```

### **多态（Polymorphism）**

rust

```
// Trait对象方式
pub struct QueryExecutor {
    storage: Box<dyn StorageEngine>,  // 运行时多态
}

// 泛型方式（编译时多态）
pub struct QueryExecutor<T: StorageEngine> {
    storage: T,  // 编译时多态
}

// 使用示例
let mem_executor = QueryExecutor::new(MemoryStorage::new());
let disk_executor = QueryExecutor::new(DiskStorage::new());
```

## **1.1 什么是行为图**

### **定义**

行为图（Behavior Diagrams）描述系统的**动态行为**，展示对象之间的交互和状态变化。

### **作用**

- 建模系统的运行时行为
- 描述对象之间的消息传递
- 展示对象的状态变化
- 刻画业务流程

***

## **1.2 行为图的分类**

text

```
UML行为图
├── 交互图（Interaction Diagrams）
│   ├── 顺序图（Sequence Diagram）—— 时间顺序的消息交互
│   ├── 通信图（Communication Diagram）—— 对象连接的消息交互
│   ├── 时序图（Timing Diagram）—— 时间约束的状态变化
│   └── 交互概览图（Interaction Overview Diagram）—— 交互的控制流
│
├── 状态图（State Diagram）—— 对象的状态变化
```

### **本讲重点**

- **顺序图**：最常用的交互图

# **Part 2: 顺序图（Sequence Diagram）**

***

## **2.1 顺序图的基本概念**

### **定义**

顺序图描述了对象之间**按时间顺序**的消息传递。

### **核心要素**

text

```
对象:      对象1          对象2          对象3
           │             │             │
生命线:     │─────────────│─────────────│
           │             │             │
消息:       │──消息1─────>│             │
           │             │──消息2─────>│
           │             │<──返回──────│
激活条:     │█████        │█████        │
           │             │             │
时间:      ↓             ↓             ↓
```

### **适用场景**

- 用例的实现细节
- 对象之间的协作
- 消息的时序关系

***

## **2.2 顺序图的组成元素**

### **1. 对象（Object）**

text

```
┌──────────┐
│ :对象类型 │  匿名对象
└──────────┘

┌──────────┐
│ 对象名    │  命名对象
│ :类型     │
└──────────┘

例如：
┌──────────┐
│ parser   │
│ :Parser  │
└──────────┘
```

### **2. 生命线（Lifeline）**

text

```
对象 ────│──── 生命线（虚线）
```

### **3. 激活条（Activation）**

text

```
对象 ────█████──── 对象正在执行操作
```

***

### **4. 消息（Message）**

**消息类型**

**符号**

**含义**

同步消息

`───>`

调用者等待返回

异步消息

`───>>`

调用者不等待

返回消息

`<───`

方法返回值

创建消息

`─create─>`

创建对象

销毁消息

`─destroy─>×`

销毁对象

### **5. 组合片段（Combined Fragment）**

**片段类型**

**符号**

**含义**

alt

`[alt]`

条件分支（if-else）

opt

`[opt]`

可选执行（if）

loop

`[loop]`

循环执行

par

`[par]`

并行执行

region

`[region]`

临界区

***

## **2.3 顺序图绘制规范**

### **布局规范**

1. **从左到右**排列参与交互的对象
2. **从上到下**表示时间顺序
3. **激活条**的顶部表示开始执行，底部表示结束
4. **消息**要清晰描述操作名称和参数

### **命名规范**

- 对象名：`对象名:类型` 或 `:类型`
- 消息名：动词短语，如 `parse(sql: String)`
- 返回消息：可省略，或用虚线箭头表示

### **示例规范**

text

```
正确的顺序图：

用户         :Parser        :Executor

 │              │─execute(ast)─>│
 │              │              │
 │<─────────────│──result──────│
```

text

text

text

# **Part 3: 状态图（State Diagram）**

***

## **3.1 状态图的基本概念**

### **定义**

状态图描述一个对象在其生命周期内的**状态变化**，以及触发状态变化的事件。

### **核心要素**

text

```
[初始状态] ●
              │
              │ 事件/条件/动作
              ▼
        ┌──────────┐
        │  状态A    │
        └──────────┘
              │
              │ 事件/条件/动作
              ▼
        ┌──────────┐
        │  状态B    │
        └──────────┘
              │
              ▼
[终止状态] ◎
```

### **适用场景**

- 单个对象的生命周期
- 状态相关的行为（如：事务状态、连接状态）
- 事件驱动的系统

***

## **3.2 状态图的组成元素**

### **1. 状态（State）**

text

```
┌─────────────┐
│   状态名     │
│             │
│ entry/动作   │ 进入状态时执行
│ do/活动      │ 在状态中执行
│ exit/动作    │ 退出状态时执行
└─────────────┘
```

### **2. 初始状态（Initial State）**

text

```
●  实心圆点
```

### **3. 终止状态（Final State）**

text

```
◎  实心圆外加圆圈
```

### **4. 转换（Transition）**

text

```
事件 [条件] / 动作
例如：点击提交 [表单有效] / 验证表单
```

***

### **5. 内部活动**

**活动类型**

**语法**

**触发时机**

进入动作

`entry/动作`

进入状态时

退出动作

`exit/动作`

退出状态时

内部活动

`do/活动`

在状态中执行

事件处理

`事件/动作`

事件发生时

### **6. 复合状态（Composite State）**

text

```
┌─────────────────────┐
│      状态A           │
│  ┌───────────────┐  │
│  │    子状态1     │  │
│  └───────────────┘  │
│         │           │
│         ▼           │
│  ┌───────────────┐  │
│  │    子状态2     │  │
│  └───────────────┘  │
└─────────────────────┘
```

***

## **3.3 状态图绘制规范**

### **命名规范**

- 状态名：名词或形容词，如`活跃`、`已提交`
- 事件名：动词短语，如`用户点击提交`
- 条件：布尔表达式，如`[余额>0]`

### **布局原则**

1. 初始状态在左上角
2. 状态从左到右、从上到下排列
3. 避免交叉的转换线
4. 使用注释解释复杂转换

### **常见错误**

- 状态过多（状态爆炸）
- 缺少初始或终止状态
- 转换条件不明确
- 事件命名模糊

***

## **3.4 实例1：事务状态图**

### **场景描述**

数据库事务的生命周期：开始、活跃、提交/回滚、结束。

### **状态图**

plantuml

```
@startuml

空闲 --> 活跃 : BEGIN事务

    [*] --> 执行中
    执行中 --> 执行中 : SQL操作
    执行中 --> 准备提交 : COMMIT
    执行中 --> 准备回滚 : ROLLBACK
    准备提交 --> 提交中 : 准备完成
    准备回滚 --> 回滚中 : 准备完成
}

活跃 --> 已提交 : COMMIT完成
活跃 --> 已回滚 : ROLLBACK完成

已提交 --> [*]
已回滚 --> [*]

注脚: 事务状态变化
@enduml
```

## **3.6 状态图与顺序图的关系**

### **区别与联系**

**维度**

**状态图**

**顺序图**

### **互补关系**


# **Part 1: 面向对象的核心概念与价值**

***

## **1.1 为什么需要面向对象？**

### **软件开发的本质挑战**

- **复杂性**：现代软件系统包含数百万行代码
- **变化性**：需求不断变化，系统需要易于修改
- **协作性**：多团队并行开发

### **面向对象的解决方案**

- **封装**：隐藏实现细节，降低认知负荷
- **继承**：复用共性，建立层次结构
- **多态**：统一接口，支持多种实现

***

## **1.2 核心概念原理解读**

### **封装（Encapsulation）**

text

```
┌─────────────────┐
│    对象         │
│  ┌───────────┐ │
│  │ 内部状态   │ │  → 数据隐藏
│  └───────────┘ │
│       ↑        │
│  ┌───────────┐ │
│  │ 公开接口   │ │  → 行为暴露
│  └───────────┘ │
└─────────────────┘
```

**原理**：将数据和操作数据的方法捆绑在一起，对外部隐藏内部实现细节，只暴露必要的接口。

**AI时代意义**：AI可以基于接口理解模块功能，无需关心内部实现，便于自动生成测试、文档和调用代码。

***

### **继承（Inheritance）**

text

```
       ┌─────────┐
       │  动物   │  ← 抽象基类
       └─────────┘
          ↑    ↑
    ┌─────┘    └─────┐
┌─────────┐    ┌─────────┐
│   狗    │    │   猫    │  ← 派生类
└─────────┘    └─────────┘
```

**原理**：通过“is-a”关系，子类自动获得父类的属性和方法，可扩展或重写。实现代码复用和层次化组织。

**AI时代意义**：AI可以自动识别类之间的继承关系，生成符合里氏替换原则的代码。

***

### **多态（Polymorphism）**

rust

```
trait Drawable {
    fn draw(&self);
}

struct Circle { radius: f64 }
struct Square { side: f64 }

impl Drawable for Circle { fn draw(&self) { /* 画圆 */ } }
impl Drawable for Square { fn draw(&self) { /* 画方 */ } }

// 统一接口，不同行为
fn render(shape: &dyn Drawable) {
    shape.draw();  // 运行时决定具体实现
}
```

**原理**：同一接口在不同对象上有不同实现，使得程序可以针对抽象编程，提高灵活性和可扩展性。

**AI时代意义**：AI可以自动识别适合多态的场景，生成trait定义和实现，确保开闭原则。

# **Part 2: 面向对象分析（OOA）原理**

***

## **2.1 OOA的目标：理解问题域**

### **核心问题**

- 系统要处理哪些**概念**？
- 概念之间有什么**关系**？
- 概念有哪些**行为**？

### **分析方法**

- **名词短语分析**：从需求文档中提取名词，识别候选对象
- **分类**：区分实体对象、边界对象、控制对象
- **抽象**：忽略细节，保留本质特征

***

## **2.2 对象发现方法**

### **名词短语分析法示例**

**需求描述**：

> 用户可以通过客户端提交SQL查询，系统解析查询后，从存储引擎中读取数据，并将结果返回给用户。

**名词提取**：

- 用户（User）
- 客户端（Client）
- SQL查询（SQLQuery）
- 系统（System）
- 解析器（Parser）
- 存储引擎（StorageEngine）
- 数据（Data）
- 结果（Result）

**筛选核心对象**：

- SQLQuery（核心概念）
- Parser（负责解析）
- StorageEngine（负责存储）
- Result（输出）

***

## **2.3 识别对象之间的关系**

### **关系类型（概念层）**

- **关联**：对象之间存在某种联系（如：Parser处理SQLQuery）
- **聚合**：整体包含部分，部分可独立存在（如：数据库包含多个表）
- **组合**：整体包含部分，部分不能独立存在（如：查询结果包含多行数据）
- **泛化**：一般与特殊的关系（如：SELECT语句是SQL语句的一种）

### **分析方法**

# **Part 3: 面向对象设计（OOD）原理**

***

## **3.1 OOD的目标：构建解决方案**

### **核心任务**

- 将分析模型转化为**设计模型**
- 定义**类**（属性、方法、可见性）
- 设计**关系**（关联、聚合、组合、继承、实现）
- 满足**非功能需求**（性能、可扩展性、可维护性）

### **设计原则（SOLID）**

- **单一职责**：一个类只有一个变化原因
- **开闭原则**：对扩展开放，对修改关闭
- **里氏替换**：子类必须能替换父类
- **接口隔离**：接口要小而专一
- **依赖倒置**：依赖抽象，不依赖具体

***

## **3.2 从概念到类：设计决策**

### **类的设计**

- **命名**：使用领域术语，如 `Parser`、`StorageEngine`
- **属性**：决定状态，如 `position: usize`、`buffer: Vec<u8>`
- **方法**：定义行为，如 `parse(&mut self) -> AST`

### **关系设计**

- **关联**：对象间的引用，决定多重性（1对1、1对多）
- **聚合/组合**：生命周期管理
- **继承/实现**：复用接口和代码

***

## **3.3 设计原则应用示例**

### **单一职责（SRP）**

rust

```
// ❌ 违反：Parser既做词法分析又做语法分析
struct Parser {
    input: String,
    pos: usize,
    tokens: Vec<Token>,
}
impl Parser {
    fn tokenize(&mut self) { /* ... */ }
    fn parse(&mut self) -> AST { /* ... */ }
}

// ✅ 符合：拆分为Lexer和Parser
struct Lexer { input: String, pos: usize }
impl Lexer { fn next_token(&mut self) -> Token { /* ... */ } }

struct Parser { lexer: Lexer }
impl Parser { fn parse(&mut self) -> AST { /* ... */ } }
```

***

### **开闭原则（OCP）**

rust

```
// 抽象存储引擎接口
trait StorageEngine {
    fn read(&self, key: &str) -> Option<Value>;
    fn write(&mut self, key: &str, value: Value);
}

// 可以任意扩展新存储引擎，无需修改上层代码
struct MemoryStorage { data: HashMap<String, Value> }
impl StorageEngine for MemoryStorage { /* ... */ }

struct DiskStorage { path: PathBuf }
impl StorageEngine for DiskStorage { /* ... */ }
```

### **依赖倒置（DIP）**

rust

```
// 高层模块（Executor）依赖抽象StorageEngine
struct Executor<T: StorageEngine> {
    storage: T,
}
impl<T: StorageEngine> Executor<T> {
    fn execute(&mut self, query: &Query) -> ResultSet {
        self.storage.read(...)  // 通过抽象调用
    }
}
```

## **3.1 状态图的本质**

### **状态图 = 对象的生命周期**

### **关键概念**

- **终止状态**：对象销毁前的状态

## **一、实验目标**

- 掌握顺序图的建模方法：描述对象间的消息交互
- 掌握状态图的建模方法：描述单个对象的状态变化
- 掌握活动图的建模方法：描述业务流程
- 掌握组件图和部署图的建模方法：描述系统架构

## **二、知识回顾：动态建模与架构图**

### **2.1 顺序图**

- **对象**：参与交互的实例
- **生命线**：对象的时间线
- **消息**：同步、异步、返回
- **激活条**：对象执行操作的时间段
- **组合片段**：alt（分支）、loop（循环）、opt（可选）、par（并行）

### **2.2 状态图**

- **状态**：对象生命周期中的阶段
- **事件**：触发状态转换的事情
- **转换**：状态之间的迁移
- **内部活动**：entry/do/exit

### **2.3 活动图**

- **活动节点**：执行的动作
- **控制节点**：决策、分叉、汇合
- **泳道**：划分责任区域

### **2.4 组件图与部署图**

- **组件图**：系统的物理模块及接口依赖
- **部署图**：节点上的组件部署及网络连接

```
## 1.1 为什么需要面向对象？

### 软件开发的本质挑战

### 封装（Encapsulation）
```

┌─────────────────┐\
│ 对象 │\
│ ┌───────────┐ │\
│ │ 内部状态 │ │ → 数据隐藏\
│ └───────────┘ │\
│ ↑ │\
│ ┌───────────┐ │\
│ │ 公开接口 │ │ → 行为暴露\
│ └───────────┘ │\
└─────────────────┘

text

```
**原理**：将数据和操作数据的方法捆绑，对外隐藏实现细节，只暴露必要接口。

```

┌─────────┐\

└─────────┘\

┌─────────┐ ┌─────────┐\
│ 狗 │ │ 猫 │ ← 派生类\
└─────────┘ └─────────┘

text

````
**原理**：通过“is-a”关系，子类自动获得父类的属性和方法，可扩展或重写。
````

# **Part 2: 面向对象分析（OOA）原理**

***

## **2.1 OOA的目标：理解问题域**

### **核心问题**

- 系统要处理哪些概念？
- 概念之间有什么关系？
- 概念有哪些行为？

### **分析方法**

- **名词短语分析**：从需求中提取名词，识别候选对象
- **分类**：区分实体对象、边界对象、控制对象
- **抽象**：忽略细节，保留本质特征

***

## **2.2 对象发现方法**

### **名词短语分析法示例**

**需求描述**：用户可以通过客户端提交SQL查询，系统解析查询后，从存储引擎中读取数据，并将结果返回给用户。

**名词提取**：用户、客户端、SQL查询、系统、解析器、存储引擎、数据、结果

**筛选核心对象**：SQLQuery、Parser、StorageEngine、Result

***

## **2.3 识别对象之间的关系**

### **关系类型（概念层）**

- **关联**：对象之间存在某种联系（如 Parser 处理 SQLQuery）
- **聚合**：整体包含部分，部分可独立存在（如 数据库包含多个表）
- **组合**：整体包含部分，部分不能独立存在（如 查询结果包含多行数据）
- **泛化**：一般与特殊的关系（如 SELECT 语句是 SQL 语句的一种）

### **分析方法**

- 从需求中寻找“拥有”、“包含”、“是一种”等描述
- 使用自然语言描述对象间的交互

# **Part 3: 面向对象设计（OOD）原理**

***

## **3.1 OOD的目标：构建解决方案**

### **核心任务**

- 将分析模型转化为设计模型
- 定义类（属性、方法、可见性）
- 设计关系（关联、聚合、组合、继承、实现）
- 满足非功能需求（性能、可扩展性、可维护性）

### **设计原则（SOLID）**

- **单一职责**：一个类只有一个变化原因
- **开闭原则**：对扩展开放，对修改关闭
- **里氏替换**：子类必须能替换父类
- **接口隔离**：接口要小而专一
- **依赖倒置**：依赖抽象，不依赖具体

***

## **3.2 从概念到类：设计决策**

### **类的设计**

- 命名：使用领域术语，如 `Parser`、`StorageEngine`
- 属性：决定状态，如 `position: usize`、`buffer: Vec<u8>`
- 方法：定义行为，如 `parse(&mut self) -> AST`

### **关系设计**

- 关联：对象间的引用，决定多重性（1对1、1对多）
- 聚合/组合：生命周期管理
- 继承/实现：复用接口和代码

***

## **3.3 设计原则应用示例**

### **单一职责（SRP）**

rust

```
// ❌ 违反：Parser既做词法分析又做语法分析
struct Parser { input: String, pos: usize, tokens: Vec<Token> }
// ✅ 符合：拆分为Lexer和Parser
struct Lexer { input: String, pos: usize }
struct Parser { lexer: Lexer }
```

### **开闭原则（OCP）**

rust

```
trait StorageEngine { fn read(&self, key: &str) -> Option<Value>; }
struct MemoryStorage { data: HashMap<String, Value> }
struct DiskStorage { path: PathBuf }
// 可以任意扩展新存储引擎，无需修改上层代码
```

### **依赖倒置（DIP）**

rust

```
struct Executor<T: StorageEngine> { storage: T }  // 依赖抽象
```

````
# Part 1: 动态建模概述

---

## 1.1 什么是动态建模？

### 定义
动态建模描述系统的**运行时行为**，关注对象之间的交互、状态变化和业务流程。

### 为什么需要动态建模？
- **验证静态设计**：类图只描述结构，动态模型验证这些结构是否能协作完成功能
- **发现交互问题**：消息顺序、并发冲突、死锁
- **沟通行为逻辑**：比文字描述更直观

---

## 1.2 动态建模的三种视角

| 视角 | 关注点 | 代表UML图 |
|------|--------|-----------|
| 交互视角 | 对象间如何传递消息 | 顺序图、通信图 |
| 状态视角 | 单个对象如何响应事件 | 状态图 |
| 流程视角 | 业务流程如何流转 | 活动图 |

### 三者的关系
- 顺序图：多个对象的一次具体交互
- 状态图：单个对象的生命周期
- 活动图：不特定对象的流程控制

---

## 1.3 AI时代动态建模的价值

### AI可以自动生成动态模型
- 从自然语言描述生成顺序图、状态图、活动图
- 从代码逆向生成动态模型（辅助理解）
- 从动态模型生成测试用例

### 开发者的角色转变
- 从“绘制者”变为“设计者和审查者”
- 核心能力：准确描述行为、审查AI生成结果

# Part 2: 顺序图深度解析

---

## 2.1 顺序图的本质

### 顺序图 = 时间线上的消息交互
- **水平轴**：参与交互的对象
- **垂直轴**：时间（从上到下）
- **消息**：对象间的通信

### 关键概念
- **生命线**：对象存在的时间段
- **激活条**：对象正在执行操作
- **消息**：同步、异步、返回
- **组合片段**：控制结构（分支、循环、并发）

---

## 2.2 顺序图的元素详解

### 对象与生命线
````

对象名:类名 :类名（匿名对象）\
│ │\
│ │\
└──────生命线─────────┘

text

```

### 消息类型
| 符号 | 含义 | 示例 |
|------|------|------|
| `─>` | 同步消息 | 调用方法，等待返回 |
| `─>>` | 异步消息 | 发送信号，不等待 |
| `<──` | 返回消息 | 方法返回值（可省略） |
| `─create─>` | 创建对象 | 实例化新对象 |
| `─destroy─>×` | 销毁对象 | 释放对象 |

---

### 组合片段
| 片段 | 含义 | 语法（PlantUML） |
|------|------|------------------|
| alt | 条件分支 | `alt [条件1]` ... `else [条件2]` ... `end` |
| opt | 可选 | `opt [条件]` ... `end` |
| loop | 循环 | `loop [次数或条件]` ... `end` |
| par | 并行 | `par` ... `end` |
| break | 中断 | `break [条件]` ... `end` |

---

## 2.3 顺序图的用途

### 主要用途
1. 描述用例的实现：展示用例中参与者的交互
2. 设计对象协作：明确消息顺序和职责分配
3. 验证接口设计：检查类之间是否有足够的方法支持交互
4. 识别并发问题：通过par片段设计并发，检查冲突

### 典型场景
- 用户登录流程
- 订单处理流程
- 数据库查询执行
- API调用链

---

## 2.4 用AI生成顺序图

### 提示词要点
- 明确参与者
- 按时间顺序描述步骤
- 指定控制逻辑
- 说明返回结果

### 示例提示词
```

请用PlantUML生成顺序图，描述用户通过Web应用查询订单的流程：\
参与者：用户、浏览器、Web服务器、订单服务、数据库\
步骤：（详细步骤）\
考虑订单不存在的情况，用alt片段处理

text

```

# Part 3: 状态图深度解析

---

## 3.1 状态图的本质

### 状态图 = 对象的生命周期
- **状态**：对象满足某种条件、执行某个活动或等待某个事件的时期
- **事件**：触发状态变化的事情
- **转换**：从一个状态到另一个状态

### 关键概念
- 初始状态：对象创建时的状态
- 终止状态：对象销毁前的状态
- 内部活动：entry（进入）、do（执行中）、exit（退出）
- 复合状态：包含嵌套子状态

---

## 3.2 状态图的元素详解

### 状态表示
```

┌─────────────┐\
│ 状态名 │\
│ entry/动作 │\
│ do/活动 │\
│ exit/动作 │\
└─────────────┘

text

```

### 转换表示
```

事件 \[条件] / 动作\
例如：点击提交 \[表单有效] / 验证表单

text

```

### 初始和终止
- ● 初始状态（实心圆）
- ◎ 终止状态（实心圆外加圆圈）

---

## 3.3 状态图的用途

### 主要用途
1. 建模对象生命周期：如订单状态（新建、已支付、已发货、已完成）
2. 设计事件驱动系统：如UI控件的状态变化
3. 验证状态合法性：确保不会出现非法状态转换
4. 生成测试用例：覆盖所有状态和转换

### 典型场景
- 订单状态管理
- 工作流引擎
- 协议状态机（如TCP连接）
- 数据库事务

---

## 3.4 用AI生成状态图

### 提示词要点
- 明确对象
- 列出所有状态（包括初始和终止）
- 定义转换（事件+条件+动作）
- 指定内部活动

### 示例提示词
```

为数据库连接对象生成状态图：\
状态：已创建、认证中、空闲、忙碌、超时、关闭\
转换事件：connect()、认证成功、执行查询、查询完成、空闲超时、close()\
在空闲状态添加entry/exit动作

text

```

# Part 4: 活动图深度解析

---

## 4.1 活动图的本质

### 活动图 = 增强版的流程图
- 支持并行和分叉/汇合
- 支持泳道（划分责任区域）
- 支持对象流（数据传递）

### 关键概念
- **活动节点**：执行的动作
- **控制节点**：决策、分叉、汇合、合并
- **边**：控制流
- **泳道**：角色或组织单元

---

## 4.2 活动图的元素详解

### 活动节点
```

┌─────────┐\
│ 活动 │\
└─────────┘

text

```

### 控制节点
```

● 初始节点\
◎ 终止节点\
◇ 决策节点（分支）\
▽ 合并节点\
── 分叉节点（并发开始）\
── 汇合节点（并发结束）

text

```

### 泳道
```

**泳道A**

**泳道B**

活动1

<br />

<br />

活动2

text

```

---

## 4.3 活动图的用途

### 主要用途
1. 业务流程建模：如订单处理、报销审批
2. 算法逻辑描述：如查询优化流程
3. 并发设计：并行执行的任务
4. 用例细化：描述用例的多种路径

### 典型场景
- 业务工作流
- 并行计算流程
- 异常处理流程
- 系统集成流程

---

## 4.4 用AI生成活动图

### 提示词要点
- 明确泳道
- 描述流程步骤（按顺序）
- 指出分支条件
- 指出并行任务

### 示例提示词
```

为订单处理流程生成活动图：\
泳道：客户、系统、仓库\
流程：客户提交订单→系统验证→（验证失败返回错误，验证通过扣库存）→创建发货单→并行：仓库拣货、通知客户→仓库发货→系统更新状态

text

```
# Part 5: 架构图与部署图

---

## 5.1 架构图的本质

### 架构图描述系统的宏观结构
- **组件**：模块、服务、库
- **接口**：提供的接口、需要的接口
- **依赖**：组件之间的依赖关系

### 常用架构图类型
- **组件图**：物理模块及接口
- **包图**：命名空间组织
- **部署图**：物理节点和网络

---

## 5.2 组件图详解

### 元素
```

\[组件] ┌─────────┐\
│ 组件名 │\
└─────────┘\
接口 ○── 提供的接口（球）\
──○ 需要的接口（插座）\
依赖 ───> 依赖关系

text

```

### 用途
- 展示系统模块划分
- 定义模块间契约
- 指导开发和测试

---

## 5.3 部署图详解

### 元素
```

节点 ┌─────────┐\
│ 节点名 │\
│ <<设备>>│\
└─────────┘\
组件部署 \[组件] 放在节点内\
通信路径 ──── 节点间连接

text

```

### 用途
- 规划物理部署
- 分析网络性能
- 设计容灾方案
```
