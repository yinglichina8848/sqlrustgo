# Part 1: 性能优化的Harness视角

---

## 1.1 为什么AI难以保证性能优化正确性

### LLM的"局部最优"陷阱

```
开发者：帮我优化这个查询的性能
AI：     好的，我帮你加个索引

实际情况：
  • AI只看到了当前查询，没看到其他依赖该表的查询
  • 加索引可能加快读，但减慢写
  • 索引占用额外存储空间
  • 多表场景下索引效果难以预测
```

### AI性能优化的上下文限制

```
┌─────────────────────────────────────────────────────┐
│           AI看到的上下文                              │
│  当前文件 / 当前函数 / 当前Query                     │
├─────────────────────────────────────────────────────┤
│  AI看不到的：                                        │
│  • 其他模块对该函数的调用关系                         │
│  • 索引对整体读写比例的影响                          │
│  • 并发场景下的锁竞争情况                            │
│  • 数据量增长后的扩展性                              │
└─────────────────────────────────────────────────────┘
```

### Harness的补偿机制

```
AI优化代码
    │
    ├──→ GitNexus impact分析（识别所有调用点）
    │
    ├──→ TPC-H基准测试（验证全查询集性能）
    │
    ├──→ check_regression.sh（自动检测性能回退）
    │
    └──→ 人工Review（AI+人=完整视野）
```

## 1.2 性能优化的三层验证

### Layer 1：单Query验证（AI可以做）

```rust

    let db = setup_database();
    b.iter(|| {

    });

```

```
TPC-H SF=1
  ├── Q1:  Pricing Summary Report
  ├── Q2:  Minimum Cost Supplier
  ├── Q3:  Shipping Priority
  ├── ... (共22个Query)
  └── Q22: Global Sales Opportunity

验证：所有Query性能都在可接受范围内
     + 任意Query性能退化 > 5% → Gate Fail
```

### Layer 3：真实工作负载（最难验证）

```
并发场景：
  • 100个并发连接
  • 混合读写操作
  • 长时间运行

验证：吞吐量、响应时间、锁竞争、内存使用
```

# Part 2: TPC-H基准测试

---

## 2.1 TPC-H是什么

### 基准测试的定义

> **TPC-H是一个决策支持基准测试，模拟商业数据仓库的复杂查询场景。**

### TPC-H的核心价值

```
为什么TPC-H是数据库的"标准体检"？

1. 真实性：22个Query代表真实业务场景
2. 复杂性：多表JOIN、聚合、子查询
3. 标准化：全球数据库厂商共同认可
4. 可比性：不同数据库同一基准可比较

通过TPC-H = 数据库"功能完整 + 性能达标"
```

### TPC-H的22个Query

| 类别 | Query | 复杂度 |
|------|-------|--------|
| 聚合 | Q1 | 单表聚合统计 |
| JOIN | Q2, Q3, Q4, Q5, Q6, Q7, Q8, Q9, Q10, Q11, Q12, Q13, Q14, Q15, Q16, Q17, Q18, Q19, Q20, Q21 | 多表JOIN |
| 子查询 | Q3, Q7, Q9, Q10, Q11, Q13, Q15, Q17, Q18, Q19, Q21, Q22 | 嵌套查询 |

## 2.2 TPC-H在Gate中的角色

### Gate的TPC-H检查流程

```

    ▼

    │
    ├──→ cargo build --all-features ✅
    ├──→ cargo test ≥90% ✅

    结果：

```

## 2.3 TPC-H的评分标准

### 通过标准（Gate阈值）

```
┌────────────────────────────────────────────────────────┐
│            TPC-H SF=1 Gate 通过标准                    │
├────────────────────────────────────────────────────────┤
│  Query      结果正确性    最大响应时间                  │
│  ──────────────────────────────────────────           │
│  Q1         ✅            < 5秒                      │
│  Q2         ✅            < 10秒                     │
│  Q3         ✅            < 10秒                     │
│  ...        ...           ...                         │
│  Q22        ✅            < 15秒                     │
├────────────────────────────────────────────────────────┤
│  全部通过 = Gate Pass                                 │
│  任意失败 = Gate Fail                                 │
└────────────────────────────────────────────────────────┘
```

### 为什么设置时间阈值

```
Query超时 = 系统不可用

风险分析：
  • Q1超时 → 报表无法生成 → 业务中断
  • Q5超时 → 订单查询卡住 → 用户流失
  • Q18超时 → 大订单分析失败 → 决策延误

Gate阈值 = 业务可接受的最大响应时间
```

# Part 3: 回归检测机制

---

## 3.1 什么是回归

### 回归的定义

> **回归 = 之前通过的检查，现在失败了**
> **性能回归 = 之前达标的性能，现在退化了**

### 回归的分类

```
功能回归：
  之前：cargo test 100%通过
  现在：cargo test 85%通过 ← 某些测试失败了

性能回归：
  之前：Q1查询 2秒
  现在：Q1查询 8秒 ← 性能严重退化

覆盖回归：
  之前：覆盖率 80%
  现在：覆盖率 65% ← 新代码未测试
```

---

## 3.2 check_regression.sh的原理

### 为什么需要自动化回归检测

```
手动回归检测的问题：
  • 开发者不知道哪些Query会受影响
  • 人眼无法发现微妙性能变化
  • 每次Review手动对比数据不现实
  • 大型项目回归检测耗时数小时

check_regression.sh的价值：
  • 自动化：CI触发，无需人工介入
  • 量化：明确的Δ百分比
  • 可追溯：历史数据存储
  • 快速：比人工快100倍
```

### check_regression.sh的执行流程

```
1. 读取基准数据 (current.json)
         │
         ▼
2. 执行测试/Query
         │
         ▼
3. 对比结果
         │
         ├──→ Δ = 0% → 缓存命中，使用基准
         │
         └──→ Δ ≠ 0% → 计算实际差异
                        │
                        ├──→ Δ < 阈值 → Pass
                        └──→ Δ > 阈值 → Fail → Gate Block
```

### 缓存机制与风险

```
check_regression.sh --skip-run

工作原理：
  • 不实际执行测试
  • 直接使用 current.json 缓存数据
  • 返回 Δ = 0%

风险：
  • 如果 current.json 是旧的 → Δ = 0% 是假象
  • 实际上性能已经退化，但未被发现

教训：
  --skip-run 不可用于 Gate 决策
  必须真实运行才能发现 concurrent_select_8t 56%回归
```

## 3.3 性能回归的实际案例

### Case: concurrent_select_8t 回归

```
问题发现：
  check_regression.sh 真实运行后发现
  concurrent_select_8t 性能退化 56%

根因分析：
  • 并发控制锁竞争加剧
  • B+Tree锁粒度过细
  • MVCC版本链过长

修复流程：
  1. 创建Issue描述问题 (#xxx)
  2. 分析锁竞争热点
  3. 优化锁算法
  4. 验证修复
  5. 合并PR
  6. Issue关闭
```

### 回归检测的闭环

```
┌─────────────────────────────────────────────────────────┐
│              check_regression.sh 闭环                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  基准数据 (current.json)                                │
│       ▲                                                │
│       │                                                │
│       │ 读取                                           │
│       │                                                │
│  ┌────┴─────┐                                          │
│  │ 执行测试  │                                          │
│  └────┬─────┘                                          │
│       │                                                │
│       │ 对比                                           │
│       ▼                                                │
│  ┌──────────┐                                          │
│  │ Δ > 阈值 │                                          │
│  └────┬─────┘                                          │
│       │                                                │
│       ├──→ Gate Fail → PR Blocked                      │
│       │         │                                      │
│       │         └──→ 创建Issue → 修复 → PR → 验证      │
│       │                                                   │
│       └──→ Δ ≤ 阈值 → Gate Pass                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 3.4 回归检测与Gate的联动

### Gate失败时的处理流程

```
BP2 Gate执行
    │
    ├──→ cargo test --all-features  ✅ (90% pass)
    ├──→ cargo llvm-cov ≥75%       ✅
    │
    └──→ check_regression.sh        ❌ FAIL
                                    │
                                    ├──→ concurrent_select_8t Δ=56%
                                    │
                                    ▼
                            PR Blocked
                                    │
                                    ▼
                            Issue #xxx 创建
                                    │
                                    ▼
                            开发者修复
                                    │
                                    ▼
                            新PR → 重新Gate
                                    │
                                    ▼
                              Gate Pass → 合并
```

### E-09 Floor在回归检测中的应用

```
check_regression.sh 失败
    │
    ├──→ Δ > 10% → 必须修复（风险不可接受）
    │
    └──→ Δ 5%~10% → 评估是否可接受
                    │
                    ├──→ 已知问题列表中有记录
                    ├──→ 有明确修复计划
                    └──→ E-09豁免 → 继续推进
```

# 核心知识点总结

---

## 1. 性能优化的Harness视角

- **AI局限**：局部最优陷阱，上下文有限
- **Harness补偿**：TPC-H + GitNexus + 回归检测
- **三层验证**：单Query → 全基准 → 真实负载

## 2. TPC-H基准

- **价值**：22个Query覆盖真实业务场景
- **Gate角色**：SF=1全部通过才算性能达标
- **阈值**：每Query有最大响应时间限制

## 3. 回归检测

- **check_regression.sh**：自动化量化检测
- **缓存陷阱**：--skip-run 不可用于Gate决策
- **闭环流程**：检测 → Block → Issue → 修复 → 验证
