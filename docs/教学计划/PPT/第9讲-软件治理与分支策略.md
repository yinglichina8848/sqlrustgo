## 1.2 软件治理的核心要素

### 分层治理模型

```
┌─────────────────────────────────────────────────────────────────────┐
│                    软件治理金字塔                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│                         ▲                                           │
│                        /│\                                          │
│                       / │ \           治理层                        │
│                      /  │  \        (流程规范)                     │
│                     /───┼───\                                       │
│                    /    │    \        约束层                        │
│                   /     │     \      (规则约束)                    │
│                  /─────┼─────\                                     │
│                 /      │      \       工具层                       │
│                /       │       \    (自动化检查)                   │
│               /────────┴────────\                                   │
│                                                                      │
│  基础层：代码规范 → 约束层：分支/PR/CI → 治理层：流程/文化      │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 治理工具链

| 层次 | 工具 | 作用 |
|------|------|------|
| 基础层 | 代码规范(ESLint, Rustfmt) | 统一风格 |
| 约束层 | 分支保护, PR审查 | 强制协作 |
| 工具层 | CI/CD, 自动化检查 | 自动验证 |
| 治理层 | 流程, 文化 | 长期健康 |

# Part 2: Git分支策略

---

## 2.1 分支策略的核心概念

### 为什么需要分支？

- **隔离**：每个AI/功能独立工作
- **并行**：多个任务同时进行
- **可追溯**：知道每次改了什么
- **可回滚**：出问题能恢复

### 分支类型

| 分支 | 用途 | 生命周期 | 管理者 |
|------|------|----------|-------|
| main | 正式版本 | 永久 | 机器人 |
| develop | 开发主线 | 永久 | 机器人 |
| feature/* | 新功能 | 临时 | AI/开发者 |
| fix/* | Bug修复 | 临时 | AI/开发者 |
| refactor/* | 重构 | 临时 | AI/开发者 |

---

## 2.2 常见分支策略对比

### Git Flow（适合多版本维护）

```
main ───●────●────●────●────>
         │         │
develop ──●────●────●────●────>
          │    │
feature ──●────    │
               │    │
fix ──────────●
```

**优点**：结构清晰，职责分明
**缺点**：分支多，管理复杂

### GitHub Flow（适合持续发布）

```
main ──────●────●────●────●────>
           │    │
feature ───●────●
```

**优点**：简单易用，适合快速迭代
**缺点**：不适合多版本

### Trunk-Based（适合多AI协作）

```
main ──────●────●────●────●────>
           │    │    │
feature ───●────●────●
```

**优点**：分支少，合并简单
**缺点**：需要强大的自动化测试

## 2.3 SQLRustGo分支策略

### 推荐策略：改进的GitHub Flow

```
main ────────────────────────────────────────────────────►
  │
  ├── develop/v2.6.0 ────────────────────────────────►
  │    │
  │    ├── feature/ai-parser       (AI #1 负责)
  │    ├── feature/ai-executor     (AI #2 负责)
  │    ├── feature/ai-storage      (AI #3 负责)
  │    │
  │    └── 定期合并到develop ─────────────────►
  │
  └── 合并到main ──────────────────────► (发布)
```

### 分支命名规范

```
feature/ai-{模块名}-{功能描述}
fix/ai-{模块名}-{问题描述}
refactor/ai-{模块名}-{改进内容}

示例：
feature/ai-parser-add-join-support
fix/ai-storage-fix-memory-leak
refactor/ai-executor-optimize-hash-join
```

# Part 3: 分支保护与代码审核

---

## 3.1 分支保护规则

### 必须保护的分支

- `main` - 正式版本，任何错误直接影响用户
- `develop/v2.6.0` - 开发主线，代码质量标杆

### 保护规则配置

| 规则 | 作用 | 为什么要 |
|------|------|---------|
| Require PR | 必须通过PR合并 | 强制审查 |
| Require 1 approval | 需要1人审批 | 人工把关 |
| Require status checks | 需要CI通过 | 自动化验证 |
| Require up to date | 需要最新代码 | 防止落后 |
| Disallow force push | 禁止强制推送 | 保护历史 |
| Disallow deletion | 禁止删除 | 保护分支 |

---

## 3.2 代码审核要点

### AI提交的代码，审核什么？

```
┌─────────────────────────────────────────────────────────────────────┐
│                    AI代码审核清单                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  □ 功能正确性                                                       │
│    ├── 代码实现了预期功能吗？                                        │
│    ├── 边界条件处理了吗？                                           │
│    └── 错误处理完善了吗？                                           │
│                                                                      │
│  □ 代码质量                                                         │
│    ├── 代码清晰易读吗？                                             │
│    ├── 命名规范吗？                                                 │
│    └── 有重复代码吗？                                               │
│                                                                      │
│  □ 测试覆盖                                                         │
│    ├── 有单元测试吗？                                               │
│    ├── 测试覆盖了新代码吗？                                         │
│    └── 测试能捕获回归吗？                                            │
│                                                                      │
│  □ 安全考虑                                                         │
│    ├── 有安全漏洞吗？                                               │
│    ├── 敏感信息处理得当吗？                                         │
│    └── 依赖安全吗？                                                 │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### AI审核的特别注意点

| AI容易出的问题 | 人类要特别检查 |
|---------------|---------------|
| 过度工程 | 简化逻辑 |
| 奇怪命名 | 遵循规范 |
| 过度注释 | 必要注释 |
| 复制代码 | 复用现有 |

---

## 3.3 合并冲突处理

### 预防冲突的策略

1. **小步提交**：每次只改一件事
2. **频繁合并**：尽早合并，减少差异
3. **模块隔离**：不同模块由不同AI负责

### 冲突处理流程

```
检测到冲突
    │
    ▼
判断冲突类型
    │
    ├─ 简单冲突（格式/注释）→ 自动解决
    │
    ├─ 逻辑冲突（两人改了同一功能）→ 人类协调
    │
    └─ 严重冲突（架构级改动）→ 重新设计
```

# 核心知识点总结

---

## 1. 软件治理

- **What**：用规则约束多AI协作
- **Why**：防止冲突、保证质量
- **How**：分支隔离 + PR审查 + CI验证

## 2. Git分支策略

- **隔离**：feature分支互不干扰
- **合并**：通过PR审查后合并
- **保护**：主分支禁止直接修改

## 3. 代码审核

- **人工审查**：补充AI的不足
- **自动化检查**：CI/CD强制执行
- **冲突处理**：预防为主，解决为辅
