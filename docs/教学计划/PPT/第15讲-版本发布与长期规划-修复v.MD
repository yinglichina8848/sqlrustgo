## 1.1 版本发布的三层Gate

### 发布前的Gate检查

```
开发分支合并到main
        │
        ▼
    BP1 Gate ──── Fail → 修复
        │ (静态检查)
        │ Pass
        ▼
    BP2 Gate ──── Fail → 修复
        │ (集成检查)
        │ Pass
        ▼
    BP3 Gate ──── Fail → E-09评估
        │ (Beta检查)       或修复
        │ Pass
        ▼
    版本Tag创建
        │
        ▼
    Release发布
```

### Beta版本 vs 正式版本

```
正式发布（v3.0.0）：
  • 需要BP1 + BP2 + BP3全部通过
  • 需要31个Issues全部闭环
  • 需要KPI全部达标
  • 需要文档完善

Beta发布（v3.0.0-beta）：
  • BP1 + BP2必须通过
  • BP3可使用E-09 Floor
  • KPI部分达标即可
  • 允许已知问题存在
```

## 1.2 版本号管理

### 语义化版本（SemVer）

```
格式：MAJOR.MINOR.PATCH

MAJOR：不兼容的API变更
MINOR：向后兼容的功能新增
PATCH：向后兼容的Bug修复

SQLRustGo版本示例：
  v2.9.0 → v2.9.1 (PATCH: Bug修复)
  v2.9.1 → v3.0.0 (MAJOR: Beta Phase 2)
  v3.0.0 → v3.0.1 (PATCH: Bug修复)
  v3.0.0 → v3.1.0 (MINOR: 新功能)
```

### Beta版本的命名

```
v3.0.0-beta.1    → Beta第1个版本
v3.0.0-beta.2    → Beta第2个版本
v3.0.0-rc.1      → Release Candidate第1个版本
v3.0.0           → 正式发布

阶段定义：
  Alpha → 内部测试
  Beta  → 外部测试（功能完整，质量待验证）
  RC    → 发布候选（质量验证中）
  GA    → 正式发布
```

## 1.4 版本发布的闭环流程

### 完整的发布闭环

```
1. 需求收集
   └→ Issue创建

2. 开发实施
   └→ PR创建 → Gate检查 → 合并

3. 版本准备
   └→ 分支合并 → BP1/BP2/BP3检查

4. 版本发布
   └→ Tag创建 → Release发布 → 文档更新

5. 监控跟踪
   └→ KPI监控 → 问题反馈 → 新Issue
```

### 为什么需要闭环

```
没有闭环：
  • Issue漂移（创建后无人跟踪）
  • PR堆积（合并后Issue未关闭）
  • 知识丢失（经验未沉淀到文档）

有闭环：
  • 每个Issue有负责人
  • 每个PR有关联的Issue
  • 每个版本有完整的Release Notes
  • 知识沉淀到GBrain SSOT
```

## 2.1 BetaPhase2的战略目标

### 从功能到可信度的转变

```
BetaPhase1（功能优先）：
  目标：实现核心功能
  评估：功能是否可用
  代价：质量可能不稳定

BetaPhase2（可信度优先）：
  目标：证明系统正确
  评估：质量指标是否达标
  代价：功能可能延迟交付

核心理念：
  "能不能证明正确" > "功能是否实现"
```

### BetaPhase2的五类KPI

```
┌────────────────────────────────────────────────────────────┐
│              BetaPhase2 KPI 目标                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Crash率     → 下降25%  （Crash次数/月）                   │
│  稳定性      → 提升20%  （MTBF平均无故障时间）             │
│  SQL兼容性   → 提升20%  （TPC-H通过率）                    │
│  审计完整性  → 提升15%  （Audit Trail覆盖率）               │
│  死锁率     → 下降15%  （死锁次数/月）                     │
│  锁竞争      → 下降5%   （锁等待时间）                     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 2.3 三层Wiki架构

### 知识管理的三层结构

```
┌─────────────────────────────────────────────────────────────┐
│                   三层Wiki架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Layer 1: QMD源文件（内容来源）                             │
│  位置：仓库 docs/ 目录                                      │
│  用途：Markdown源文件，可版本控制                           │
│  特点：内容最完整，可直接编辑                               │
│                                                             │
│  Layer 2: GBrain SSOT（知识索引）                           │
│  位置：192.168.0.250:8081                                  │
│  用途：结构化知识库，可跨节点查询                           │
│  特点：内网访问，与Wiki同步                                 │
│                                                             │
│  Layer 3: GitHub Wiki（内容展示）                           │
│  位置：GitHub/Gitea Wiki页面                               │
│  用途：最终用户可读的内容                                   │
│  特点：可直接访问，但内容可能滞后                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 三层同步机制

```
QMD源文件变更
        │
        ├──→ push到仓库
        │
        ├──→ GitHub Wiki自动同步（push sync）
        │
        └──→ GBrain SSOT更新（内网同步）
                      │
                      └──→ 跨节点可查询
```

### 为什么需要三层

```
单一Wiki的问题：
  • Wiki内容与代码不同步
  • 知识分散在多个平台
  • 跨平台查询困难

三层架构的价值：
  • QMD：版本控制，内容可追溯
  • GBrain：结构化查询，知识可发现
  • Wiki：用户友好，内容可访问
```

## 2.4 GBrain SSOT的治理作用

### 什么是SSOT

> **SSOT（Single Source of Truth，单一事实来源）= 权威的、可信的、唯一的知识来源**

### GBrain SSOT的价值

```
知识管理的问题：
  • Notion有部分知识
  • Confluence有部分知识
  • README有部分知识
  • 邮件里有部分知识
  • 分散 → 找不到 → 重复工作

GBrain SSOT的价值：
  • 所有项目知识集中存储
  • 跨节点查询
  • 版本同步
  • 权威可信
```

### GBrain SSOT的内容

```
GBrain SSOT存储内容：
  • 10条Pattern（设计模式）
  • 5条Strategy（策略）
  • 3条Episode（关键事件）
  • 10条Fact（核心事实）

当前状态：
  • 28条知识已写入SSOT
  • 同步到Wiki供外部访问
  • 内网节点(250)可拉取
```

## 3.4 持续改进机制

### Gate数据的分析闭环

```
Gate执行 → 数据收集 → 趋势分析 → 优化措施

示例：覆盖率改进
  Month 1: 覆盖率 65%
  Month 2: 覆盖率 70% (+5%)
  Month 3: 覆盖率 75% (+5%)
  Month 4: 覆盖率 78% (+3%) ← 增速放缓

分析：
  • 新代码覆盖率提升困难
  • 现有代码测试已饱和

优化措施：
  • 引入mutation testing
  • 重点关注核心模块覆盖率
  • 调整覆盖率阈值分布
```

### 流程优化的ISO 9001闭环

```
Plan（计划）：
  定义下一阶段的质量目标

Do（执行）：
  实施Gate系统和检查

Check（检查）：
  收集Gate数据，分析趋势

Act（改进）：
  根据分析结果优化流程
```
