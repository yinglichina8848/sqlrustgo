## 💡 每周核心思考问题

每周围绕以下三个问题进行反思：

### 问题1：如何优化和改进，提升SE的效率和质量？

- AI能帮我自动化哪些工作？
- 哪些工作必须人工完成？

### 问题2：AI编程时代，人类的哪些工作和能力是AI无法取代的？
- AI不知道"用户真正想要什么"
- AI没有"业务感觉"
- AI擅长"常见情况"，但"例外"才是真正考验

### 问题3：我应该加强哪方面的学习和练习？
- 工具使用 → 问题分析 → 系统思维 → 沟通表达 → 价值判断
- 我现在会什么？还不会什么？应该学什么？

## 🕹️ 第八周：测试驱动开发与Alpha版本（手动档第1周）

> **本阶段**：🕹️ 手动档 - 从AI辅助转向手动实现
> **核心转变**：从"让AI帮我做"到"我自己做，AI辅助审查"

### 实验目的

1. 掌握测试驱动开发（TDD）方法
2. 能够使用AI辅助生成测试用例
3. 能够提高测试覆盖率至70%以上
4. 能够完成Alpha版本发布

### 🚗→🕹️ 阶段转变：从自动档到手动档

**本周是"手动档"的开始！**

| 自动档阶段 | 手动档阶段 |
|-----------|-----------|
| AI生成代码，我来运行 | 我自己写代码，AI帮我审查 |
| "帮我做一个..." | "帮我看看我写的对不对" |
| 追求结果正确 | 追求理解原理 |

**本周的核心转变**：
- 不再完全依赖AI生成代码
- 自己动手写测试用例
- AI的角色从"执行者"变成"审查者"

### 实验环境

- Rust工具链（1.75+）
- TRAE IDE
- Git
- SQLRustGo项目代码

### 实验内容

#### 任务1：运行现有测试并分析覆盖率（20分钟）

```bash
# 1. 克隆最新代码
git pull origin develop/v2.6.0

# 2. 运行所有测试
cargo test --all-features

# 3. 安装覆盖率工具
cargo install cargo-tarpaulin

# 4. 生成覆盖率报告
cargo tarpaulin --out Html --output-dir coverage

# 5. 查看覆盖率
# 打开 coverage/tarpaulin-report.html
```

**分析要点**：
- 当前覆盖率是多少？
- 哪些模块覆盖率较低？
- 哪些代码路径未被测试覆盖？

#### 任务2：使用AI辅助生成测试用例（30分钟）

**提示词示例**：

```
我需要为SQLRustGo的词法分析器生成测试用例。
现有代码位于 src/parser/lexer.rs
请生成以下测试：
1. 关键字识别测试（SELECT, FROM, WHERE, INSERT, UPDATE, DELETE等）
2. 标识符识别测试
3. 数字字面量测试
4. 字符串字面量测试
5. 运算符识别测试
6. 边界条件测试（空输入、单字符、特殊字符等）

请使用Rust的 #[test] 属性编写测试代码。
```

**扩展任务**：为语法分析器生成测试用例

#### 任务3：补充测试用例并验证覆盖率（30分钟）

1. 根据AI生成的测试建议，补充缺失的测试
2. 运行测试确保全部通过
3. 重新生成覆盖率报告
4. 验证覆盖率是否达到70%

```bash
# 运行特定模块的测试
cargo test --package parser
cargo test --package executor
cargo test --package storage

# 验证覆盖率
cargo tarpaulin --out Html --packages parser,executor,storage
```

#### 任务4：运行质量门禁检查（20分钟）

```bash
# 1. 编译检查
cargo build --all-features

# 2. 测试检查
cargo test --all-features

# 3. Clippy检查
cargo clippy --all-features -- -D warnings

# 4. 格式化检查
cargo fmt --check --all
```

#### 任务5：创建Alpha版本（20分钟）

```bash
# 1. 创建Alpha版本标签
git tag -a v0.1.0-alpha -m "Alpha版本发布"

# 2. 推送标签
git push origin v0.1.0-alpha

# 3. 创建GitHub Release（在GitHub网页上操作）
```

3. 完成Alpha版本发布
4. 提交实验报告

### 实验目的

1. 理解Git分支策略
2. 能够配置分支保护规则
3. 掌握多AI协同开发模式
4. 能够创建和管理功能分支

### 实验环境

- GitHub账号
- Git客户端
- SQLRustGo仓库

### 实验内容

#### 任务1：分析SQLRustGo现有分支策略（15分钟）

```bash
# 1. 查看所有分支
git branch -a

# 2. 查看分支历史
git log --oneline --graph --all --decorate

# 3. 查看远程分支
git fetch origin
git branch -r
```

**分析要点**：
- 当前的分支结构是什么？
- main分支和develop分支的关系？
- feature分支的命名规范？

#### 任务2：配置开发分支（20分钟）

```bash
# 1. 确保本地develop分支最新
git checkout develop/v2.6.0
git pull origin develop/v2.6.0

# 2. 创建功能分支
git checkout -b feature/week9-lab

# 3. 创建初始提交
git commit --allow-empty -m "chore: create feature branch for week9 lab"

# 4. 推送到远程
git push origin feature/week9-lab
```

#### 任务3：配置分支保护规则（25分钟）

在GitHub上配置：

1. 进入仓库 Settings → Branches
2. Add rule → 输入 `develop/v2.6.0`
3. 配置保护规则：
   - ✅ Require pull request reviews before merging（1 approval）
   - ✅ Require status checks to pass before merging
   - ✅ Require branches to be up to date before merging
   - ❌ Allow force pushes（禁用）
   - ❌ Allow deletions（禁用）

#### 任务4：测试分支保护（20分钟）

```bash
# 1. 尝试直接推送 develop 分支（应该失败）
git checkout develop/v2.6.0
echo "test" >> README.md
git commit -m "test: direct commit"
git push origin develop/v2.6.0  # 应该被拒绝

# 2. 通过PR方式合并
git checkout feature/week9-lab
echo "test via PR" >> README.md
git commit -m "docs: test PR workflow"
git push origin feature/week9-lab
# 然后在GitHub上创建PR
```

### 实验要求

1. 理解分支策略的概念和作用
2. 能够配置分支保护规则
3. 完成功能分支的创建和推送
4. 理解PR工作流

### 实验报告内容

1. 分支策略分析报告
2. 分支保护规则配置截图
3. PR创建和审核流程记录
4. 遇到的问题和解决方法

### 实验环境

- GitHub账号
- SQLRustGo项目

### 实验内容

#### 任务1：执行发布前检查（20分钟）

```bash
# 1. 确保代码是最新的
git checkout develop/v2.6.0
git pull origin develop/v2.6.0

# 2. 运行所有门禁检查
./scripts/run_gates.sh

# 3. 确认所有检查通过
```

#### 任务2：创建版本标签（15分钟）

```bash
# 1. 创建版本标签
git tag -a v1.0.0 -m "Release v1.0.0 - First stable release"

# 2. 推送标签
git push origin v1.0.0

# 3. 验证标签
git tag -l
git show v1.0.0
```

#### 任务3：创建GitHub Release（25分钟）

在GitHub上创建Release：

1. 进入仓库 Releases 页面
2. 点击 "Draft a new release"
3. 填写信息：
   - Tag: v1.0.0
   - Title: SQLRustGo v1.0.0 - 首个稳定版本
   - Release notes: 见下文

创建发布说明：
