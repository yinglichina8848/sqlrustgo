## 一、概述

本文档记录所有已审批的门禁豁免（Exemption）和延期（Deferral）项。每次 R-Gate / G-Gate 通过前，所有豁免/延期必须在此记录并经过 Tech Lead 审批。

**重要原则**: 豁免不等于绕过。豁免是对不可抗力因素（如基础设施缺失、外部依赖不可控）的理性承认，不是对质量标准的放弃。

## 三、豁免/延期类型定义

| 类型 | 说明 | 示例 |
|------|------|------|
| **豁免 (Exemption)** | 某项检查在当前版本不满足，但有合理解释，不影响发布 | storage 覆盖率低但已规划改进路线 |
| **延期 (Deferral)** | 某项检查因基础设施未就绪，延至下一版本 | 性能基准尚未建立 |
| **先决条件 (Prerequisite)** | 检查本身必须做，但需要先完成准备工作 | R9 需先建立基准才能执行回归判定 |
| **条件通过 (Conditional Pass)** | 某项在特定条件下通过，主条件不满足但替代指标达标 | QPS 退化 ≤5% 视为 PASS |

---

## 四、复审机制

每个豁免/延期项都有 **复审日期**。在复审日期前，责任方必须：

1. 提供进展报告（更新本文档的"备注"列）
2. 或证明问题已解决，申请关闭豁免
3. Tech Lead 判断是否续期豁免

逾期未复审的豁免项视为 **自动失效**，下一版本门禁必须重新检查。

---

## 五、豁免申请流程

```
发现门禁不满足 → 评估是否可豁免
        │
        ├── 否 → 修复代码/配置使检查通过
        │
        └── 是 → 填写豁免记录（ID、原因、审批人、复审日期）
                   │
                   ▼
              Tech Lead 审批
                   │
                   ▼
              记录到本文档
                   │
                   ▼
              门禁通过（带豁免记录）
```
