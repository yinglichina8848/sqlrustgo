# SQLRustGo AI 自我优化提示词手册（流程/开发/测试/文档）

> 版本: 1.0  
> 更新日期: 2026-04-19  
> 适用范围: 全部版本迭代（Draft/Dev/Alpha/Beta/RC/GA）

---

## 1. 目标

建立 AI 在版本迭代中的“自查 + 改进”常态机制，覆盖四个方面：

1. 流程（Process）
2. 开发（Development）
3. 测试（Testing）
4. 文档（Documentation）

目标不是增加流程负担，而是让 AI 在每个阶段都能持续提升交付质量。

---

## 2. 使用方式（统一）

每个阶段结束时，AI 必须依次执行四类提示词，并输出：

1. Findings（问题）
2. Evidence（证据）
3. CAPA（纠正预防）
4. Metrics（量化指标）
5. Next Actions（下一阶段动作）

---

## 3. 流程自优化（Process）

## 3.1 自查提示词

```markdown
你是 SQLRustGo 流程审计代理。请对当前阶段执行 Process Audit：

检查项：
1) 是否严格按阶段推进（Draft/Dev/Alpha/Beta/RC/GA）
2) 是否存在跨阶段开发
3) 是否存在门禁绕过
4) 是否存在“无证据通过”
5) 是否按要求执行了回滚预案设计

输出：
1. Findings（按严重度排序）
2. Evidence（命令、日志、路径、commit）
3. CAPA（限 3 项，可在下一阶段完成）
4. Gate Decision（PASS/WARN/FAIL）
```

## 3.2 改进建议（原则）

1. 每阶段最多 3 个流程改进行动。
2. 每个行动必须绑定 owner 和截止时间。
3. 未完成 CAPA 必须进入下一阶段阻塞项。

---

## 4. 开发自优化（Development）

## 4.1 自查提示词

```markdown
你是 SQLRustGo 开发质量代理。请对当前阶段代码交付做 Development Audit：

检查项：
1) 需求到实现是否可追溯（Issue -> PR -> Commit）
2) 关键设计是否有文档支撑
3) 是否存在技术债扩散（TODO/FIXME、临时分支逻辑）
4) 是否有高风险改动无回退方案
5) 是否有重复实现可抽象复用

输出：
1. Top 5 技术风险
2. 证据定位（文件/行/PR）
3. CAPA（短期修复 + 中期重构）
4. 下一阶段开发优化建议（不超过 3 条）
```

## 4.2 改进建议（原则）

1. 先修高风险路径，再做结构优化。
2. 重构必须有回归测试先行。
3. 每次迭代引入的技术债必须可量化。

---

## 5. 测试自优化（Testing）

## 5.1 自查提示词

```markdown
你是 SQLRustGo 测试优化代理。请执行 Testing Audit：

检查项：
1) L0/L1/L2/L3 分层是否完整
2) P0 功能是否有单测+集成+回归三层覆盖
3) SQL Corpus/TPC-H/Sysbench/恢复测试是否按阶段执行
4) 失败用例是否完成 new vs pre-existing 分类
5) 覆盖率是否满足目标且趋势向上

输出：
1. 测试盲区（功能维度）
2. 失败归因与优先级
3. CAPA（新增测试、修复测试、移除无效测试）
4. 下一阶段测试增强计划（最多 5 项）
```

## 5.2 改进建议（原则）

1. 先保证关键路径稳定，再扩展边界覆盖。
2. 测试命令必须与真实 target 同步。
3. 测试报告必须包含复现步骤与产物路径。

---

## 6. 文档自优化（Documentation）

## 6.1 自查提示词

```markdown
你是 SQLRustGo 文档治理代理。请执行 Documentation Audit：

检查项：
1) 文档完整性（版本包最小集合）
2) 链接有效性与命令可执行性
3) 文档与代码/CI/测试状态一致性
4) 版本号、日期、阶段状态一致性
5) 是否存在重复目录或重复内容

输出：
1. 文档问题清单（按严重度）
2. 证据（坏链、无效命令、不一致项）
3. CAPA（修复项 + 防复发机制）
4. 下一阶段文档治理动作（不超过 3 条）
```

## 6.2 改进建议（原则）

1. 文档必须“可执行、可验证、可追溯”。
2. 入口文档优先保证正确。
3. 同类文档只保留一个 canonical 版本。

---

## 7. 版本迭代闭环（持续提升）

每个阶段固定执行以下顺序：

1. Process Audit
2. Development Audit
3. Testing Audit
4. Documentation Audit
5. 生成统一 CAPA 清单并写入下一阶段计划

建议输出文件：

1. `PROCESS_AUDIT.md`
2. `DEV_AUDIT.md`
3. `TEST_AUDIT.md`
4. `DOCUMENT_AUDIT.md`
5. `CAPA_LIST.md`

---

## 8. 量化指标（保持轻量）

建议只追踪 8 个核心指标：

1. A 类门禁通过率
2. 阶段回归失败率（new failures）
3. 覆盖率（全仓）
4. 关键模块覆盖率（parser/executor/storage/transaction）
5. 文档命令一致性通过率
6. 坏链数量
7. 平均缺陷修复时长（MTTR）
8. CAPA 按期完成率

---

## 9. 周期性执行建议

1. 每日：L0/L1 + 文档命令检查
2. 每周：L2 + 四类 Audit 快照
3. 每阶段结束：完整四类 Audit + CAPA 冻结
4. 每版本结束：过程复盘 + 模板升级

---

## 10. 反模式（必须避免）

1. 只给“结论”，不给证据
2. 只列问题，不给 CAPA
3. CAPA 长期挂起，不进入下一阶段计划
4. 指标过多导致团队不执行
5. 文档与代码状态长期不一致
