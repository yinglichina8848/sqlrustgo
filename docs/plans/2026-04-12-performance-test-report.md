## 二、根本原因分析

### 2.1 核心问题

**SQLRustGo 在大数据量下性能差的原因是：索引存在但未被使用**

### 2.2 发现的问题

| 问题 | 位置 | 影响 |
|------|------|------|
| **scan() 返回全部记录** | `engine.rs:517-521` | 60万行全量扫描，谓词在executor层才过滤 |
| **索引方法未调用** | `StorageEngine` trait | `search_index()` 和 `range_index()` 存在但executor未调用 |
| **无索引扫描优化** | `SeqScanExec` | 没有尝试使用 `range_index` 进行范围查询 |
| **无代价模型决策** | `PathSelector` | 未根据代价选择使用索引扫描还是全表扫描 |

### 2.3 StorageEngine 已有接口

```rust
// StorageEngine trait (engine.rs:184-188)
fn search_index(&self, table: &str, column: &str, key: i64) -> Vec<u32>;
fn range_index(&self, table: &str, column: &str, start: i64, end: i64) -> Vec<u32>;

// MemoryStorage 已实现
indexes: HashMap<String, SimpleBPlusTree>
hash_indexes: HashMap<String, Arc<HashIndex<i64, u32>>>
```

### 2.4 执行流程问题

```
当前流程:
SQL → Parser → Planner → Executor
                    ↓
              SeqScanExec
                    ↓
              storage.scan()  ← 返回全部60万行
                    ↓
              FilterExec     ← 在executor层过滤

应该的流程:
SQL → Parser → Planner → Optimizer
                              ↓
                        选择 IndexScan
                              ↓
                        storage.range_index()  ← 只返回满足条件的row_ids
                              ↓
                        直接返回结果
```

## 三、优化方案

### 3.1 短期优化（1-2周）

| 优化项 | 预期效果 | 复杂度 |
|--------|----------|--------|
| 谓词下推到存储层 | 10-50x | 中 |
| 范围查询使用BTree | 100x+ | 低 |
| 主键/唯一索引利用 | 100x+ | 低 |
| LIMIT早期终止 | 2-5x | 低 |

### 3.2 中期优化（1个月）

| 优化项 | 预期效果 | 复杂度 |

| 查询计划缓存 | 重复查询10x | 中 |

## 五、结论

SQLRustGo 在小数据量下性能优异（向量化执行贡献3-4倍提升），但在大数据量下因索引未使用而落后于SQLite。

通过实现**谓词下推**和**索引扫描优化**，预期可将大数据量性能提升**10-100倍**，达到或超过SQLite水平。
