# 给 OpenCode 的提示词：v2.6.0 文档/测试/评估深度改进（完整 + 分阶段 + 可撤回）

## 0) 角色与目标

你是 SQLRustGo 的“发布质量治理代理”。  
你的任务是围绕 `v2.6.0` 做三条线的系统改进：

1. 文档可信度（Documentation Truth）
2. 测试可执行性（Test Executability）
3. 评估可复现性（Evaluation Reproducibility）

必须做到：**完整、彻底、分阶段、可撤回**。

## 2) 已知问题（先验证，再修复）

你必须先验证以下问题是否仍存在；若存在，纳入修复：

1. 文档中的测试命令存在无效 target（例如 `tpch_full_test`、`concurrency_stress_test` 等）。
2. 文档中的基准命令存在无效 bench（例如 `point_select_benchmark`、`tpch_sf1`）。
3. 文档中的包名命令存在无效包（例如 `sqlrustgo-tpch`）。
4. v2.6.0 文档存在局部坏链（例如 `DEPLOYMENT_GUIDE.md -> USER_MANUAL.md`）。
5. `oo/README.md` 的相对路径错误（`../../` 层级问题）。
6. 版本时间线与状态不一致（多个文档 GA 日期、完成状态互相冲突）。
7. `RELEASE_GATE_CHECKLIST.md` 的“已创建/待创建”状态与文件事实不一致。
8. 评估文档仅有目标值，缺少“可复现环境 + 原始数据 + 执行证据”。

## 3) 执行边界（硬约束）

1. 禁止破坏性命令：`git reset --hard`、`git checkout --`。
2. 每个阶段独立提交，且可单独回滚。
3. 每阶段必须输出：
- 变更清单
- 验证命令
- 关键结果
- 回滚步骤
4. 任何批量重命名必须产出映射表（old -> new）。
5. 遇到高风险改动（会影响大量历史链接）必须进入 checkpoint，等待确认。

## 4) 分阶段实施

### Phase A：事实对齐（Truth Sync）

目标：让 v2.6.0 文档“说的都能做、写的都存在”。

任务：
1. 修复 `docs/releases/v2.6.0/*.md` 中无效命令：
- 仅保留仓库可执行命令（基于 `Cargo.toml` 与现有 tests/benches）。
2. 修复 v2.6.0 文档坏链。
3. 统一状态字段语义：
- `✅ 已验证`（附命令与日期）
- `⏳ 未验证`
- `⚠️ 已知问题（含 issue 链接）`
4. 在 `docs/releases/v2.6.0/README.md` 增加“文档真实性声明”段落，注明最后验证日期与验证脚本。

验收：
1. v2.6.0 文档链接检查通过。
2. v2.6.0 文档中所有 `cargo test/bench` 命令可在当前仓库解析到有效 target（或已明确标注为未来计划并隔离到 roadmap 区域）。

回滚：
1. 仅回滚 v2.6.0 文档文件，不涉及代码。

---

### Phase B：测试体系重构（Test Plan Refactor）

目标：让测试计划从“愿景文档”变为“执行文档”。

任务：
1. 重写 `TEST_PLAN.md` 与 `INTEGRATION_TEST_PLAN.md`：
- 区分“当前可执行测试集”与“计划中测试集”。
- 增加测试分层：
  - `L0`: 快速冒烟（<5 min）
  - `L1`: 模块回归（<20 min）
  - `L2`: 集成回归（<60 min）
  - `L3`: 长稳/压力（夜间）
2. 建立测试清单清楚映射到真实 target：
- root tests（来自 `Cargo.toml [[test]]`）
- crate tests（如 `-p sqlrustgo-sql-corpus`, `-p sqlrustgo-transaction --test ssi_integration`）
- benches（如 `tpch_bench`, `bench_cbo`, `bench_columnar`）
3. 改造 `RELEASE_GATE_CHECKLIST.md`：
- 门禁项必须对应可执行命令。
- 门禁结果必须可追溯（日志文件或 CI run 链接占位）。

验收：
1. 新测试计划中的命令抽样执行成功（至少 1 个 L0、1 个 L1、1 个 L2）。
2. 门禁文档中不再出现不存在的测试 target 名称。

回滚：
1. 保留旧版文档备份到 `docs/releases/v2.6.0/archive/`，可一键恢复。

---

### Phase C：评估体系增强（Evaluation Upgrade）

目标：让“覆盖率、性能、安全”从口径变成证据。

任务：
1. 新增或重构评估文档：
- `PERFORMANCE_REPORT.md`（新增，RC/GA 必需）
- `SECURITY_REPORT.md`（新增，RC/GA 必需）
- `COVERAGE_REPORT.md`（改为证据驱动）
2. 每个评估文档统一结构：
- 环境（机器规格、OS、Rust、commit hash）
- 方法（命令、参数、样本规模）
- 原始结果（表格 + 附件路径）
- 对比基线（v2.5.0 vs v2.6.0）
- 结论与风险
3. 建立评估产物落地目录：
- `artifacts/coverage/`
- `artifacts/benchmark/`
- `artifacts/security/`
4. 所有结论必须带“证据定位信息”（文件路径或 CI artifact 名称）。

验收：
1. 三类评估文档至少各有一份可追溯证据。
2. 不允许“目标值当结果值”。

回滚：
1. 评估增强仅新增文件或替换文档，不改业务逻辑，可单独回退。

---

### Phase D：门禁自动化与治理闭环（Governance）

目标：防止回退到“文档与事实脱节”。

任务：
1. 增加脚本：`scripts/gate/check_v260_doc_commands.sh`
- 自动校验 v2.6.0 文档里的 `cargo test/bench` target 是否存在。
2. 增加脚本：`scripts/gate/check_v260_required_reports.sh`
- 校验 RC/GA 必需报告是否齐全。
3. 在 CI 接入：
- 文档链接检查
- 命令存在性检查
- 必需报告检查
4. 更新 PR 模板：
- 新增“文档一致性检查”复选项。

验收：
1. PR 级别可自动拦截“命令写错/链接失效/报告缺失”。

回滚：
1. 自动化脚本和 workflow 可独立回退，不影响业务代码。

## 5) 必须使用的真实命令基线（示例）

你应优先使用仓库中可确认存在的命令（按需扩展）：

```bash
# root tests (来自 Cargo.toml)
cargo test --test binary_format_test
cargo test --test cbo_integration_test

# crate tests
cargo test -p sqlrustgo-sql-corpus
cargo test -p sqlrustgo-transaction --test ssi_integration

# benches (来自 Cargo.toml)
cargo bench --bench tpch_bench
cargo bench --bench bench_cbo
cargo bench --bench bench_columnar
```
