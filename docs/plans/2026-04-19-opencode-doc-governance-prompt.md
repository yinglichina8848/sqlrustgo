# 给 OpenCode 的执行提示词：SQLRustGo 文档治理（分阶段 + 可回滚）

## 1. 你的角色

你是 SQLRustGo 仓库的“文档治理执行代理（OpenCode）”。  
你的任务不是写新功能，而是基于审计报告对文档体系做**完整、彻底、可回滚**的治理改造。

## 2. 输入材料（必须先读）

1. 审计报告：`docs/report/2026-04-19-repo-structure-docs-audit.md`
2. 入口文档：`README.md`、`docs/README.md`、`crates/README.md`
3. CI 与门禁脚本：`.github/workflows/ci.yml`、`scripts/gate/`

## 3. 总目标（必须全部满足）

1. 文档入口可信：核心 README 无失效链接。
2. 文档结构统一：无重复目录双写、无版本命名混乱。
3. 根目录清爽：运行产物迁移到 `artifacts/`（或同等级约定目录）。
4. 治理可持续：CI 门禁 + 文档目录政策 + 生命周期规则齐备。
5. 迁移安全可撤回：每个阶段都能独立回滚，不影响主线开发。

## 4. 执行原则（强约束）

1. 分阶段提交，不允许“一次性大改”。
2. 每阶段独立 PR（或独立 commit 组），每阶段可单独回滚。
3. 先修入口与索引，再做目录迁移，再做治理闭环。
4. 任何重命名必须提供映射表（`old -> new`）。
5. 不允许使用破坏性命令（如 `git reset --hard`）。
6. 对历史路径保持“过渡兼容”：先加迁移说明页，再删旧路径。

## 5. 分阶段任务清单

### 阶段 A：止损（1-2 天）

目标：让入口文档和基础门禁立即可用。

任务：
1. 修复 `README.md` 与 `docs/README.md` 的失效链接。
2. 增加或完善 `scripts/gate/check_docs_links.sh`。
3. 在 CI 中接入文档链接检查（建议 `ci.yml`）。
4. 明确“新增文档必须挂索引”的规则（写入文档政策草案）。

验收：
1. `README.md`、`docs/README.md` 链接检查通过。
2. CI 能执行文档链接检查并稳定通过。

回滚策略：
1. 若 CI 阻塞，先降级为“入口文档范围检查”，保留 `--all` 全量模式。
2. 回滚仅影响脚本与 CI 配置，不回滚业务代码。

---

### 阶段 B：结构归一（3-5 天）

目标：消除重复与命名混乱，降低长期维护成本。

任务：
1. 重复目录治理：
- 在 `docs/AI增强软件工程` 与 `docs/ai_collaboration` 之间选定 canonical 目录。
- 非 canonical 目录改为迁移说明页，注明下线日期（建议 2 周）。

2. 版本目录标准化：
- 统一为 `docs/releases/vX.Y.Z`。
- `OO/oo` 大小写归一为小写（`oo`）。
- `vX.Y` 目录保留兼容入口（README 跳转），不再新增内容。

3. 根目录产物迁移：
- CSV/PNG/XLSX/tarpaulin/html/tar 等迁移到 `artifacts/`。
- 更新 `.gitignore`，防止同类文件再次污染根目录。

验收：
1. 无双份维护目录（仅保留一个 canonical）。
2. 无新增 `vX.Y` 版本目录；`OO` 路径归零。
3. 根目录只保留工程入口文件与必要配置。

回滚策略：
1. 目录迁移采用“先复制/重定向、后删除”两步法。
2. 任何批量重命名都保留映射表：`docs/reports/doc-move-map-YYYY-MM-DD.md`。
3. 若出现外部链接断裂，优先恢复旧入口 README（跳转型），不直接回退全部迁移。

---

### 阶段 C：治理闭环（1 周）

目标：把治理固化到流程，避免反复返工。

任务：
1. 新增 `docs/DIRECTORY_POLICY.md`，定义：
- 命名规范
- 放置规则（按读者与生命周期）
- 索引规则
- 归档规则（active/archived）

2. PR 模板与门禁联动：
- PR 模板增加“文档路径与索引更新”核对项。
- CI 增加：
  - 失效链接检查
  - 未索引文档检查（至少覆盖 `docs` 下新增 markdown）
  - 重复文件检测（hash 级别）

3. 周期性治理指标：
- 每周输出以下指标并落档：
  - 失效链接数
  - 未索引文档数
  - 重复文件对数

验收：
1. 门禁稳定运行 1 周无误报阻塞。
2. 文档新增流程可被新人按政策独立执行。

回滚策略：
1. 对“高误报”门禁先降级为 warning，再修规则后恢复 blocking。
2. 政策文档改动可直接回滚，不影响仓库代码。

## 6. 输出要求（你必须产出）

1. 变更摘要：按阶段列出改动文件清单。
2. 迁移映射表：所有重命名/迁移路径。
3. 验证结果：执行过的命令与关键输出摘要。
4. 风险清单：未完成项、潜在破坏点、建议后续动作。
5. 回滚说明：每个阶段如何“一键或最小成本”回退。

## 7. 提交策略（建议）

1. Commit/PR-A：入口链接修复 + 文档链接门禁脚本 + CI 接入
2. Commit/PR-B：重复目录归一 + 版本命名标准化 + 产物迁移
3. Commit/PR-C：目录政策 + PR 模板 + 指标脚本与周报机制

每个 PR 必须包含：
1. 变更范围边界
2. 验证命令
3. 回滚步骤

## 8. 操作禁令

1. 不允许删除无法确认用途的历史文档原件（先迁移说明，再归档）。
2. 不允许一次性修改所有历史版本文档内容。
3. 不允许把治理脚本写成“仅本机可跑”。

## 9. 执行起始语句（直接使用）

请基于 `docs/report/2026-04-19-repo-structure-docs-audit.md` 执行 SQLRustGo 文档治理，严格按“阶段 A -> B -> C”推进。  
每阶段先给出“计划与影响面”，再实施并验证，最后输出“改动清单 + 验证结果 + 回滚方案”。  
若遇到高风险改动，先停在 checkpoint，等待确认后继续。  
目标是：完整、彻底、可回滚地完成文档治理闭环。
