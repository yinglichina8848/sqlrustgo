## 三、操作步骤

### 步骤1：创建功能分支并实现小功能（30分钟）

#### 1.1 创建新功能分支

```bash
# 确保develop分支最新
git checkout develop/v2.6.0
git pull origin develop/v2.6.0

# 创建新功能分支
git checkout -b feature/add-uppercase-function
```

#### 1.2 实现一个小功能

编辑 `crates/common/src/lib.rs` 添加辅助函数：

```rust
/// 将字符串转换为大写
pub fn uppercase(s: &str) -> String {
    s.to_uppercase()
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_uppercase() {
        assert_eq!(uppercase("hello"), "HELLO");
        assert_eq!(uppercase("World"), "WORLD");
    }
}
```

#### 1.3 提交代码

```bash
git add .
git commit -m "feat: add uppercase helper function"
```

#### ✅ 检查点1：记录分支和提交

---

### 步骤2：创建Pull Request（20分钟）

#### 2.1 推送分支

```bash
git push origin feature/add-uppercase-function
```

#### 2.2 在GitHub上创建PR

1. 访问分支页面
2. 点击 "Compare & pull request"
3. 填写PR信息：

```markdown
## 变更类型
- [x] 新功能

## 变更说明
添加了一个大写转换辅助函数，用于字符串处理。

## 测试说明
已添加单元测试，覆盖了基本功能。

## 相关Issue
无
```

#### ✅ 检查点2：保存PR链接

---

### 步骤3：代码审核实践（30分钟）

#### 3.1 作为审核者进行审核

使用另一个GitHub账号（或让同学）进行审核：

1. 登录审核账号
2. 找到PR
3. 进行代码审核

#### 3.2 审核检查清单

| 检查项 | 状态 | 备注 |
|--------|------|------|
| 代码是否实现了预期功能 | ⬜ | |
| 代码是否清晰易读 | ⬜ | |
| 是否有足够的测试 | ⬜ | |
| 是否更新了文档 | ⬜ | |
| 是否符合编码规范 | ⬜ | |
| 是否有安全风险 | ⬜ | |

#### 3.3 审核意见类型

- **必需修改（MUST）**：必须修改的问题
- **建议修改（SUGGEST）**：建议但非强制的改进
- **问题讨论（QUESTION）**：需要讨论的问题

#### 3.4 响应审核意见

```bash
# 根据意见修改代码
git add .
git commit -m "fix: address review comments"
git push origin feature/add-uppercase-function
```

#### ✅ 检查点3：记录审核过程

## 五、思考与反思

### 🎯 本周能力阶段：Context Engineering → 过渡到Harness Engineering

**本周重点**：开始尝试定义规则和约束

### 问题1：优化与改进
- 在进行代码审核时，我是否开始学会定义"好代码的标准"？
- 我是否能向AI清晰表达"什么样的代码是不合格的"？

### 问题2：AI不可替代的能力
- AI能判断"这段代码是否符合项目规范"吗？
- 人类的哪些经验（如"这段代码以后会很难维护"）是AI难以理解的？

### 问题3：自我提升
- 我是否开始尝试给AI设定一些"边界"或"规则"？
- 当AI违反了我设定的规则时，我如何处理？
