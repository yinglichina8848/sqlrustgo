# TPC-H 性能分析报告：SQLRustGo vs SQLite

**日期**: 2026-04-03  
**分支**: develop/v2.1.0  
**作者**: SQLRustGo Team  

---

## 📋 执行摘要

本次 TPC-H 基准测试揭示了 SQLRustGo 在大数据量场景下的**严重性能问题**。

### 关键发现

| 数据规模 | SQLRustGo vs SQLite | 结论 |
|----------|---------------------|------|
| **SF=0.1** (600K 行) | **64x faster** | ✅ 优势明显 |
| **SF=0.3** (1.8M 行) | **~30x slower** | ❌ 严重问题 |
| **SF=1** (6M 行) | 无法测试 (OOM) | ❌ 受限 |

**核心问题**: SQLRustGo 的内存存储 (MemoryStorage) 在数据量增长时性能急剧下降，无法有效处理大规模数据集。

## 🔍 问题分析

### 1. 根本原因：内存存储架构问题

SQLRustGo 使用纯内存存储 (MemoryStorage)，在数据量增长时面临：

#### a) 内存分配与访问效率

| 问题 | 描述 |
|------|------|
| **无索引支持** | 全表扫描，O(n) 复杂度 |
| **内存碎片** | 动态分配导致缓存效率低 |
| **GC 压力** | Rust 堆分配触发 GC 暂停 |

#### b) 查询执行引擎问题

```rust
// 当前实现：全表扫描 + 过滤
for row in &self.tables[table_name] {
    if evaluate_where(&row, condition)? {
        result.push(project_row(&row, select_cols)?);
    }
}
```

**问题**:
1. 无索引优化
2. 无查询计划优化
3. 无向量化执行

### 2. SQLite 的优化技术

| 技术 | SQLRustGo | SQLite |
|------|-----------|--------|
| **B-Tree 索引** | ❌ 无 | ✅ 有 |
| **查询优化器** | ❌ 无 | ✅ 有 |
| **向量化执行** | ❌ 无 | ✅ 有 |
| **WAL 模式** | N/A | ✅ 有 |
| **内存映射** | ❌ 无 | ✅ 有 |

### 3. 数据规模影响

| 规模 | lineitem 行数 | 预估内存 | SQLRustGo 问题 |
|------|---------------|----------|----------------|
| SF=0.1 | 600K | ~480 MB | 轻微 |
| SF=0.3 | 1.8M | ~1.4 GB | **严重** |
| SF=0.5 | 3.0M | ~2.4 GB | **严重** |
| SF=1 | 6.0M | ~4.8 GB | **OOM** |

## 💡 改进建议

### 短期优化 (v2.2.0)

1. **添加基础索引支持**
   - 主键索引
   - 外键索引
   - 单列索引

2. **查询计划优化**
   - 谓词下推
   - 列裁剪
   - 早期过滤

3. **内存管理优化**
   - 对象池
   - 预分配策略
   - 减少堆分配

### 长期优化 (v3.0.0)

1. **磁盘存储引擎**
   - WAL 支持
   - B-Tree 页面管理
   - 内存缓存层

2. **向量化执行引擎**
   - SIMD 指令
   - 列式存储
   - Batch 处理

3. **成本模型优化器**
   - 基于规则的优化
   - 统计信息收集
   - 索引选择

## 📝 结论

SQLRustGo 在小数据量场景 (SF=0.1) 下展现出**显著性能优势** (64x faster)，但随着数据量增长，性能优势迅速消失并在 SF=0.3 时完全逆转。

**核心问题**: 纯内存存储架构无法有效扩展到中等规模数据集。

**建议**:
1. 优先实现磁盘存储引擎
2. 添加基础索引支持
3. 建立性能回归测试，防止优化后退
