# 灾难恢复演练剧本

**版本**: v3.1.0  
**日期**: 2026-05-14  
**状态**: Beta 完成

---

## 1. 概述

### 1.1 目的

本文档提供 SQLRustGo 灾难恢复的标准操作流程，确保在各种故障场景下能够快速、一致地恢复正常运营。

### 1.2 范围

- 崩溃恢复
- 介质故障恢复
- 人为错误恢复
- 灾难恢复

### 1.3 关键指标

| 指标 | 目标 | 说明 |
|------|------|------|
| RTO | < 5 分钟 | Recovery Time Objective |
| RPO | < 1 分钟 | Recovery Point Objective |
| 恢复成功率 | 100% | 所有测试场景 |

## 3. 故障分类与响应

### 3.1 故障分类

| 级别 | 类型 | 响应时间 | 处理方式 |
|------|------|----------|----------|
| P0 | 崩溃无法启动 | 15 分钟 | 紧急恢复 |

## 4. 恢复场景

### 场景 1: 崩溃恢复

#### 触发条件
- 数据库进程异常退出
- 系统重启
- 断电后重启

#### 恢复步骤

```
┌─────────────────────────────────────────────────────────────┐
│ 步骤 1: 评估状态                                              │
├─────────────────────────────────────────────────────────────┤
│ 1. 检查数据库状态                                              │
│    $ systemctl status sqlrustgo                               │
│                                                              │
│ 2. 检查错误日志                                               │
│    $ tail -100 /var/log/sqlrustgo/error.log                  │
│                                                              │
│ 3. 确定崩溃原因                                               │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 2: 启动恢复                                              │
├─────────────────────────────────────────────────────────────┤
│ 1. 停止服务 (如果仍在运行)                                     │
│    $ systemctl stop sqlrustgo                                │
│                                                              │
│ 2. 备份当前状态                                               │
│    $ cp -r /var/lib/sqlrustgo /backup/pre-recovery-$(date) │
│                                                              │
│ 3. 启动恢复模式                                               │
│    $ sqlrustgo --recover --db-path=/var/lib/sqlrustgo       │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 3: 验证恢复                                              │
├─────────────────────────────────────────────────────────────┤
│ 1. 检查 WAL 恢复日志                                           │
│    $ grep "WAL recovery" /var/log/sqlrustgo/recovery.log   │
│                                                              │
│ 2. 验证数据一致性                                             │
│    $ sqlrustgo -e "SELECT count(*) FROM pg_catalog.pg_class"│
│                                                              │
│ 3. 验证审计链完整性                                           │
│    $ sqlrustgo -e "SELECT audit_verify_chain()"             │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 4: 恢复服务                                              │
├─────────────────────────────────────────────────────────────┤
│ 1. 启动服务                                                   │
│    $ systemctl start sqlrustgo                               │
│                                                              │
│ 2. 检查健康状态                                               │
│    $ curl http://localhost:5432/health                      │
│                                                              │
│ 3. 通知相关方                                                 │
│    - 发送恢复完成通知                                         │
│    - 更新状态页面                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 恢复后检查清单

- [ ] 数据库进程运行中
- [ ] 监听端口可访问
- [ ] 查询返回正确结果
- [ ] 审计链验证通过
- [ ] 无错误日志
- [ ] 监控指标正常

### 场景 2: WAL 文件损坏恢复

#### 触发条件
- 磁盘写入错误
- 文件系统损坏
- 人为误操作

#### 恢复步骤

```
┌─────────────────────────────────────────────────────────────┐
│ 步骤 1: 确认 WAL 损坏                                        │
├─────────────────────────────────────────────────────────────┤
│ 1. 检查 WAL 状态                                              │
│    $ sqlrustgo -e "SHOW wal_status"                         │
│                                                              │
│    预期输出: "WAL_CORRUPT" 或 "WAL_MISSING"                 │
│                                                              │
│ 2. 检查 WAL 文件                                              │
│    $ ls -la /var/lib/sqlrustgo/wal/                        │
│                                                              │
│ 3. 检查校验和                                                  │
│    $ sha256sum /var/lib/sqlrustgo/wal/*.log                 │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 2: 从 Checkpoint 恢复                                    │
├─────────────────────────────────────────────────────────────┤
│ 1. 识别最近的正常 Checkpoint                                  │
│    $ ls -lt /var/lib/sqlrustgo/data/*.ckpt | head -1        │
│                                                              │
│ 2. 从 Checkpoint 恢复                                        │
│    $ sqlrustgo --recover-from-checkpoint \                   │
│        --checkpoint=/var/lib/sqlrustgo/data/latest.ckpt    │
│                                                              │
│ 3. 检查恢复状态                                               │
│    $ sqlrustgo -e "SELECT recovery_status()"                │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 3: 验证数据完整性                                        │
├─────────────────────────────────────────────────────────────┤
│ 1. 检查表行数                                                 │
│    $ for tbl in $(sqlrustgo -e "SELECT tablename FROM pg_tables"); do│
│        echo "$tbl: $(sqlrustgo -e "SELECT count(*) FROM $tbl")"    │
│      done                                                     │
│                                                              │
│ 2. 验证引用完整性                                             │
│    $ sqlrustgo -e "SELECT verify_referential_integrity()"    │
│                                                              │
│ 3. 验证应用审计                                               │
│    $ sqlrustgo -e "SELECT audit_verify_chain()"             │
└─────────────────────────────────────────────────────────────┘
```

#### 潜在数据丢失

| 场景 | 可能丢失数据 |
|------|--------------|
| Checkpoint 后 WAL 全部丢失 | 0 |
| Checkpoint 后部分 WAL | 最后 Checkpoint 后的未提交事务 |
| Checkpoint 损坏 | 最近两个 Checkpoint 之间的数据 |

### 场景 3: 审计链断裂恢复

#### 触发条件
- 审计日志被篡改
- 哈希链验证失败
- 签名验证失败

#### 恢复步骤

```
┌─────────────────────────────────────────────────────────────┐
│ 步骤 1: 确认审计链断裂                                       │
├─────────────────────────────────────────────────────────────┤
│ 1. 运行审计验证                                               │
│    $ sqlrustgo -e "SELECT audit_verify_chain()"            │
│                                                              │
│    预期: {"is_valid": false, "first_invalid_event": N}     │
│                                                              │
│ 2. 检查错误日志                                               │
│    $ grep "AUDIT" /var/log/sqlrustgo/error.log | tail -20 │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 2: 评估损害范围                                         │
├─────────────────────────────────────────────────────────────┤
│ 1. 确定断裂位置                                               │
│    $ sqlrustgo -e "SELECT audit_verify_chain_breaking_point()"│
│                                                              │
│ 2. 检查是否有备份                                             │
│    $ ls /backup/audit/*.log.gz | tail -5                    │
│                                                              │
│ 3. 评估篡改时间范围                                           │
│    根据 first_invalid_event 的时间戳判断                      │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 3: 恢复策略                                              │
├─────────────────────────────────────────────────────────────┤
│ 策略 A: 从备份恢复 (推荐)                                     │
│    1. 停止审计写入                                           │
│    2. 恢复最近的完整备份                                     │
│    3. 验证备份的审计链                                       │
│    4. 从备份时间点重新开始审计                               │
│                                                              │
│ 策略 B: 标记断裂点继续                                        │
│    1. 创建断裂事件快照                                       │
│    2. 标记断裂事件 (audit_chain_break)                      │
│    3. 从断点继续记录新审计事件                              │
│    4. 在文档中记录断裂详情                                  │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 4: 通知与报告                                           │
├─────────────────────────────────────────────────────────────┤
│ 1. 生成审计断裂报告                                           │
│    $ sqlrustgo -e "SELECT audit_generate_break_report()"   │
│                                                              │
│ 2. 通知安全团队                                               │
│                                                              │
│ 3. 提交安全事件报告                                           │
│                                                              │
│ 4. 更新风险评估                                               │
└─────────────────────────────────────────────────────────────┘
```

### 场景 4: 完整灾难恢复

#### 触发条件
- 数据中心级故障
- 硬件完全损坏
- 勒索软件攻击

#### 前提条件
- 异地备份可用
- 恢复环境准备就绪
- RTO 计划激活

#### 恢复步骤

```
┌─────────────────────────────────────────────────────────────┐
│ 阶段 1: 灾难确认 (0-15 分钟)                                │
├─────────────────────────────────────────────────────────────┤
│ 1. 确认灾难状态                                              │
│    - 检查硬件状态                                             │
│    - 确认无法原地恢复                                         │
│                                                              │
│ 2. 激活灾难恢复计划                                          │
│    - 通知应急响应团队                                         │
│    - 激活异地恢复环境                                         │
│                                                              │
│ 3. 评估 RTO                                                 │
│    - 确定最新可恢复点                                         │
│    - 通知利益相关方 RTO 影响                                 │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段 2: 环境准备 (15-60 分钟)                               │
├─────────────────────────────────────────────────────────────┤
│ 1. 准备恢复服务器                                            │
│    - 安装操作系统                                             │
│    - 配置网络                                                 │
│    - 安装 SQLRustGo                                           │
│                                                              │
│ 2. 验证备份可用性                                            │
│    - 检查备份完整性                                           │
│    - 验证 WAL 备份                                           │
│    - 确认审计备份                                             │
│                                                              │
│ 3. 准备恢复脚本                                              │
│    - 编辑 recovery脚本                                       │
│    - 测试网络连接                                             │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段 3: 数据恢复 (60-240 分钟)                              │
├─────────────────────────────────────────────────────────────┤
│ 1. 恢复最新数据备份                                           │
│    $ sqlrustgo --restore \                                   │
│        --backup=/backup/latest/data \                        │
│        --target=/var/lib/sqlrustgo                          │
│                                                              │
│ 2. 应用 WAL 备份                                             │
│    $ sqlrustgo --replay-wal \                               │
│        --wal-dir=/var/lib/sqlrustgo/wal \                   │
│        --backup=/backup/latest/wal                           │
│                                                              │
│ 3. 恢复审计备份                                              │
│    $ sqlrustgo --restore-audit \                            │
│        --backup=/backup/latest/audit                         │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段 4: 验证与上线 (240-300 分钟)                            │
├─────────────────────────────────────────────────────────────┤
│ 1. 数据完整性验证                                             │
│    - 验证所有表行数                                           │
│    - 验证引用完整性                                           │
│    - 验证审计链                                               │
│                                                              │
│ 2. 功能测试                                                  │
│    - 运行 OQ 测试套件                                         │
│    - 验证关键业务查询                                         │
│                                                              │
│ 3. 切换流量                                                  │
│    - 更新 DNS/负载均衡器                                     │
│    - 监控错误率                                               │
│    - 通知用户                                                 │
└─────────────────────────────────────────────────────────────┘
```
