## 1. 目的与范围

本规范是 SQLRustGo GMP (Greybrand Manufacturing Process) 的总纲，定义了可信可证明软件系统的核心原则、流程框架和质量标准。

**适用范围**:
- SQLRustGo 数据库内核开发
- MIS 类型管理信息系统开发
- 需要形式化验证的嵌入式/关键系统开发

**核心目标**: 确保每一行代码、每一个变更、每一次发布都有完整的证据链支撑，达到"可信、可证明"的质量标准。

## 2. 核心原则

### 2.1 可追溯原则 (Traceability)

```
[Requirement] → [Design] → [Implementation] → [Test] → [Release] → [Operation]
     ↑_______________↑_______________↑_______________↑_______________↑
              证据链完整闭合
```

**强制要求**:
- 每个需求必须有唯一标识符 (REQ-XXXX)
- 每个实现必须关联至少一个需求标识符
- 每个测试用例必须映射到需求标识符
- 每次发布必须包含完整的证据包

### 2.2 可证实原则 (Provability)

**证据等级定义**:

| 等级 | 标识 | 证据类型 | 适用场景 |
|------|------|----------|----------|
| L1 | FORMAL | 形式化证明 | 关键安全算法、协议 |
| L2 | TESTED | 自动化测试 + 覆盖率达到 95%+ | 核心业务逻辑 |
| L3 | VERIFIED | 人工审查 + 签名确认 | 一般业务逻辑 |
| L4 | REVIEWED | 同行评审 | 辅助功能 |

**证据要求矩阵**:

| 系统类型 | 最低证据等级 | 覆盖要求 |
|----------|-------------|----------|
| 数据库内核 | L1 或 L2 | 95% 分支覆盖 |
| MIS 系统 | L2 或 L3 | 85% 分支覆盖 |
| 形式化验证系统 | L1 | 100% 路径覆盖 |

### 2.3 变更控制原则 (Change Control)

**禁止的无控变更**:
```
✗ 直接在 main/master 分支修改
✗ 绕过 CI/CD 直接部署
✗ 手动修改生产环境配置
✗ 未经测试的 hotfix 直接合并
```

**正确的变更流程**:
```
feature/xxx → PR → Review → CI Pass → Merge → CD Deploy → Sign-off
```

### 4.2 文档版本控制

**版本格式**: `v{major}.{minor}.{patch}`
- major: 不兼容的重大变更
- minor: 向后兼容的功能增加
- patch: 向后兼容的问题修复

**版本标记**: 每个文档头部必须包含版本、状态、创建日期、最后更新字段。

## 5. 证据体系

### 5.1 证据类型定义

| 证据类型 | 标识 | 必需性 | 保留期限 |
|----------|------|--------|----------|
| 需求追踪证据 | REQ-EVD | 必须 | 永久 |
| 设计审查证据 | DES-EVD | 必须 | 永久 |
| 代码审查证据 | COD-EVD | 必须 | 永久 |
| 测试执行证据 | TST-EVD | 必须 | 永久 |
| 性能基准证据 | PRF-EVD | 条件必需 | 5年 |
| 安全扫描证据 | SEC-EVD | 必须 | 5年 |
| 发布批准证据 | REL-EVD | 必须 | 永久 |

### 5.2 证据包格式

每个发布必须附带完整的证据包，结构如下:

```

├── 00_MANIFEST.json          # 证据包清单
├── 01_REQUIREMENTS.json      # 需求追踪矩阵
├── 02_DESIGN_APPROVAL.json   # 设计审批记录
├── 03_CODE_REVIEW.json       # 代码审查记录
├── 04_TEST_RESULTS.json      # 测试执行结果
├── 05_COVERAGE_REPORT.json   # 覆盖率报告
├── 06_SECURITY_SCAN.json     # 安全扫描报告
├── 07_PERFORMANCE.json       # 性能基准数据
└── 08_SIGNATURES.json        # 发布签名
```

## 6. 检查清单

### 6.1 代码层面禁止模式 (PROHIBITED)

以下模式在代码中严格禁止，CI 必须拦截:

| 禁止模式 | 正确替代 | 检测工具 |
|----------|----------|----------|
| `unsafe` 块无注释 | 使用安全抽象层 | cargo-geiger, manual |
| 硬编码凭证 | 环境变量/密钥管理 | gitleaks, secret-scan |
| 未处理的 Result/Option | 错误传播或处理 | clippy::expect_used |
| SQL 拼接字符串 | 参数化查询 | sql-lint, 人工审查 |
| 共享可变状态 | 消息传递/锁 | clippy::await_holding_lock |
| 魔法数字 | 命名常量 | clippy::unwrap_used |
| 异常吞没 | 错误日志+传播 | error-chain 审查 |

### 6.2 提交信息规范

**格式**: `<type>(<scope>): <subject> [EVIDENCE: <id>]`

**type 类型**:
- feat: 新功能
- fix: 修复
- docs: 文档
- style: 格式
- refactor: 重构
- test: 测试
- chore: 维护
- proof: 形式化证明

**示例**:
```
feat(auth): add JWT validation with formal proof [EVIDENCE: PRF-2026-001]
fix(query): resolve hash join memory leak [EVIDENCE: TST-2026-042]
```

## 7. 合规性要求

### 7.1 必须满足的条件

1. **所有 PR 必须关联 REQ**: 每个 PR 必须包含 `Closes #REQ-XXXX` 或 `Related to #REQ-XXXX`
2. **测试覆盖率门槛**: 
   - 新代码: ≥ 90% 分支覆盖
   - 修改代码: ≥ 85% 分支覆盖
   - 总代码库: ≥ 80% 分支覆盖
3. **静态分析清零**: 禁止存在 HIGH/CRITICAL 级别警告
4. **安全扫描通过**: 无新引入的 CVE 漏洞

### 7.2 门控检查 (Gate Checks)

```
PR 创建 ──▶ 门控 1 ──▶ 门控 2 ──▶ 门控 3 ──▶ 合并
              │            │            │
              ▼            ▼            ▼
          单元测试      覆盖率      安全扫描
          格式检查      检查        依赖审计
              │            │            │
              ▼            ▼            ▼
           通过/失败   达标/不达标   无漏洞/有漏洞
```
