## 1. 核心理念：人类建立流程，AI 协作执行

### 1.1 传统模式 vs 协作模式

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        传统模式：人类控制一切                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   人类 ──→ 分配任务 ──→ 等待完成 ──→ 检查结果 ──→ 修正问题                   │
│                                                                              │
│   问题：人类需要持续干预，无法规模化                                          │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│                        协作模式：人类建立流程，AI 自动执行                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   人类 ──→ 建立流程规范 ──→ AI 自动执行 ──→ 定期检查 ──→ 调整流程            │
│      │                                                                              │
│      │         ┌──────────────────────────────────────────┐                │
│      │         │              流程自动化                   │                │
│      │         │  • Issue 发布任务                         │                │
│      │         │  • Agent 自动领取并执行                   │                │
│      │         │  • PR 自动提交                           │                │
│      │         │  • 交叉审核后自动合并                     │                │
│      │         └──────────────────────────────────────────┘                │
│      │                                                                       │
│   关键：人类只需建立规则，AI 按规则自动运行                                    │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 协作三要素

| 要素 | 说明 | 人类职责 | AI 职责 |
|:-----|:-----|:---------|:---------|
| **流程** | 工作规则和步骤 | 设计和建立 | 遵守执行 |
| **沟通** | 信息同步机制 | 设立渠道 | 自动同步 |
| **审核** | 质量保证机制 | 设定标准 | 交叉审核 |

### 1.3 核心原则

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          AI 协作开发核心原则                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   1. 流程先行                                                                │
│      • 先建立规范，再开始开发                                                │
│      • 规范要清晰、可执行                                                    │
│      • AI 按规范自动运行                                                     │
│                                                                              │
│   2. 职责分离                                                                │
│      • 人类：决策、审核、调整流程                                            │
│      • AI：执行、汇报、交叉审核                                              │
│                                                                              │
│   3. 自动同步                                                                │
│      • 重要信息发布到 Issue                                                  │
│      • 进展通过群聊同步                                                      │
│      • 定期刷新获取最新状态                                                  │
│                                                                              │
│   4. 交叉审核                                                                │
│      • Agent A 的 PR 由 Agent B 审核                                        │
│      • 审核通过后自动合并                                                    │
│      • 人类只在必要时干预                                                    │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. OpenClaw：多 Agent 协作平台

### 2.1 什么是 OpenClaw？

OpenClaw 是一个多 Agent 协作平台，支持：

- **多模型接入**：Claude、GPT、DeepSeek 等
- **飞书集成**：通过群聊进行协作
- **身份管理**：每个 Agent 有独立身份
- **技能系统**：可扩展的能力模块

### 2.2 安装 OpenClaw

```bash
# 克隆项目
git clone https://github.com/clawhub/openclaw.git
cd openclaw

# 安装依赖
pip install -r requirements.txt

# 或使用 Docker
docker build -t openclaw .
docker run -d -p 8080:8080 openclaw
```

### 2.3 配置 Agent 身份

在 OpenClaw 中，每个 Agent 需要有独立的身份标识：

```yaml
# openclaw_config.yaml
agents:
  - name: 高小原
    id: gaoxiaoyuan-macmini
    email: gaoxiaoyuan-macmini@users.noreply.github.com
    role: 开发者
    platform: macmini
    
  - name: 高小药
    id: gaoxiaoyao-macbookpro
    email: gaoxiaoyao-macbookpro@users.noreply.github.com
    role: 开发者
    platform: macbookpro
    
  - name: Codex
    id: codex-controller
    email: codex@users.noreply.github.com
    role: 总控
    platform: server
```

### 2.4 接入飞书

#### 步骤 1：创建飞书应用

1. 访问 [飞书开放平台](https://open.feishu.cn/)
2. 创建企业自建应用
3. 获取 App ID 和 App Secret

#### 步骤 2：配置权限

在飞书应用管理中添加以下权限：

| 权限 | 说明 |
|:-----|:-----|
| `im:chat` | 获取群组信息 |
| `im:chat:readonly` | 读取群消息 |
| `im:message` | 发送消息 |
| `im:message:send_as_bot` | 以应用身份发消息 |

#### 步骤 3：配置 OpenClaw

```yaml
# feishu_config.yaml
feishu:
  app_id: cli_xxxxxxxxxxxx
  app_secret: xxxxxxxxxxxxxxxx
  
  # 群聊配置
  groups:
    - name: 龙虾开发群
      chat_id: oc_271b7f6065714b2be67a5debe8697cd3
      
  # 消息处理
  message_handler:
    # 只响应 @ 消息
    mention_only: true
    # 忽略自己的消息
    ignore_self: true
```

#### 步骤 4：启动服务

```bash
# 启动 OpenClaw 服务
python main.py --config openclaw_config.yaml

# 验证飞书连接
curl http://localhost:8080/health
```

### 2.5 Agent 身份切换

每个 Agent 在执行任务时需要切换到对应身份：

```bash
# 切换到高小原身份
export AGENT_ID=gaoxiaoyuan-macmini
export AGENT_NAME=高小原

# 切换到高小药身份
export AGENT_ID=gaoxiaoyao-macbookpro
export AGENT_NAME=高小药
```

## 3. 飞书群聊：建立沟通机制

### 3.1 创建协作群聊

#### 步骤 1：创建群聊

1. 在飞书中创建新群聊
2. 邀请所有 Agent 加入
3. 邀请人类协调者（李哥）

#### 步骤 2：配置群聊机器人

将 OpenClaw 创建的飞书应用添加到群聊：

1. 群设置 → 添加机器人 → 选择应用
2. 确保机器人有消息发送权限

### 3.2 沟通规范

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          飞书群聊沟通规范                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   规则 1：@ 指定接收者                                                       │
│                                                                              │
│   ✅ 正确：                                                                  │
│   @高小原 请你负责解析器重构任务                                             │
│   @高小药 执行器优化任务分配给你                                             │
│                                                                              │
│   ❌ 错误：                                                                  │
│   今天天气不错（没有 @，Agent 不应回复）                                     │
│   大家加油！（群发消息，不是任务要求）                                        │
│                                                                              │
│   ─────────────────────────────────────────────────────────────────────────  │
│                                                                              │
│   规则 2：重要信息发 Issue                                                   │
│                                                                              │
│   • 任务分配 → 发布到 Issue                                                  │
│   • 进展汇报 → 评论到 Issue                                                  │
│   • 问题讨论 → Issue + 群聊同步                                              │
│                                                                              │
│   ─────────────────────────────────────────────────────────────────────────  │
│                                                                              │
│   规则 3：定期刷新状态                                                       │
│                                                                              │
│   • 每 30 分钟刷新 Issue 列表                                                │
│   • 有新任务自动领取执行                                                     │
│   • 完成后自动汇报到群聊                                                     │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.3 消息风暴预防

**问题**：多个 Agent 同时响应同一条消息，导致刷屏

**解决方案**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          消息风暴预防机制                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   1. 分析消息受众                                                            │
│      • 消息是说给谁的？                                                      │
│      • 有没有 @ 我？                                                        │
│      • 是任务还是闲聊？                                                      │
│                                                                              │
│   2. 判断是否需要回复                                                        │
│      • @ 了我 → 必须回复                                                     │
│      • @ 了别人 → 不回复                                                     │
│      • 没有 @ → 不回复（可能是闲聊）                                         │
│                                                                              │
│   3. 回复时避免 @ 所有人                                                     │
│      • 只 @ 相关人员                                                         │
│      • 汇报工作时不需要 @                                                    │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 4. GitHub 多身份配置

### 4.1 为什么需要多身份？

| 原因 | 说明 |
|:-----|:-----|
| **身份隔离** | 区分不同 Agent 的提交 |
| **权限控制** | 不同身份有不同的分支权限 |
| **审核机制** | Agent A 的 PR 由 Agent B 审核 |
| **审计追踪** | 清晰记录每个变更的来源 |

### 4.2 身份配置方案

#### 方案：工作目录隔离

```
workspace/
   gaoxiaoyuan/           # 高小原身份
       sqlrustgo/
       sqlrg-testing/
   gaoxiaoyao/            # 高小药身份
       sqlrustgo/
       sqlrg-testing/
```

#### SSH 配置

```bash
# 生成 SSH 密钥对
ssh-keygen -t ed25519 -C "gaoxiaoyuan@users.noreply.github.com" -f ~/.ssh/id_ed25519_yuan
ssh-keygen -t ed25519 -C "gaoxiaoyao@users.noreply.github.com" -f ~/.ssh/id_ed25519_yao

# 配置 SSH
cat >> ~/.ssh/config << 'EOF'
Host github-yuan
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_yuan

Host github-yao
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_yao
EOF
```

#### 目录设置

```bash
# 高小原工作目录
cd ~/workspace/gaoxiaoyuan/sqlrustgo
git remote set-url origin git@github-yuan:minzuuniversity/sqlrustgo.git
git config user.name "gaoxiaoyuan-macmini"
git config user.email "gaoxiaoyuan-macmini@users.noreply.github.com"
export GH_CONFIG_DIR=~/workspace/gaoxiaoyuan/.gh

# 高小药工作目录
cd ~/workspace/gaoxiaoyao/sqlrustgo
git remote set-url origin git@github-yao:minzuuniversity/sqlrustgo.git
git config user.name "gaoxiaoyao-macbookpro"
git config user.email "gaoxiaoyao-macbookpro@users.noreply.github.com"
export GH_CONFIG_DIR=~/workspace/gaoxiaoyao/.gh
```

### 4.3 身份切换脚本

```bash
#!/bin/bash
# switch-agent.sh - 切换 Agent 身份

case "$1" in
    "yuan")
        cd ~/workspace/gaoxiaoyuan/sqlrustgo
        export AGENT_ID=gaoxiaoyuan-macmini
        export AGENT_NAME=高小原
        export GH_CONFIG_DIR=~/workspace/gaoxiaoyuan/.gh
        echo "切换到高小原身份 🌱"
        ;;
    "yao")
        cd ~/workspace/gaoxiaoyao/sqlrustgo
        export AGENT_ID=gaoxiaoyao-macbookpro
        export AGENT_NAME=高小药
        export GH_CONFIG_DIR=~/workspace/gaoxiaoyao/.gh
        echo "切换到高小药身份 🦞"
        ;;
    *)
        echo "用法: source switch-agent.sh [yuan|yao]"
        ;;
esac
```

### 5.3 自动化流程

#### Agent 自动领取任务

```bash
# 每 30 分钟执行一次
while true; do
    # 获取未分配的 Issue
    gh issue list --state open --json number,title,assignees \
        | jq '.[] | select(.assignees | length == 0)' \
        | while read issue; do
            # 领取任务
            issue_number=$(echo $issue | jq -r '.number')
            gh issue edit $issue_number --add-assignee $AGENT_ID
            
            # 评论确认
            gh issue comment $issue_number --body "🤖 $AGENT_NAME 已领取此任务"
        done
    
    sleep 1800  # 30 分钟
done
```

#### 任务完成后汇报

```bash
# PR 合并后自动汇报
gh issue comment $ISSUE_NUMBER --body "
## ✅ 任务完成

- **PR**: #$PR_NUMBER
- **审核人**: @$REVIEWER
- **合并时间**: $(date)

### 变更摘要
$CHANGES_SUMMARY
"
```

## 6. PR 交叉审核机制

### 6.1 交叉审核原则

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          PR 交叉审核原则                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   规则 1：不能审核自己的 PR                                                  │
│                                                                              │
│   高小原提交 PR → 高小药审核                                                 │
│   高小药提交 PR → 高小原审核                                                 │
│                                                                              │
│   ─────────────────────────────────────────────────────────────────────────  │
│                                                                              │
│   规则 2：审核通过后自动合并                                                 │
│                                                                              │
│   • 审核通过 → 自动 merge                                                    │
│   • 有问题 → 评论说明，等待修复                                              │
│                                                                              │
│   ─────────────────────────────────────────────────────────────────────────  │
│                                                                              │
│   规则 3：审核要点                                                           │
│                                                                              │
│   • 代码风格是否一致                                                         │
│   • 测试是否通过                                                             │
│   • 是否关联 Issue                                                           │
│   • 文档是否更新                                                             │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 6.2 审核流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          PR 审核流程                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   1. Agent A 提交 PR                                                        │
│      gh pr create --base alpha --head feature/xxx                           │
│                                                                              │
│   2. 在群聊中通知审核人                                                      │
│      "@高小药 PR #42 已提交，请审核"                                         │
│                                                                              │
│   3. Agent B 审核代码                                                        │
│      gh pr view 42                                                           │
│      gh pr diff 42                                                           │
│                                                                              │
│   4. 审核结果                                                                │
│      • 通过：gh pr review 42 --approve                                      │
│      • 有问题：gh pr review 42 --request-changes                            │
│                                                                              │
│   5. 通过后合并                                                              │
│      gh pr merge 42 --merge                                                  │
│                                                                              │
│   6. 汇报到 Issue                                                            │
│      gh issue comment $ISSUE --body "PR #42 已合并"                         │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 6.3 审核检查清单


## 8. 常见问题与解决方案

### Q1: Agent 之间如何避免消息风暴？

**A**: 遵循 @ 规则：
- 消息是说给谁的，就 @ 谁
- 没有 @ 的，不是任务/要求，可能是闲聊
- 不是对自己说的，不需要回复

### Q2: 如何处理 Agent 额度用尽？

**A**: 配置多模型切换：
- 主力模型：DeepSeek（便宜）
- 备用模型：Google Gemini（免费额度）
- 当一个模型额度用尽，自动切换到下一个

### Q3: PR 有冲突怎么办？

**A**: 
1. Agent rebase 到最新 alpha
2. 解决冲突后重新测试
3. 重新提交审核

### Q4: 如何保证代码质量？

**A**: 
1. 每个 PR 必须通过 CI 测试
2. 至少 1 个 Agent review
3. 遵循代码规范和最佳实践

### Q5: 人类需要多久干预一次？

**A**: 
- 正常情况：每天检查一次 Issue 状态
- 有问题：群聊中 @ 人类协调者
- 重大决策：人类最终确认
