# SQLRustGo 版本演化计划 v2.0

> **版本**: v2.0
> **更新日期**: 2026-03-13
> **依据**: 2026-03-13 讨论结论
> **核心原则**: 版本号语义化，1.x → 单机基础，2.x → 单机高性能（向量化 + CBO），3.x → 分布式数据库

---

## 一、版本号语义说明

| 主版本 | 技术范式 | 核心特征 | 代表系统类比 |
|--------|----------|----------|--------------|
| 1.x | 单机基础数据库 | 火山模型、行存、B+Tree、MVCC、SQL-92 | SQLite, PostgreSQL（早期） |
| 2.x | 单机高性能分析引擎 | 向量化执行、列存、CBO、并行执行 | DuckDB, ClickHouse（单机） |
| 3.x | 分布式数据库 | 分布式执行、分片、共识、分布式事务 | TiDB, CockroachDB |

**演进路径**: 1.x (可用) → 2.x (快) → 3.x (大)

---

## 三、版本演进核心逻辑

**v1.x（2026–2027）**：用两年时间打造一个功能完整，稳定可靠的单机数据库内核，涵盖 SQL 核心功能、存储、事务、索引、统计信息等，为后续性能革命打下坚实基础。

**v2.x（2028）**：集中突破单机性能，用一年时间实现向量化执行、列式存储、CBO 和并行执行，使 SQLRustGo 成为高性能分析型数据库（对标 DuckDB）。

**v3.x（2029）**：迈入分布式领域，用一年时间构建分布式执行、事务和优化能力，实现水平扩展，最终成为一个完整的分布式数据库系统（对标 TiDB/CockroachDB）。

---

## 四、与原有计划的衔接说明

- v1.3–v1.9 完全沿用已制定的详细任务（见 DEVELOPMENT_PLAN.md），仅微调了部分可观测性增强的顺序，确保与版本目标一致。

- 原 v3.0（向量化）和 v3.1（列存）被合并到新的 v2.x 系列，因为它们是单机性能优化的核心，不应与分布式混为一谈。

- 原 v3.2（分布式）作为新 v3.0 的起点，后续 v3.1、v3.2 按分布式深度递进。

这样调整后，版本号真正反映了技术代际，有利于项目宣传和社区认知，同时保留了所有已规划的功能点，开发进度不受影响。

## 五、版本节奏建议

| 系列 | 节奏 | 说明 |
|------|------|------|
| v1.x | 每 3 个月 | 保持稳定迭代 |
| v2.x | 每 3 个月 | 集中突破性能 |
| v3.x | 每 3 个月 | 快速构建分布式能力 |

**整体时间跨度**: 约 3.5 年（2026.03 – 2029.08），符合 solo 开发者或小团队的可持续节奏。
