## 一、核心原则

### 1.1 分支生命周期

**核心思想：分支不关，但"门禁"越来越严格**

开发分支 (`develop-x.x.x`) 从创建到 GA 发布一直存在，不会在 Alpha/Beta/RC 阶段关闭。但在不同阶段，允许的提交类型会逐渐收紧。

### 1.2 版本阶段流程

```
develop-x.x.x
    │
    ├── Draft (草稿)
    │   └── 允许：架构、目录、接口设计
    │
    ├── Alpha (内测)
    │   └── 允许：功能开发、新模块
    │
    ├── Beta (公测)
    │   └── 允许：Bug 修复、小功能、性能优化
    │
    ├── RC (发布候选)
    │   └── 允许：仅 Critical Bug 修复
    │
    └── GA (正式发布)
        ├── tag v1.x.x
        └── merge → main
```

## 二、分支结构

### 2.1 主分支架构

```
main
 │
 ├─ develop                    # 下一个版本开发
 │
 ├─ develop/v1.2.0              # 当前版本开发
 │    ├─ feature/*             # 功能分支
 │    ├─ fix/*                # 修复分支 (推荐)
 │    └─ bugfix/*             # 修复分支 (别名)
 │
 └─ develop-1.3.0              # 下一版本预研
```

### 2.2 分支命名规范

> ⚠️ **重要**: 所有 v1.2.0+ 的功能分支和修复分支必须包含版本号

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 功能 | `feature/v1.x.0-<功能名>` | `feature/v1.2.0-vector-execution` |
| 修复 | `fix/v1.x.0-<问题描述>` | `fix/v1.2.0-index-tests` |
| 文档 | `docs/v1.x.0-<文档类型>` | `docs/v1.2.0-release-notes` |

### 2.3 分支用途说明

| 分支 | 用途 | 保护级别 |
|------|------|----------|
| `main` | 稳定版本，只接受 release merge | 🔴 最高 |
| `develop` | 下一个版本开发 | 🟡 中等 |
| `develop-x.x.x` | 当前版本开发 | 🟡 中等 |
| `feature/*` | 新功能开发 | 🟢 低 |
| `fix/*` | Bug 修复 (推荐) | 🟢 低 |
| `bugfix/*` | Bug 修复 (别名) | 🟢 低 |
| `release/x.x` | 维护分支 | 🔴 高 |

## 三、版本阶段详解

### 3.1 Draft 阶段 (草稿)

**目标**: 完成架构设计和接口定义

| 项目 | 规则 |
|------|------|
| **允许提交** | ✅ 架构调整、目录重构、接口设计、模块拆分 |
| **禁止提交** | ❌ 无 |
| **门禁要求** | 编译通过 |
| **通过标准** | 架构评审通过、接口定义完整 |

**典型工作**:
- 目录结构调整
- 模块拆分
- Crate 重构
- 接口设计
- 200+ 文件级架构设计

### 3.2 Alpha 阶段 (内测)

**目标**: 完成功能开发

| 项目 | 规则 |
|------|------|
| **允许提交** | ✅ 新功能、新模块、优化器、执行器 |
| **禁止提交** | ❌ 架构变化、目录变化 |
| **门禁要求** | 测试通过率 ≥ 80% |
| **通过标准** | 所有计划功能完成、核心测试通过 |

**典型工作**:
- Cascades Optimizer
- Vector Execution
- Planner
- Storage Engine

### 3.3 Beta 阶段 (公测)

**目标**: 稳定功能、修复 Bug

| 项目 | 规则 |
|------|------|
| **允许提交** | ✅ Bug 修复、性能优化、测试补充、小功能 |
| **禁止提交** | ❌ 架构变化、API 变化、新模块 |
| **门禁要求** | 测试通过率 ≥ 95%、Clippy 零警告 |
| **通过标准** | 无 P0/P1 Bug、性能达标 |

**典型工作**:
- Bug 修复
- 性能优化
- 测试覆盖率提升
- 文档完善

### 3.4 RC 阶段 (发布候选)

**目标**: 最终验证

| 项目 | 规则 |
|------|------|
| **允许提交** | ✅ 仅 Critical Bug 修复 |
| **禁止提交** | ❌ 新代码、新模块、重构、功能变更 |
| **门禁要求** | 测试 100% 通过、CI 全绿 |
| **通过标准** | 无开放 Bug、门禁全部通过 |

**典型工作**:
- Critical Bug 修复
- 文档最终确认
- 发布准备

### 3.5 GA 阶段 (正式发布)

**目标**: 正式发布

| 项目 | 规则 |
|------|------|
| **允许提交** | ❌ 禁止修改 |
| **操作** | 创建 Tag、合并到 main |
| **门禁要求** | 所有门禁通过 |
| **后续** | 进入维护模式 |

**发布操作**:
```bash
# 1. 创建 Tag
git tag v1.2.0
git push origin v1.2.0

# 2. 合并到 main
git checkout main
git merge develop/v1.2.0
git push origin main

# 3. 创建维护分支
git checkout -b release/1.2
git push origin release/1.2
```

## 四、阶段转换规则

### 4.1 阶段转换检查清单

```
Draft → Alpha
├── ✅ 架构设计完成
├── ✅ 接口定义完整
├── ✅ 编译通过
└── ✅ 架构评审通过

Alpha → Beta
├── ✅ 所有功能开发完成
├── ✅ 测试通过率 ≥ 80%
├── ✅ 无 P0 Bug
└── ✅ 功能评审通过

Beta → RC
├── ✅ 测试通过率 ≥ 95%
├── ✅ Clippy 零警告
├── ✅ 无 P0/P1 Bug
└── ✅ 性能达标

RC → GA
├── ✅ 测试 100% 通过
├── ✅ CI 全绿
├── ✅ 无开放 Bug
├── ✅ 文档完整
└── ✅ 发布审批通过
```

### 4.2 阶段回退规则

| 场景 | 处理方式 |
|------|----------|
| RC 发现 Critical Bug | 回退到 Beta，修复后重新进入 RC |
| Beta 发现架构问题 | 回退到 Alpha，重新设计 |
| Alpha 发现设计缺陷 | 回退到 Draft，重新规划 |

## 五、分支保护规则

### 5.1 AI 协作开发特别规则

由于采用 AI 协作开发模式，必须设置严格的分支保护：

| 分支 | 保护规则 |
|------|----------|
| `main` | 禁止直接 push，必须通过 PR + 2人审核 |
| `develop-x.x.x` | 禁止直接 push，必须通过 PR + 1人审核 |
| `feature/*` | 允许直接 push，但需要 CI 通过 |

### 5.2 PR 合并规则

| 阶段 | 分支前缀 | 审核人数 | CI 要求 | 额外要求 |
|------|----------|----------|---------|----------|
| Draft | `feature/v1.x.0-*`, `fix/v1.x.0-*` | 1 | 编译通过 | 架构评审 |
| Alpha | `feature/v1.x.0-*`, `fix/v1.x.0-*` | 1 | 测试通过 | 功能测试 |
| Beta | `feature/v1.x.0-*`, `fix/v1.x.0-*` | 2 | Clippy + 测试 | 回归测试 |
| RC | `fix/v1.x.0-*` | 2 | 全部门禁 | 发布审批 |

## 六、维护分支管理

### 6.1 维护分支创建时机

GA 发布后，`develop-x.x.x` 转换为 `release/x.x` 维护分支：

```
develop/v1.2.0  →  release/1.2
```

### 6.2 维护分支规则

| 项目 | 规则 |
|------|------|
| **允许提交** | 仅 Bug 修复、安全补丁 |
| **版本号** | 1.2.1, 1.2.2, ... |
| **生命周期** | 直到 EOL 或下一个 minor 版本发布 |

## 七、常见问题

### Q1: 什么时候关闭 develop-x.x.x？

**答案**: 只有两种情况：

1. **版本发布**: GA 后转换为 `release/x.x` 维护分支
2. **版本取消**: 项目决定取消该版本开发

### Q2: 可以同时存在多个 develop 分支吗？

**答案**: 可以，推荐结构：

```
develop/v1.2.0  # 当前版本
develop-1.3.0  # 下一版本预研
develop        # 未来版本
```

### Q3: Hotfix 如何处理？

**答案**: 在 `release/x.x` 分支上修复，然后：

```bash
# 1. 在 release 分支修复
git checkout release/1.2
# ... fix ...
git commit -m "fix: critical bug"

# 2. 创建补丁版本
git tag v1.2.1

# 3. 合并回 main
git checkout main
git merge release/1.2
```
