# MVCC 实现详解

> Multi-Version Concurrency Control: 快照隔离、版本链、可见性判断

## 1. MVCC 概述

### 1.1 MVCC 原理

```
┌─────────────────────────────────────────────────────────────┐
│                    MVCC 原理                                 │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  传统锁 (Pessimistic):                                     │
│  ─────────────────────────────────────────────────────────  │
│  T1: SELECT → 锁定行 → T2: 等待 → T1: 释放 → T2: 读取   │
│                                                              │
│  MVCC (Optimistic):                                        │
│  ─────────────────────────────────────────────────────────  │
│  T1: 读取 v1 (快照)                                       │
│  T2: 读取 v1 (另一快照) ← 并发                           │
│  T1: 写入 v2                                              │
│  T2: 读取 v1 (不变)                                       │
│  T1: COMMIT                                              │
│  T2: COMMIT                                              │
│                                                              │
│  优点: 读不阻塞写，写不阻塞读                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心概念

```
┌─────────────────────────────────────────────────────────────┐
│                    MVCC 核心概念                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────┐  ┌─────────────────┐  ┌────────────┐ │
│  │   事务 ID       │  │   时间戳        │  │   快照     │ │
│  │                 │  │                 │  │            │ │
│  │ TxId: u64      │  │ begin_ts       │  │ Snapshot  │ │
│  │ 全局递增        │  │ commit_ts     │  │ 特定时刻   │ │
│  │                 │  │                 │  │ 数据视图   │ │
│  └─────────────────┘  └─────────────────┘  └────────────┘ │
│                                                              │
│  ┌─────────────────┐  ┌─────────────────┐                  │
│  │   版本链        │  │   可见性判断    │                  │
│  │                 │  │                 │                  │
│  │ v3 → v2 → v1  │  │ begin < cts    │                  │
│  │ 逆序链表       │  │ 快照时间戳内   │                  │
│  │                 │  │ 事务已提交     │                  │
│  └─────────────────┘  └─────────────────┘                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 2. MVCC 数据结构

### 2.1 版本记录

```rust
pub struct VersionRecord {
    pub tx_id: TxId,           // 事务 ID
    pub begin_ts: u64,         // 事务开始时间戳
    pub commit_ts: Option<u64>, // 提交时间戳 (None = 未提交)
    pub value: Vec<u8>,       // 行数据
    pub prev_version: Option<Pointer>,  // 指向旧版本的指针
}
```

### 2.2 快照结构

```rust
pub struct Snapshot {
    pub snapshot_timestamp: u64,      // 快照时间戳
    pub active_txs: HashSet<TxId>,   // 活跃事务集合
}

impl Snapshot {
    /// 判断版本是否对当前快照可见
    pub fn is_visible(&self, tx_id: TxId, commit_ts: Option<u64>) -> bool {
        // 1. 如果是当前事务的修改 → 可见
        // 2. 如果 commit_ts <= snapshot_timestamp → 可见
        // 3. 如果事务在活跃列表中 → 不可见
        // 4. 否则 → 不可见
    }
}
```

## 3. 读操作执行

### 3.1 快照读取流程

```
SELECT * FROM orders WHERE id = 1
    │
    ▼
┌─────────────────────────────────────────────┐
│              Storage Layer                    │
│  ┌─────────────────────────────────────┐  │
│  │ 1. 获取当前事务的快照                │  │
│  │    snapshot_timestamp: 500           │  │
│  │    active_txs: {101, 102}          │  │
│  └─────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│              MVCC Engine                     │
│  ┌─────────────────────────────────────┐  │
│  │ 1. 定位行 "orders:1"               │  │
│  │ 2. 遍历版本链                       │  │
│  │ 3. 对每个版本调用 is_visible()     │  │
│  │ 4. 返回第一个可见版本               │  │
│  └─────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│              Version Selection                │
│  ┌─────────────────────────────────────┐  │
│  │ Version 3: tx_id=102, cts=450      │  │
│  │   - 102 in active_txs? YES        │  │
│  │   - is_visible? NO                │  │
│  │                                     │  │
│  │ Version 2: tx_id=101, cts=400      │  │
│  │   - 101 in active_txs? YES        │  │
│  │   - is_visible? NO                │  │
│  │                                     │  │
│  │ Version 1: tx_id=50, cts=300      │  │
│  │   - 50 not in active_txs         │  │
│  │   - 300 <= 500? YES               │  │
│  │   - is_visible? YES ← 返回此版本  │  │
│  └─────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 3.2 可见性算法

```rust
impl Snapshot {
    pub fn is_visible(&self, tx_id: TxId, commit_ts: Option<u64>) -> bool {
        // 规则 1: 当前事务的修改总是可见
        if tx_id == self.current_tx_id {
            return true;
        }

        // 规则 2: 未提交事务的修改不可见
        if self.active_txs.contains(&tx_id) {
            return false;
        }

        // 规则 3: 提交时间戳在快照之前的才可见
        match commit_ts {
            Some(cts) => cts <= self.snapshot_timestamp,
            None => false,  // 未提交则不可见
        }
    }
}
```

## 4. 写操作执行

### 4.1 写操作流程

```
UPDATE orders SET total = 100 WHERE id = 1
    │
    ▼
┌─────────────────────────────────────────────┐
│              Validation (Snapshot Isolation)  │
│  ┌─────────────────────────────────────┐  │
│  │ 1. 检查是否有并发写冲突              │  │
│  │ 2. 检查目标行是否被其他事务修改     │  │
│  │ 3. 如果有冲突 → 事务回滚           │  │
│  └─────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│              Write Phase                      │
│  ┌─────────────────────────────────────┐  │
│  │ 1. 读取当前版本 (for undo)         │  │
│  │ 2. 生成新版本                       │  │
│  │    - tx_id: 当前事务 ID            │  │
│  │    - begin_ts: 当前时间戳          │  │
│  │    - commit_ts: None (未提交)      │  │
│  │    - prev_version: 指向当前版本    │  │
│  │ 3. 添加到版本链头部               │  │
│  │ 4. 生成 undo 记录                 │  │
│  └─────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│              Commit Phase                     │
│  ┌─────────────────────────────────────┐  │
│  │ 1. 更新版本 commit_ts              │  │
│  │ 2. 清理过期版本                   │  │
│  │ 3. 释放锁                         │  │
│  └─────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 4.2 First-Writer-Wins vs First-Committer-Wins

```
┌─────────────────────────────────────────────────────────────┐
│                    冲突处理策略                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  First-Writer-Wins (Tandem):
│  ─────────────────────────────────────────────────────────  │
│  T1: 写入 v2 (tx_id=100)                                 │
│  T2: 写入 v3 (tx_id=101)                                 │
│  T1: COMMIT (早于 T2)                                   │
│  T2: COMMIT → 检测到冲突 → T2 回滚                      │
│                                                              │
│  First-Committer-Wins (Serial SI):
│  ─────────────────────────────────────────────────────────  │
│  T1: 写入 v2 (tx_id=100)                                 │
│  T2: 写入 v3 (tx_id=101)                                 │
│  T1: COMMIT → 通过验证 (101 未提交)                       │
│  T2: COMMIT → 检测到冲突 → T2 回滚                       │
│                                                              │
│  SQLRustGo 使用: First-Committer-Wins (Snapshot Isolation) │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 5. 版本链管理

### 5.1 GC (垃圾回收)

```
┌─────────────────────────────────────────────────────────────┐
│                    版本链 GC                                │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  问题: 版本链无限增长                                       │
│  ─────────────────────────────────────────────────────────  │
│  T1 (100) → T2 (200) → T3 (300) → T4 (400) → ...      │
│                                                              │
│  GC 策略:                                                   │
│  ─────────────────────────────────────────────────────────  │
│  1. 基于时间: 删除 commit_ts < oldest_active_snapshot     │
│  2. 基于引用计数: 无活跃事务引用时删除                      │
│  3. 主动合并: 版本合并                                   │
│                                                              │
│  示例:                                                     │
│  ─────────────────────────────────────────────────────────  │
│  oldest_active_snapshot = 350                             │
│  T1 (commit_ts=200) → 可以删除                          │
│  T2 (commit_ts=300) → 可以删除                          │
│  T3 (commit_ts=400) → 保留                              │
│  T4 (commit_ts=500) → 保留 (最新)                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 GC 触发条件

```rust
pub struct GCConfig {
    /// 超过此阈值的旧版本被清理
    pub version_age_threshold: u64,
    /// 检查间隔
    pub gc_interval_ms: u64,
    /// 每次清理的最大版本数
    pub max_versions_per_row: usize,
}

impl GCConfig {
    pub fn should_gc(&self, version: &VersionRecord, oldest_snapshot: u64) -> bool {
        match version.commit_ts {
            Some(cts) => cts < oldest_snapshot,
            None => false,  // 未提交的版本不能 GC
        }
    }
}
```

## 6. SSI (Serializable Snapshot Isolation)

### 6.1 SSI 检测原理

```
┌─────────────────────────────────────────────────────────────┐
│                    SSI 冲突检测                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  两种危险结构:                                              │
│  ─────────────────────────────────────────────────────────  │
│                                                              │
│  1. 读写冲突 (RW):                                        │
│  ─────────────────────────────────────────────────────────  │
│  T1: SELECT x (read)     T2: SELECT x (read)            │
│  T1: UPDATE x (write)    T2: UPDATE x (write)            │
│  T1: COMMIT              T2: COMMIT ← 都要成功?         │
│                                                              │
│  2. 写读冲突 (WR):                                        │
│  ─────────────────────────────────────────────────────────  │
│  T1: SELECT x (read)     T2: UPDATE x (write)            │
│  T1: UPDATE x (write)    T2: COMMIT                       │
│  T1: COMMIT ← 成功?                                      │
│                                                              │
│  SSI 解决方案: 检测并回滚其中之一                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 SSI 状态跟踪

```rust
pub struct SSIState {
    /// 读集合: 本事务读取的行
    read_set: HashSet<RowKey>,
    /// 写集合: 本事务写入的行
    write_set: HashSet<RowKey>,
    /// 冲突图: 记录读写依赖关系
    dependency_graph: HashMap<TxId, HashSet<TxId>>,
}
```

## 8. 覆盖率差距分析

### 8.1 当前覆盖率

| 组件 | 行覆盖率 | 说明 |
|------|----------|------|
| mvcc.rs | ~70% | 基础 MVCC |
| version_chain.rs | ~60% | 版本链 |
| ssi.rs | ~50% | Serializable |

### 8.2 差距原因

1. **GC 优化**: 尚未实现高效的 GC
2. **版本合并**: 同一事务多次修改未合并
3. **SSI 优化**: 冲突检测有性能问题

| 阶段 | 任务 | 目标覆盖率 |
|------|------|-----------|
| v3.1.0 | 高效 GC | 80% |
