### 1.2 ACID 保障

```
┌─────────────────────────────────────────────────────────────┐
│                    ACID 与崩溃恢复                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Atomicity (原子性)                                         │
│  ├── 事务要么全部提交，要么全部回滚                        │
│  └── WAL 确保原子性: 提交前的所有操作都已记录              │
│                                                              │
│  Consistency (一致性)                                       │
│  ├── 崩溃后数据库必须恢复到一致状态                        │
│  └── 恢复算法确保一致性约束不被破坏                        │
│                                                              │
│  Isolation (隔离性)                                         │
│  ├── MVCC 确保并发事务隔离                                  │
│  └── 崩溃恢复不影响其他事务                                │
│                                                              │
│  Durability (持久性)                                        │
│  ├── COMMIT 必须等 WAL 刷盘完成                            │
│  └── 崩溃后已提交事务的数据必须存在                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 2. 崩溃场景分析

### 2.1 场景 1: WAL 写入前崩溃

```
时间线 ─────────────────────────────────────────────────────►

事务 T1:
│
│  ┌─────────────┐
├──►│ 修改 Buffer  │
│   └─────────────┘
│         │
│         ▼
│   ┌─────────────┐
└───┤ 崩溃!       │  ← Buffer 修改丢失
    └─────────────┘
          │
          │ 恢复后
          ▼
    ┌─────────────┐
    │ T1 不存在   │  ← 正确：事务未提交
    └─────────────┘

结论: 需要回滚 T1
```

### 2.2 场景 2: WAL 写入后、数据页写入前崩溃

```
时间线 ─────────────────────────────────────────────────────►

事务 T1:
│
│   ┌─────────────┐
├──►│ 写 WAL      │
│   └─────────────┘
│         │ lsn=1001
│         ▼
│   ┌─────────────┐
└───┤ 崩溃!       │  ← Buffer 修改丢失，但 WAL 存在
    └─────────────┘
          │
          │ 恢复后
          ▼
    ┌─────────────────────────────────────────────┐
    │ 扫描 WAL，发现 T1 的 BEGIN 和 INSERT 条目   │
    │ 但没有 COMMIT 条目                          │
    └─────────────────────────────────────────────┘
          │
          ▼
    ┌─────────────┐
    │ T1 回滚     │  ← 正确：未提交事务回滚
    └─────────────┘

结论: 需要回滚 T1 (没有 COMMIT)
```

### 2.3 场景 3: COMMIT 写入后崩溃

```
时间线 ─────────────────────────────────────────────────────►

事务 T1:
│
│   ┌─────────────┐
├──►│ 写 WAL      │
│   └─────────────┘
│         │
│   ┌─────────────┐
├──►│ 修改 Buffer │
│   └─────────────┘
│         │
│   ┌─────────────┐
├──►│ COMMIT WAL  │
│   └─────────────┘
│         │
│   ┌─────────────┐
└───┤ 崩溃!       │  ← Buffer 修改丢失，但 COMMIT 已记录
    └─────────────┘
          │
          │ 恢复后
          ▼
    ┌─────────────────────────────────────────────┐
    │ 扫描 WAL，发现 T1 的 BEGIN、INSERT、COMMIT  │
    └─────────────────────────────────────────────┘
          │
          ▼
    ┌─────────────────────────────────────────────┐
    │ 重放 WAL: 重新执行 INSERT 操作              │
    └─────────────────────────────────────────────┘
          │
          ▼
    ┌─────────────┐
    │ T1 已提交   │  ← 正确：已提交事务恢复
    └─────────────┘

结论: 需要重做 T1 (REDO)
```

### 3.2 恢复算法实现

```rust
pub fn recover(&self) -> RecoveryResult {
    // Phase 1: Analysis - 构建事务状态表
    let mut tx_states: HashMap<u64, TxState> = HashMap::new();

    for entry in self.scan_wal() {
        match entry.entry_type {
            WalEntryType::Begin => {
                tx_states.insert(entry.tx_id, TxState::Active);
            }
            WalEntryType::Commit => {
                if let Some(state) = tx_states.get_mut(&entry.tx_id) {
                    state.outcome = TxOutcome::Committed;
                }
            }
            WalEntryType::Rollback => {
                if let Some(state) = tx_states.get_mut(&entry.tx_id) {
                    state.outcome = TxOutcome::RolledBack;
                }
            }
            WalEntryType::Prepare => {
                if let Some(state) = tx_states.get_mut(&entry.tx_id) {
                    state.has_prepare = true;
                }
            }
            _ => { /* data entries */ }
        }
    }

    // Phase 2: REDO - 重做已提交事务
    for entry in self.scan_wal() {
        let state = match tx_states.get(&entry.tx_id) {
            Some(s) => s,
            None => continue, // 未知事务，跳过
        };

        if state.outcome == TxOutcome::Committed {
            self.redo_entry(entry);
        }
    }

    // Phase 3: UNDO - 回滚未提交事务
    for entry in self.scan_wal().rev() { // 逆序回滚
        let state = match tx_states.get(&entry.tx_id) {
            Some(s) => s,
            None => continue,
        };

        if state.outcome == TxOutcome::Active {
            self.undo_entry(entry);
        }
    }
}
```

### 5.2 协调者恢复

```rust
fn recover_coordinator(&self) {
    // 1. 恢复事务状态表
    let tx_states = self.load_tx_states();

    for (gid, state) in tx_states {
        match state {
            TxState::Committed => {
                // 已提交，通知所有参与者
                for participant in state.participants {
                    self.send_commit(participant, gid);
                }
            }
            TxState::Prepared => {
                // 等待决定，可以选择 commit 或 rollback
                // 这里选择 rollback（保守策略）
                for participant in state.participants {
                    self.send_rollback(participant, gid);
                }
            }
            TxState::Unknown => {
                // 查询参与者状态
                self.recover_unknown_tx(gid);
            }
        }
    }
}
```

## 6. 崩溃注入测试

### 6.1 测试框架

```rust
/// 崩溃注入点
pub enum CrashPoint {
    BeforeWalWrite,     // WAL 写入前
    AfterWalWrite,     // WAL 写入后，提交前
    BeforeCommit,       // 提交标记前
    AfterCommit,        // 提交标记后
    BeforeCheckpoint,   // 检查点前
    AfterCheckpoint,    // 检查点后
}

/// 启用崩溃注入
pub fn enable_crash_injection(point: CrashPoint) {
    CRASH_INJECT_ENABLED.store(true, Ordering::SeqCst);
    CRASH_INJECT_POINT.store(point as usize, Ordering::SeqCst);
}

/// 触发崩溃
fn trigger_crash() {
    eprintln!("[CRASH INJECTION] Triggering crash at configured point");
    std::process::abort();
}
```

### 6.2 测试用例矩阵

| 崩溃点 | 事务状态 | 恢复后期望 |
|--------|----------|------------|
| BeforeWalWrite | 无 WAL | 事务回滚 |
| AfterWalWrite | BEGIN + INSERT (无 COMMIT) | 事务回滚 |
| BeforeCommit | 有 COMMIT 条目但未刷盘 | 事务回滚 |
| AfterCommit | COMMIT 已刷盘 | 事务提交 |
| BeforeCheckpoint | 检查点未完成 | 从上次的检查点恢复 |
| AfterCheckpoint | 检查点完成 | 正常 |

### 6.3 测试用例

```rust
#[test]
fn test_crash_after_wal_write() {
    // 1. 创建数据库和表
    let db = create_test_db();

    // 2. 插入数据
    db.execute("INSERT INTO t VALUES (1)");

    // 3. 启用崩溃注入
    enable_crash_injection(CrashPoint::AfterWalWrite);

    // 4. 执行提交（触发崩溃）
    db.execute("COMMIT"); // 崩溃发生

    // 5. 重新启动数据库
    let db2 = create_test_db();

    // 6. 验证：数据应该不存在（事务回滚）
    let result = db2.query("SELECT * FROM t");
    assert_eq!(result.len(), 0);
}
```

## 9. 覆盖率差距分析

### 9.1 当前覆盖率

| 组件 | 行覆盖率 | 说明 |
|------|----------|------|
| recovery.rs (transaction) | ~45% | 分布式事务恢复 |
| wal.rs (storage) | ~75% | WAL 恢复 |
| crash_inject_test.rs | ~60% | 崩溃注入测试 |

### 9.2 差距原因

1. **增量恢复未实现**: 每次都全量扫描 WAL
2. **并行恢复未实现**: 单线程恢复
3. **PITR (时间点恢复)**: 未实现
4. **分布式恢复边界情况**: 脑裂场景未覆盖
