# SQLRustGo v2.7.0 版本定义与开发计划

> 日期: 2026-04-19  
> 当前基线: `alpha/v2.6.0`  
> 目标版本: `v2.7.0`  
> 目标定位: 达到 MySQL 5.7 生产可用水平（普通中小企业场景）+ 可集成 GMP 合规审核系统（全文检索/RAG/图检索）

---

## 1. 2.6.0 现状评估（用于 2.7.0 立项）

基于 `docs/releases/v2.6.0/` 与当前仓库状态，v2.6.0 的主要结论：

1. `SQL-92` 扩展已显著推进，但文档与真实可执行测试目标仍有偏差。
2. 生产就绪门禁框架已经存在，但“证据化交付”（覆盖率、性能、安全）仍需加强。
3. MySQL 5.7 对齐还缺关键项：完整事务隔离落地、复制能力、备份恢复体系、运维可观测闭环。
4. GMP 方向能力已经有雏形（Vector/Graph/GMP 文档与模块），但未形成稳定的检索服务面。

结论：`v2.6.0` 适合作为“生产候选前夜（pre-production candidate）”，`v2.7.0` 应定位为“生产化与合规检索平台化版本”。

---

## 2. 长期版本演进重定义（统一口径）

为避免现有路线图多口径冲突，统一如下：

1. `v2.6.0`: SQL 与内核收敛（功能正确性与门禁基线）
2. `v2.7.0`: 生产化版本（MySQL 5.7 级能力 + GMP 检索平台）
3. `v2.8.0`: 分布式增强版本（横向扩展、跨节点一致性强化）
4. `v3.0.0`: 云原生/平台化版本（多租户、自动弹性、运维平台）

---

## 3. v2.7.0 版本定义

### 3.1 一句话定义

`v2.7.0` 是 SQLRustGo 第一个“可在普通中小企业生产环境落地”的 GA 目标版本，同时提供 GMP 合规检索系统所需的多模态查询能力。

### 3.2 目标能力边界

1. MySQL 5.7 对齐范围（生产必要子集，不追求全部边缘特性）：
- SQL DDL/DML 主路径
- 事务与隔离（至少 RC/RR/可配置 Serializable 路径）
- 索引与优化器可用
- 备份恢复与崩溃恢复
- 基础复制能力（单主多从）
- 监控与运维指标

2. GMP 合规检索能力：
- 全文检索（关键字/BM25）
- 语义检索（RAG，向量召回 + 重排）
- 图检索（关系追溯、影响路径）
- 混合检索编排（单入口、可解释返回）

3. 集成 QMD 能力（作为可插拔检索编排组件）：
- 通过 CLI/MCP/HTTP 方式接入 QMD
- 接入其 hybrid retrieval（FTS + vector + rerank）能力
- 支持离线/本地优先部署策略

---

## 4. MySQL 5.7 对齐目标（v2.7.0 DoD）

### 4.1 功能对齐（必须）

1. SQL 主路径: `SELECT/INSERT/UPDATE/DELETE/JOIN/GROUP BY/HAVING/Subquery`
2. 事务: ACID、WAL、崩溃恢复、隔离级别回归测试
3. 约束: PK/UK/FK（含级联策略）
4. 索引与计划: B+Tree + 基础 CBO 可用
5. 运维: 慢查询、连接池、指标导出、健康检查

### 4.2 非功能指标（必须）

1. 稳定性: 72h 长稳压测通过，无数据损坏
2. 性能: 关键 OLTP 场景达到 `v2.6.0` 目标以上并可复现
3. 可观测性: Prometheus 指标完整、关键路径可追踪
4. 可恢复性: 备份/恢复演练通过，RTO/RPO 有明确定义

---

## 5. GMP + QMD 集成架构（v2.7.0）

### 5.1 架构分层

1. `SQLRustGo Core`: 事务、存储、SQL、Graph、Vector
2. `Retrieval Orchestrator`: 混合检索编排（全文/向量/图）
3. `QMD Bridge`: 对接 QMD 的 MCP/HTTP/CLI 入口（可配置）
4. `GMP Compliance API`: 审核、追溯、证据检索统一接口

### 5.2 查询路由策略

1. `lex`（关键词/BM25）: 制度文本、SOP、批记录精确命中
2. `vec`（语义）: 偏语义问题与相似案例召回
3. `graph`（关系）: 批次-设备-人员-偏差事件追溯
4. `hybrid`（融合）: RRF/重排后返回统一证据包

### 5.3 证据返回规范

统一返回：
- 命中片段
- 来源文档/表/图节点
- 评分与排序依据
- 审计时间戳
- 可追溯路径（Graph path）

## 6. v2.7.0 分阶段开发计划

## Phase A（4 周）: 内核生产化

目标：打通 MySQL 5.7 生产必要能力的核心闭环。

任务：
1. 事务/WAL/恢复路径全链路验证与缺陷清零
2. FK/约束/索引主路径补齐
3. 备份恢复脚本与演练文档落地
4. Release Gate 与证据化报告模板固化

退出条件：
1. 基础回归全绿
2. 崩溃恢复与恢复一致性测试通过

---

## Phase B（4 周）: 检索引擎融合

目标：形成可运行的“全文 + 语义 + 图”融合检索服务。

任务：
1. 增加 `qmd-bridge` 模块（可置于 `crates/tools` 或独立 crate）
2. 统一检索 API（支持 `lex/vec/graph/hybrid`）
3. 加入重排与阈值策略
4. 输出证据包格式（为 GMP 审核消费）

退出条件：
1. 同一查询可返回多路证据
2. 所有返回可定位到原始数据

---

## Phase C（3 周）: GMP 场景化能力

目标：面向 GMP 审核系统的关键查询场景上线。

任务：
1. 典型场景库：偏差追溯、批次影响分析、CAPA 关联
2. 图关系模板：Batch/Device/Process/Deviation/Action
3. 合规报表接口与导出
4. 审计日志与权限边界完善

退出条件：
1. 关键 GMP 查询场景全部通过验收用例
2. 审计证据链完整

---

## Phase D（3 周）: RC/GA 冲刺

目标：完成 `v2.7.0-rc1` 到 `v2.7.0-GA` 的发布门禁。

任务：
1. 性能基线复测与回归对比
2. 安全审计与依赖审计
3. 72h 长稳压测
4. 发布文档、升级指南、回滚预案冻结

退出条件：
1. 全部门禁通过
2. 具备中小企业生产部署指南

---

## 7. 版本时间线（建议）

1. `v2.7.0-alpha`: 2026-06-15（Phase A 完成）

3. `v2.7.0-rc1`: 2026-08-01（Phase D 开始）

## 8. 关键风险与缓解

1. 风险：MySQL 5.7 对齐范围过大导致延期  
缓解：坚持“生产必要子集”+ 明确不做项清单

2. 风险：QMD 集成引入外部依赖波动  
缓解：定义 `qmd-bridge` 抽象层，保留降级路径（仅内部检索）

3. 风险：GMP 场景需求膨胀  
缓解：先固化 Top 10 审核查询场景，剩余进入 2.7.x

4. 风险：评估报告不可复现  
缓解：所有性能/安全/覆盖率结论必须绑定命令、环境和产物路径

---

## 9. v2.7.0 验收门禁（最终）

1. 功能门禁：MySQL 5.7 关键子集通过
2. 质量门禁：CI、回归、文档命令检查全绿
3. 评估门禁：覆盖率/性能/安全报告可复现
4. 业务门禁：GMP 核心检索场景通过并可审计
5. 运维门禁：备份恢复与回滚演练通过
