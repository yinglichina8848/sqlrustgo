## 2. 执行正确性保证

### 2.1 事务隔离：SSI + MVCC

SQLRustGo v3.1.0 实现可序列化快照隔离（SSI）结合多版本并发控制（MVCC），以确保事务正确性。

#### 2.1.1 SSI（可序列化快照隔离）

SSI 在不阻塞读取器的情况下提供可序列化隔离，在提交时检测危险结构模式（RW-WR 循环）。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         SSI 事务流程                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  T1: BEGIN                                                                  │
│      │                                                                         │
│      ▼                                                                         │
│  ┌─────────────────┐                                                          │
│  │ 创建快照        │  snapshot = {commit_ts, active_tx_list}                 │
│  └────────┬────────┘                                                          │
│           │                                                                    │
│           ▼                                                                    │
│  ┌─────────────────┐                                                          │
│  │   读取阶段      │  MVCCStorage.read(key, snapshot)                        │
│  │                 │  ├── 查找所有版本                                         │
│  │                 │  ├── 检查可见性 (is_visible)                              │
│  │                 │  └── 在 SsiDetector 中记录读取                            │
│  └────────┬────────┘                                                          │
│           │                                                                    │
│           ▼                                                                    │
│  ┌─────────────────┐                                                          │
│  │   写入阶段      │  MVCCStorage.write(key, value, tx_id)                    │
│  │                 │  ├── 追加到 VersionChain                                │
│  │                 │  └── 在 SsiDetector 中记录写入                          │
│  └────────┬────────┘                                                          │
│           │                                                                    │
│           ▼                                                                    │
│  ┌─────────────────┐                                                          │
│  │ 提交验证        │  SsiDetector.validate_commit(tx_id)                     │
│  │                 │  ├── 检查 RW-WR 循环                                     │
│  │                 │  ├── 检查 WR-WR 冲突                                     │
│  │                 │  └── 如果检测到循环 → 中止                               │
│  └────────┬────────┘                                                          │
│           │                                                                    │
│           ├─ 成功 ──▶ 提交: 更新 commit_ts, 截断版本                          │

#### 2.1.3 范围谓词的 Gap 锁

Gap 锁通过锁定索引条目之间的间隙来防止幻读。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Gap 锁机制                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Index: [10] [20] [30] [40]                                                 │
│                                                                              │
│  Gap:   (10,20)  (20,30)  (30,40)  (40,+∞)                                 │
│                                                                              │
│  SQL: SELECT * FROM t WHERE id > 15 AND id < 25 FOR UPDATE                   │
│                                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │  锁获取:                                                              │    │
│  │                                                                       │    │
│  │  1. WHERE id > 15 → Gap { start: 15, end: None }                    │    │
│  │     锁: (15, +∞)                                                     │    │
│  │                                                                       │    │
│  │  2. WHERE id < 25 → Gap { start: None, end: 25 }                    │    │
│  │     锁: (-∞, 25)                                                     │    │
│  │                                                                       │    │
│  │  3. 组合效果: 锁 (15, 25)                                            │    │
│  │     防止: INSERT 22, UPDATE 18, DELETE 21                            │    │
│  │                                                                       │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                                                                              │
│  锁类型映射:                                                                │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │  条件        │  锁目标                  │  锁                       │    │
│  │-------------│-------------------------│-------------------------│    │
│  │  id = 5     │  NextKey(5)             │  记录 5 + (5,next)      │    │
│  │  id > 5     │  Gap{start:5, end:None} │  (5, +∞)                │    │
│  │  id < 5     │  Gap{start:None, end:5} │  (-∞, 5)                │    │
│  │  id >= 5    │  Gap{start:5, end:None}│  [5, +∞)                │    │
│  │  id <= 5    │  Gap{start:None, end:5} │  (-∞, 5]                │    │
│  │  id BETWEEN │  Gap{start:x, end:y}   │  (x, y)                 │    │
│  │    x AND y  │                         │                          │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘

## 4. 可验证性

### 4.1 TLA+ 形式化规范

SQLRustGo v3.1.0 包含关键组件的 TLA+ 规范，支持系统正确性的形式化验证。

#### 4.1.1 MVCC 可见性规范

```tla
(* MVCC Visibility Rules *)
IsVisible(version, snapshot) ==
    /\ version.commit_ts < snapshot.commit_ts
    /\ version.tx_id \notin snapshot.active_txs
    /\ ~version.aborted

ReadSnapshot(tx_id, key) ==
    LET versions == GetVersionChain(key)
    IN  CHOOSE v \in versions :
        /\ IsVisible(v, CreateSnapshot(tx_id))
        /\ \A w \in versions :
            w.commit_ts > v.commit_ts => ~IsVisible(w, CreateSnapshot(tx_id))
```

#### 4.1.2 SSI 检测规范

```tla
(* SSI Dangerous Structure Detection *)
DetectCycle(graph) ==
    LET reachable == { path \in Seq(Node) :
        /\ path[1] = StartNode
        /\ \A i \in 1..(Len(path)-1) : 
            edge(path[i], path[i+1]) \in graph
        /\ path[Len(path)] = StartNode
    }
    IN  reachable /= {}

ValidateCommit(tx) ==
    LET read_set == GetReadSet(tx)
        write_set == GetWriteSet(tx)
        rw_conflicts == { r \in read_set : 
            \E w \in write_set : w.key = r.key }
        wr_conflicts == { w \in write_set : 
            \E r \in read_set : r.key = w.key }
    IN  /\ ~DetectCycle(BuildConflictGraph(tx, rw_conflicts, wr_conflicts))
        /\ AllWritesValid(tx, write_set)
```

### 5.4 行级安全

行级安全根据用户上下文过滤数据：

```rust
// 行级安全执行
fn execute_with_row_policy(query: &Query, user: &User) -> Result<Query> {
    let row_policy = get_row_policy(user, query.table())?;
    match row_policy {
        RowPolicy::None => Ok(query.clone()),
        RowPolicy::Filter(expr) => {
            // 追加 WHERE 子句到现有 WHERE
            let new_where = combine_where(query.where(), expr);
            Query { where: new_where, ..query.clone() }
        },
        RowPolicy::ColumnMask(cols) => {
            // 屏蔽特定列
            let masked_cols = cols.iter()
                .map(|c| ColumnMask::new(c, MaskType::Redacted))
                .collect();
            Query { column_masks: masked_cols, ..query.clone() }
        }
    }
}
```

#### L-05: MERGE 执行器未实现

```
缓解策略：
1. 使用 INSERT ... ON CONFLICT DO UPDATE（PostgreSQL 语法）
2. 使用应用层 UPSERT 逻辑（SELECT + INSERT/UPDATE）
3. 在应用程序代码中使用事务实现 MERGE

示例变通方案：
-- 不使用 MERGE INTO target USING source ON ...
-- 使用: INSERT ... ON DUPLICATE KEY UPDATE（MySQL 语法）

INSERT INTO target (id, name)
VALUES (?, ?)
ON DUPLICATE KEY UPDATE name = VALUES(name);

-- 或应用层：
let existing = target.find_by_id(id)?;
if existing.is_some() {
    target.update(id, data)?;
} else {
    target.insert(data)?;
}
```

- 非常高写入并发（> 1000 TPS 持续）
- 大型分析数据集（使用 OLAP 数据库进行分析）
- 需要 HSM 的监管合规（等待 v3.2.0 HSM 支持）

### 8.3 v3.2.0 路线图

| 功能 | 优先级 | 预期影响 |
|---------|----------|-----------------|
| 列式存储 | P1 | OLAP 性能提升 10 倍 |
| HSM 密钥管理 | P1 | 监管合规 |
| Syslog 审计导出 | P2 | SIEM 集成 |
| MERGE 执行器 | P1 | UPSERT 简化 |
| 并行查询执行 | P2 | 多核利用 |

### 8.4 最终建议

1. **对于新项目**：推荐 SQLRustGo v3.1.0 用于需要信任执行保证的嵌入式 OLTP 工作负载。

2. **对于现有 v3.0.0 用户**：由于 SSI 检测、审计链和加密方面的重大改进，建议升级。

3. **对于监管合规**：在合规性评估中记录已知限制（L-01 至 L-05）。TLA+ 规范和 SHA-256 审计链提供了强有力的尽职调查证据。

4. **对于混合 OLTP/OLAP 工作负载**：如第 7.2 节所述，使用 SQLRustGo 处理 OLTP，使用单独的 OLAP 数据库进行分析。
