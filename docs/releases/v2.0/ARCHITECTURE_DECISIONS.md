## 什么是 ADR?

架构决策记录 (Architecture Decision Records) 是记录架构决策的文档，包含决策背景、决策内容、备选方案和权衡理由。

## ADR 001: 使用 Rust 作为实现语言

### 状态: 已批准

### 背景

选择数据库系统的实现语言，需要考虑性能、安全性和开发效率。

### 决策

使用 Rust 作为主要开发语言。

### 理由

| 因素 | Rust | Java | Go |
|------|------|------|-----|
| 性能 | ✅ 高 | 中 | 高 |
| 内存安全 | ✅ 无 GC | GC | GC |
| 并发安全 | ✅ 所有权系统 | 中 | 中 |
| 开发效率 | 中 | ✅ 高 | ✅ 高 |
| 生态系统 | 增长中 | ✅ 成熟 | ✅ 成熟 |

**选择 Rust 的原因**:
1. 内存安全，无需 GC，适合数据库场景
2. 所有权系统提供天然的并发安全
3. 接近 C 的性能，适合性能敏感场景
4. 现代语言特性，提高开发效率

### 后果

- **正面**: 高性能、内存安全、并发安全
- **负面**: 学习曲线陡峭、编译时间长、生态系统相对年轻

## ADR 002: 采用 Volcano 模型作为执行引擎

### 状态: 已批准

### 背景

选择查询执行模型，需要平衡简单性、灵活性和性能。

### 决策

采用 Volcano 模型（迭代器模型）作为执行引擎基础。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| Volcano | 简单、灵活、易优化 | 每次迭代开销 |
| 向量化 | 批处理、高吞吐 | 实现复杂 |
| 物化 | 简单 | 内存占用大 |

### 决策理由

1. **简单性**: Volcano 模型概念简单，易于理解和实现
2. **灵活性**: 易于添加新算子，优化规则
3. **可优化性**: 支持多种优化策略（谓词下推、投影下推）
4. **兼容性**: 与主流数据库（PostgreSQL、MySQL）模型一致

### 后果

- **正面**: 实现简单、易于维护、可扩展性好
- **负面**: 行级迭代有开销，后续通过向量化优化

## ADR 003: 使用 B+Tree 作为默认索引

### 状态: 已批准

### 背景

选择索引结构，需要考虑读性能、写性能和空间效率。

### 决策

使用 B+Tree 作为默认索引结构。

### 备选方案

| 方案 | 读性能 | 写性能 | 空间 | 适用场景 |
|------|--------|--------|------|----------|
| B+Tree | 高 | 中 | 大 | 范围查询、主键 |
| LSM Tree | 中 | 高 | 小 | 写入密集 |
| Hash | 极高 | 高 | 大 | 点查询 |

### 决策理由

1. **范围查询优势**: B+Tree 叶子节点链表顺序，适合范围查询
2. **稳定性**: 性能可预测，无 LSM 的波动
3. **实现成熟**: B+Tree 实现参考丰富
4. **兼容性**: 与 MySQL、PostgreSQL 一致

### 后果

- **正面**: 范围查询高效、实现成熟
- **负面**: 写入时需维护索引、有空间开销

## ADR 004: 采用 MVCC 实现事务并发控制

### 状态: 已批准

### 背景

选择事务并发控制机制，需要考虑隔离级别支持和性能。

### 决策

采用 MVCC (Multi-Version Concurrency Control) 作为主要并发控制机制。

### 备选方案

| 方案 | 隔离级别 | 性能 | 实现复杂度 |
|------|----------|------|------------|
| MVCC | 全部 | 高 | 高 |
| 2PL | 全部 | 中 | 中 |
| 乐观锁 | RC | 高 | 低 |

### 决策理由

1. **读不阻塞**: 读取操作不阻塞写操作，提高并发
2. **快照隔离**: 支持 REPEATABLE READ 和 READ COMMITTED
3. **无锁读**: 读取快照无需加锁，提高读性能
4. **行业标准**: PostgreSQL、MySQL 都采用 MVCC

### 后果

- **正面**: 高并发读性能、支持多级别隔离
- **负面**: 需要垃圾回收、存储开销增加

## ADR 005: 分层架构设计

### 状态: 已批准

### 背景

组织系统架构，需要平衡模块化、可维护性和性能。

### 决策

采用分层架构：Parser → Planner → Optimizer → Executor → Storage。

### 架构层次

```
┌─────────────────────────────────────────┐
│              Client Layer              │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│              Service Layer               │
│   (Connection Pool, Session Management)  │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│           Core Engine Layer             │
│  Parser → Planner → Optimizer → Executor │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│             Storage Layer                │
│  (Buffer Pool, B+Tree, WAL, Catalog)   │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│           Transaction Layer              │
│     (MVCC, Lock Manager, Recovery)      │
└─────────────────────────────────────────┘
```

### 决策理由

1. **模块化**: 每层职责清晰，便于开发和维护
2. **可测试**: 每层可独立测试
3. **可扩展**: 易于添加新功能
4. **性能**: 减少层间调用开销

### 后果

- **正面**: 结构清晰、易于维护、可测试性好
- **负面**: 层间调用有开销（通过优化缓解）

## ADR 006: 使用 WAL 实现持久化

### 状态: 已批准

### 背景

确保数据持久性，需要可靠的崩溃恢复机制。

### 决策

使用 Write-Ahead Logging (WAL) 作为持久化机制。

### 备选方案

| 方案 | 可靠性 | 性能 | 实现复杂度 |
|------|--------|------|------------|
| WAL | 高 | 中 | 中 |
| 影子页面 | 高 | 低 | 高 |
| 定期刷盘 | 低 | 高 | 低 |

### 决策理由

1. **可靠性**: WAL 是业界标准，数据不丢失
2. **性能**: 顺序写入，开销小
3. **恢复快**: 崩溃后快速恢复
4. **实现参考**: PostgreSQL、SQLite 成功实践

### 后果

- **正面**: 数据可靠、恢复快速
- **负面**: 需要定期检查点、控制 WAL 大小

## ADR 007: 基于成本的优化器

### 状态: 已批准

### 背景

选择查询优化策略，需要平衡优化质量和实现复杂度。

### 决策

实现基于成本的优化器 (CBO)，辅以规则优化。

### 备选方案

| 方案 | 质量 | 复杂度 | 性能 |
|------|------|--------|------|
| 规则优化 | 中 | 低 | 快 |
| CBO | 高 | 高 | 慢 |
| 混合 | 高 | 中 | 中 |

### 决策理由

1. **质量**: CBO 能选择更优的执行计划
2. **统计信息**: 利用表统计信息进行估算
3. **可扩展**: 易于添加新规则
4. **兼容性**: 与 PostgreSQL、MySQL 一致

### 后果

- **正面**: 优化质量高、可利用统计信息
- **负面**: 实现复杂、需要维护统计信息

## ADR 008: Workspace 模块化设计

### 状态: 已批准

### 背景

组织 Rust 代码结构，需要平衡编译速度和模块化。

### 决策

使用 Cargo workspace 组织多 crate 模块。

### 模块划分

```
sqlrustgo/
├── crates/
│   ├── parser/      # SQL 解析
│   ├── planner/    # 查询规划
│   ├── optimizer/  # 查询优化
│   ├── executor/   # 执行引擎
│   ├── storage/   # 存储引擎
│   ├── transaction/# 事务管理
│   ├── catalog/    # 目录服务
│   └── server/     # 服务器
└── src/
    └── lib.rs      # 主入口
```

### 决策理由

1. **编译优化**: 修改单 crate 只需重新编译
2. **模块化**: 清晰分离关注点
3. **测试**: 易于单独测试各模块
4. **生态**: 便于发布独立 crate

### 后果

- **正面**: 编译快、结构清晰、易于测试
- **负面**: 依赖管理复杂、测试协调成本

## ADR 009: 支持多种存储引擎

### 状态: 已批准

### 背景

满足不同场景需求，需要灵活的存储后端。

### 决策

抽象存储引擎接口，支持多种实现。

### 存储引擎

| 引擎 | 特点 | 适用场景 |
|------|------|----------|
| MemoryStorage | 内存、极快 | 测试、开发 |
| FileStorage | 磁盘、持久 | 生产环境 |
| ColumnarStorage | 列式、分析 | OLAP |

### 决策理由

1. **灵活性**: 满足不同场景
2. **可扩展**: 易于添加新引擎
3. **测试**: MemoryStorage 便于测试

### 后果

- **正面**: 灵活、可扩展、便于测试
- **负面**: 接口抽象有开销
