# v2.8.0 → MySQL 5.7 替代可行性评估

> **评估日期**: 2026-05-02
> **评估基准**: SQLRustGo v2.8.0 GA develop/v2.8.0 (159beb3b)
> **对比目标**: MySQL 5.7 (生产级)
> **总评分**: 62/100 — **不推荐生产替代**

---

## 二、逐维度深度分析

### 2.1 SQL 语言覆盖度 — ❌ [35/100]

**MySQL 5.7 核心 SQL 特性 vs v2.8.0 差距**:

| 功能类别 | MySQL 5.7 | v2.8.0 状态 | 差距 |
|----------|-----------|-------------|------|
| DDL (CREATE/ALTER/DROP) | 完整 | 基础 DDL ✅ | 缺 ALTER TABLE ADD/DROP/MODIFY |
| DML (SELECT/INSERT/UPDATE/DELETE) | 完整 | 基础 ✅ | 缺子查询优化, 缺LATERAL |
| JOIN (INNER/LEFT/RIGHT/FULL/CROSS) | 完整 | 基础 ✅ FULL 新增 | N/A |
| 窗口函数 | ROW_NUMBER/RANK/DENSE_RANK/NTILE/ LEAD/LAG/FIRST_VALUE/LAST_VALUE | ROW_NUMBER/RANK/DENSE_RANK ✅ | 缺 NTILE/LEAD/LAG/NTH_VALUE |
| CTE (WITH ... AS) | 支持递归WITH | ❌ 不支持 | **严重缺口** |
| JSON 函数 | JSON_EXTRACT/JSON_OBJECT/JSON_TABLE | ❌ 不支持 | **严重缺口** |
| 子查询 | 完整, 含相关子查询 | 基础, 缺复杂优化 | 中等缺口 |
| 聚合增强 | GROUPING SETS/CUBE/ROLLUP | ❌ 不支持 | 中等缺口 |
| 存储过程 | 完整控制流 | 基础 IF/WHILE/LOOP ✅ | 缺游标/异常处理 |
| 触发器 | BEFORE/AFTER/INSTEAD OF | 基础 ✅ | 缺触发器链 |
| 事件调度器 | CREATE EVENT | ❌ 不支持 | 缺口 |
| 全文索引 | MATCH ... AGAINST | ❌ 不支持 | 缺口 |
| 空间数据类型 | GEOMETRY/POINT/LINESTRING | ❌ 不支持 | 缺口 |
| 分区表 | RANGE/LIST/HASH/KEY/SUBPARTITION | RANGE/LIST/HASH/KEY ✅ | 缺子分区 |

**SQL Corpus 通过率**: 174/426 = **40.8%**
**Parser 行覆盖率**: 54.70% (PARSER_COVERAGE_ANALYSIS.md)

**评估**: SQL 兼容性是目前最大的生产化障碍。一个 MySQL 5.7 应用迁移过来，90%+ 的 SQL 会因缺少特性而失败。

### 2.2 存储引擎 — ⚠️ [50/100]

| 特性 | MySQL InnoDB | v2.8.0 | 差距 |
|------|-------------|--------|------|

| 二级索引 | ✅ B+Tree | ❌ 未实现 | **严重** |
| 缓冲池 | ✅ 可配置, LRU | ✅ BufferPool | 基础可用 |
| 行格式 | ✅ COMPACT/REDUNDANT/COMPRESSED | ❌ 固定格式 | 严重 |
| 自适应哈希索引 | ✅ | ❌ | 中等 |
| 变更缓冲区 | ✅ Change Buffer | ❌ | 中等 |
| 外键 | ✅ CASCADE/SET NULL | 基础约束 ✅ | 待完善 |
| CHECK 约束 | ✅ | ✅ v2.8.0 新增 | N/A |
| 数据压缩 | ✅ | ❌ | 缺口 |

| 向量存储 | ❌ | ✅ 试验性 | 差异特性 |

**评估**: B+Tree + BufferPool + WAL 提供了基本可用的存储骨架，但缺少 InnoDB 15年来积累的优化（聚簇索引、二级索引、AHI、Change Buffer），生产负载下性能不可预期。

| 特性 | MySQL 5.7 | v2.8.0 | 差距 |

| 隔离性 (I) | ✅ RU/RC/RR/SERIALIZABLE | ✅ RC/RR 声明 | 缺SERIALIZABLE |

| 2PC | ✅ XA 事务 | ✅ Prepare entry | 分布式基础 |

**评估**: 事务/WAL 部分是最接近 MySQL 5.7 水准的。但 MVCC 版本管理仅 4 个测试，缺少实际并发场景验证。

| 特性 | MySQL 5.7 | v2.8.0 | 差距 |

## 三、关键性能缺口汇总

## 四、适用场景建议

### ✅ 可试用 (非生产)

| 场景 | 理由 | 限制 |
|------|------|------|
| 开发/测试环境 | 基础 SQL 功能可用 | 复杂查询和性能不可靠 |
| 学习/原型 | 轻量级, 零配置启动 | 无生产保障 |
| 嵌入式 | Rust 编译, 可嵌入 | 功能有限 |
| 单用户工具 | REPL 模式, 小数据量 | 无并发 |

### ⚠️ 谨慎尝试 (生产不可用但可POC)

| 场景 | 理由 | 风险 |
|------|------|------|
| 数据量 < 1GB | 全表扫描可接受 | 缺少索引, 大表不可用 |
| 并发 < 10 | 8线程测试通过 | 100+并发未验证 |
| 简单 CRUD | DDL/DML 基础功能通过 | 复杂查询不可用 |
| 单机部署 | 分布式可用但无运维工具 | 故障恢复手动 |

### ❌ 不可替代 (不要尝试)

| 场景 | 原因 |
|------|------|
| OLTP 高并发 > 100 | 无性能基准, 无连接池 |
| 复杂 SQL 查询 | SQL Corpus 40.8%, 大量特性缺失 |
| 金融级 ACID | 缺 SERIALIZABLE, 缺 XA |
| 多数据中心 | 无 Group Replication |
| 地理空间数据 | 无 GEOMETRY 支持 |
| JSON 文档存储 | 无 JSON 函数 |
| 已有 MySQL 应用迁移 | SQL 不兼容率 59.2% |

## 五、达到 MySQL 5.7 生产替代的路线图

| 阶段 | 目标 | 关键交付 | 预估时间 |
|------|------|---------|---------|
| **v2.9.0** | 生产候选 | SQL Corpus ≥ 80%, 二级索引, sysbench 基准, 半同步复制, 慢查询日志 | 6个月 |
| **v3.0.0** | 有限生产 | SQL Corpus ≥ 95%, CTE, JSON, 角色管理, 数据加密, 连接池, 监控集成 | 12个月 |
| **v4.0.0** | MySQL 5.7 替代 | 全文索引, 空间数据, 组复制, events, all window functions, performance_schema | 18个月 |

**必要的团队规模**: 当前 1-5 人 → 需要至少 10-15 人核心开发团队 + 3-5 人 QA/SRE 才能认真考虑生产替代。

## 六、最终结论

```
SQLRustGo v2.8.0 → MySQL 5.7 生产替代可行性
=============================================
综合评分:     45.5/100
生产就绪度:   Alpha/Beta (非 GA)
推荐替代?:    ❌ 不推荐
适合场景:     开发测试/原型/嵌入式
最小差距:     v3.0.0 (12个月)
完全替代:     v4.0.0 (18个月+)
```

**一句话**: v2.8.0 是一个令人印象深刻的原型，展示了 Rust 在数据库领域的可行性。但距离 MySQL 5.7 的生产替代还有 **3-4 个大版本** 的距离，核心障碍不是功能数量而是生产化的深度 —— 索引子系统、性能基准、运维工具体系。
