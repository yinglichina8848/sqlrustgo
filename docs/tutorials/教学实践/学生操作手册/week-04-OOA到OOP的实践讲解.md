## 回顾与引入

### 第3周我们做了什么？

1. **OOA**：从需求中提取名词，发现对象（User、Book、BorrowRecord）
2. **OOD**：设计类结构，应用SOLID原则，绘制类图
3. **OOP**：用Rust实现基础代码（Student、Book、BorrowRecord）

### 本周的问题

- 有了静态结构（类图），如何描述**动态行为**？
- 系统如何与用户交互？对象之间如何协作？
- 对象内部状态如何变化？业务流程如何流转？
- 系统如何部署？组件如何组织？

### 答案：动态建模 + 架构设计

| 类型 | 描述 |
|------|------|
| 动态建模（顺序图、状态图、活动图） | 描述系统的**行为** |
| 架构设计（组件图、部署图） | 描述系统的**物理结构** |

最终通过**迭代细化**，将设计转化为可运行的OOP代码

## OOA → OOD → OOP 的核心思想

---

### OOA（面向对象分析）

**"做什么"**

1. 理解问题域，识别现实世界中的对象、属性、关系
2. **产出**：用例图、对象发现列表、领域模型

### OOD（面向对象设计）

**"怎么做"**

1. 定义软件对象、接口、协作，应用设计原则与模式
2. **产出**：类图、顺序图、状态图、活动图、组件图、部署图

### OOP（面向对象编程）

**"实现"**

1. 用具体编程语言实现设计，将对象转化为代码
2. **产出**：可运行的软件系统

**本周定位**：在OOD层面完善动态模型与架构，并迭代细化至OOP

## UML图例的综合运用——为什么要这么多图？

### UML是一套"多视角"建模语言

| UML图 | 问题 | 视角 |
|-------|------|------|
| 用例图 | 谁用系统？做什么？ | 需求视角 |
| 类图 | 系统有哪些对象？它们有什么关系？ | 静态结构视角 |
| 顺序图 | 对象如何交互？消息如何传递？ | 动态交互视角 |
| 状态图 | 对象的生命周期如何？状态如何变化？ | 状态变迁视角 |
| 活动图 | 业务流程如何？决策与并行？ | 流程视角 |
| 组件图 | 系统由哪些模块组成？接口如何定义？ | 架构视角 |
| 部署图 | 软件部署在哪些硬件上？ | 物理部署视角 |

### 综合运用的意义

1. 不同图回答不同问题，**互为补充**
2. 从不同角度验证设计的**一致性与完整性**
3. 便于**团队沟通**，降低理解偏差
4. 为代码实现提供**清晰的蓝图**

## 以"借书"为例——如何综合运用UML图

**场景**：学生借一本书

| UML图 | 作用 |
|-------|------|
| 用例图 | 明确"借书"是学生的用例，与"查询图书"有关联 |
| 类图 | 定义参与借书的类（User、Book、BorrowRecord、BorrowService） |
| 顺序图 | 描述学生、界面、控制器、服务、数据库之间的消息交互 |
| 状态图 | 刻画Book对象从"可借"到"已借出"的状态变迁 |
| 活动图 | 描述借书流程中的决策（是否超限？是否可借？） |
| 组件图 | 展示借书服务与其他组件的依赖关系 |
| 部署图 | 展示借书服务运行在哪个应用服务器上 |

**结论**：UML不是孤立的，而是围绕一个功能从多个角度**完整刻画**系统。

## 步骤1——用例图：定义系统边界与功能

### 为什么先用用例图？

1. 明确**谁**（参与者）使用系统的**哪些功能**（用例）
2. 建立用户与系统的**交互契约**
3. 指导后续动态建模的**范围**

### 借阅系统的用例图分析

**参与者**：学生、教师、图书管理员

**核心用例**：借书、还书、查询图书、管理图书、计算罚款

**关系**：
- 借书和还书都包含"查询图书"
- 还书包含"计算罚款"
- "预约图书"扩展"借书"

### AI生成用例图的提示词（课堂演示）

```
请为"高校图书借阅系统"生成用例图，使用Mermaid语法。

参与者：学生、教师、图书管理员
用例：登录、查询图书、借书、还书、管理图书、预约图书、查看借阅历史、计算罚款

关系：
- 学生和教师可以登录、查询图书、借书、还书、预约图书、查看借阅历史
- 图书管理员可以登录、管理图书、查看所有借阅记录
- 借书和还书都包含查询图书（<<include>>）
- 还书包含计算罚款（<<include>>）
- 预约图书扩展借书（<<extend>>）
```

**讲解要点**：为什么这样定义参与者？为什么用include/extend？体现用例的**重用与扩展**。

## 步骤2——顺序图：细化"借书"用例的交互

### 为什么用顺序图？

1. 用例图只说"借书"，但**如何借**？对象如何交互？
2. 顺序图揭示对象间的**消息传递、时间顺序、激活周期**
3. 帮助识别**缺失的对象或职责**

### 借书流程的交互分析

**参与者**：学生、UI、BorrowController、BookService、UserService、BorrowService、BookDB、BorrowDB

**关键步骤**：验证图书 → 验证用户借阅上限 → 创建记录 → 更新状态

**异常处理**：图书不存在、用户超限等用 `alt` 片段

### AI生成顺序图的提示词（课堂演示）

```
请为"高校图书借阅系统"的借书流程生成顺序图，使用PlantUML语法。

参与者：学生、UI、BorrowController、BookService、UserService、BorrowService、BookDB、BorrowDB

流程：
1. 学生输入图书ID提交借书请求
2. UI调用控制器的borrowBook(bookId, userId)
3. 控制器调用BookService的getBook(bookId)
4. BookService查询BookDB返回图书信息
5. 控制器调用UserService的checkBorrowLimit(userId)
6. UserService查询BorrowDB返回当前借阅数量
7. 如果未超限，控制器调用BorrowService的createBorrowRecord
8. BorrowService创建记录并更新图书状态
9. 返回借书成功消息

异常处理：图书不存在或用户超限时，返回错误消息
```

**讲解要点**：激活条的作用？为什么需要控制器？alt片段如何对应if-else？

## 步骤3——状态图：刻画"图书"对象的生命周期

### 为什么用状态图？

1. 类图只显示了图书的属性（如 `available: bool`），但无法表达状态转换的**规则和事件**
2. 状态图回答：一个对象在什么事件下从什么状态变为什么状态，以及状态中的**内部活动**

### 图书状态分析

| 要素 | 内容 |
|------|------|
| **状态** | 可借、已借出、预约中、遗失、报废 |
| **事件** | 借书、还书、预约、取消预约、报失、找到、报废 |
| **内部活动** | entry（进入时执行）、do（持续执行）、exit（退出时执行） |

### AI生成状态图的提示词（课堂演示）

```
请为"图书"对象生成状态图，使用PlantUML语法。

状态：可借（Available）、已借出（Borrowed）、预约中（Reserved）、遗失（Lost）、报废（Discarded）
初始状态：可借
终止状态：报废

转换事件：
- 借书：Available → Borrowed
- 预约：Available → Reserved
- 取消预约：Reserved → Available
- 借书（预约后）：Reserved → Borrowed
- 还书：Borrowed → Available
- 报失：Borrowed → Lost
- 找到：Lost → Available
- 报废：Available/Borrowed/Lost → Discarded

内部活动：
- Borrowed状态：entry/记录借出时间；do/计算借阅时长；exit/记录归还时间
```

**讲解要点**：为什么需要"预约中"状态？entry/do/exit分别对应什么时机？状态图如何指导代码中的状态模式或枚举？

## 步骤4——活动图：描述"还书"业务流程

### 为什么用活动图？

1. 顺序图强调对象**交互**，活动图强调**流程和并发**
2. 活动图可以跨越多个对象或**泳道**，展示业务过程中的**决策、并行、合并**

### 还书流程分析

| 要素 | 内容 |
|------|------|
| **泳道** | 学生、系统、仓库（体现职责分离） |
| **决策** | 是否超期 → 计算罚款 |
| **并行** | 记录还书完成 与 更新图书状态 同时进行 |

### AI生成活动图的提示词（课堂演示）

```
请为"高校图书借阅系统"的还书流程生成活动图，使用PlantUML语法。

泳道：学生、系统、仓库

流程：
- 学生提交还书请求
- 系统验证借阅记录
- 如果超期，计算罚款并生成罚单
- 系统更新图书状态为可借
- 并行：
   - 系统记录还书完成
   - 仓库更新库存信息
- 结束

使用fork/join表示并行，使用diamond表示决策
```

**讲解要点**：泳道对应什么？（对象或组织单元）并行如何实现？活动图与顺序图**互补关系**。

## 步骤5——组件图：模块化与可替换性

### 为什么用组件图？

1. 类图描述**微观对象**，组件图描述**宏观模块**
2. 组件是**物理封装、可替换**的单元，有明确的接口
3. 组件图体现**高内聚、低耦合**的设计思想

### 借阅系统的组件划分

| 层次 | 组件 |
|------|------|
| 前端 | 用户交互 |
| 服务组件 | 借书、还书、查询、用户、罚款 → 每个服务有独立职责 |
| 数据访问层 | 统一的数据访问接口 |
| 数据库 | MySQL/PostgreSQL |

### AI生成组件图的提示词（课堂演示）

```
请为"高校图书借阅系统"生成组件图，使用PlantUML语法。

组件：
- 前端应用（Frontend）
- 借书服务（BorrowService）
- 还书服务（ReturnService）
- 查询服务（QueryService）
- 用户服务（UserService）
- 罚款服务（FineService）
- 数据访问接口（Repository Interface）
- 数据库（Database）

依赖关系：
- 前端应用依赖所有服务组件
- 各服务组件依赖数据访问接口
- 数据访问接口由数据库实现

使用lollipop符号表示接口
```

**讲解要点**：依赖倒置原则在组件图中的体现？接口如何降低耦合？

## 步骤6——部署图：物理架构与扩展性

### 为什么用部署图？

1. 组件图说明**逻辑部署**，部署图说明**物理部署**

|------|------|
| 客户端 | 浏览器/手机 |

| 应用服务器 | Tomcat（运行服务组件） |

## 高级内容——从OOD到OOP的迭代细化

### 为什么需要迭代？

1. 一次性写出完美设计很难
2. 通过迭代，逐步添加细节，验证可行性
3. AI辅助可以快速将设计转化为代码，并提供反馈

### 迭代过程（四轮）

| 轮次 | 内容 |
|------|------|
| 第一轮 | 生成基础结构（trait、结构体、模块） |
| 第二轮 | 完善业务逻辑（约束、计算、异常） |
| 第三轮 | 增加持久化抽象（Repository模式，依赖倒置） |
| 第四轮 | 整合为可运行原型 |

### 核心思想

1. **分离关注点**：业务逻辑与存储逻辑分开
2. **依赖倒置**：服务依赖抽象接口，不依赖具体实现
3. **开闭原则**：对扩展开放，对修改封闭

## 总结

### 核心收获

1. UML图是**多视角**的沟通工具，不是负担
2. 从OOA到OOD到OOP是**逐步细化**的过程
3. 动态建模（顺序图、状态图、活动图）补充了静态结构图（类图）的不足
4. 架构图（组件图、部署图）关注**宏观组织与物理部署**
5. AI可以帮助我们快速生成设计图和代码，但我们需要理解**为什么这样设计**

1. 尝试为系统增加新的功能（如预约、邮件通知），看如何修改UML图和代码

3. 探索更多设计模式在系统中的运用
