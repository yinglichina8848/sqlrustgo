## 二、版本开发流程

### 2.1 标准版本生命周期

```
┌─────────────────────────────────────────────────────────────────┐
│                    版本开发标准流程                              │
└─────────────────────────────────────────────────────────────────┘

1. 规划阶段
   ├── 确定版本目标与定位
   ├── 分析需求与功能
   └── 制定开发计划

2. 开发阶段
   ├── 创建开发分支 (develop/vX.Y.Z)
   ├── 实现功能开发
   ├── 持续测试与修复
   └── 文档编写

3. RC 阶段
   ├── 门禁检查
   ├── 测试覆盖评估
   ├── 创建 RC 分支 (rc/vX.Y.Z)
   └── 发布测试

4. GA 发布
   ├── 创建发布分支 (release/vX.Y.Z)
   ├── 创建标签 (vX.Y.Z)
   ├── 合并到 main
   └── 发布公告
```

### 2.2 分支策略

| 分支类型 | 命名规则 | 用途 | 生命周期 |
|----------|----------|------|----------|
| main | main | 受保护的主干 | 永久 |
| develop | develop/vX.Y.Z | 开发分支 | 版本周期 |
| release | release/vX.Y.Z | 发布分支 | 临时 |
| rc | rc/vX.Y.Z | RC 测试 | 临时 |
| feature | feature/<功能名> | 功能开发 | 临时 |
| fix | fix/<问题名> | Bug 修复 | 临时 |

### 2.3 分支保护规则

- `main` 分支受保护，禁止直接推送
- 所有变更通过 PR 合并
- 需要至少 1 位审核者批准
- CI 检查必须通过

## 三、门禁检查体系

### 3.1 门禁检查清单

每个版本发布前必须通过以下检查：

| 检查项 | 要求 | 工具 |
|--------|------|------|
| 编译检查 | 无 error | cargo build |
| 单元测试 | 全部通过 | cargo test |
| 集成测试 | 全部通过 | cargo test --workspace |
| SQL-92 测试 | 100% 通过 | test/sql92 |
| 格式化 | 通过 | cargo fmt --check |
| Clippy | 无 error | cargo clippy |
| 覆盖率 | ≥70% (RC), ≥80% (GA) | cargo tarpaulin |

### 3.2 测试覆盖率目标

| 阶段 | 目标覆盖率 | 说明 |
|------|------------|------|
| Alpha | ≥50% | 基础功能覆盖 |
| Beta | ≥65% | 主要功能覆盖 |
| RC | ≥70% | 接近生产可用 |
| GA | ≥80% | 生产可用 |

## 六、PR 合并流程

### 6.1 PR 创建

1. 从功能分支创建 PR
2. 目标分支: develop/vX.Y.Z
3. 填写标题和描述
4. 关联相关 Issue

### 6.2 PR 审核

1. 至少 1 位审核者批准
2. CI 检查全部通过
3. 无合并冲突

### 6.3 PR 合并

1. 切换审核账号 (yinglichina8848)
2. 批准 PR
3. 使用 admin 权限合并
4. 删除功能分支

## 十一、关键原则

### 11.1 开发原则

1. **先测试后开发**: TDD 优先
2. **小步快跑**: 频繁提交 PR
3. **文档优先**: 文档与代码同步
4. **门禁把控**: 不达标不发布

### 11.2 代码质量原则

1. **无 warning**: clippy 检查
2. **格式化**: cargo fmt
3. **测试覆盖**: 核心功能 100%
4. **向后兼容**: 不破坏现有功能

### 11.3 协作原则

1. **审核制度**: PR 必须审核
2. **代码所有**: 共同维护
3. **问题透明**: Issue 公开跟踪
4. **知识共享**: 文档驱动
